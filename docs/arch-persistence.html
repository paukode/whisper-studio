Persistence

Everything Whisper Studio remembers between requests lives in one SQLite file. This page walks the schema, the per-session config latch, the companion tables, the startup migrations, and the locking that keeps a background write from clobbering a foreground save.

There is no external database and no server-side session cache to warm. A single file, storage/sessions.db, holds every chat, transcript, cost line, and scheduled-run record. It opens in WAL mode so the app can read while a write is in flight, and the schema evolves through numbered migrations that run once at startup. The data-access code lives in server/infrastructure/sessions.py.

One database, WAL mode

Every connection is opened through a small context manager in server/infrastructure/sessions.py that sets three pragmas on the way in:

PragmaValueWhy
journal_modeWALReaders and a single writer proceed concurrently, so a list query never blocks on an in-flight save.
busy_timeout5000 msWhen the writer is held, a second connection waits up to five seconds instead of raising database is locked.
synchronousNORMALFewer fsyncs than FULL; safe with WAL for a local-first, single-node app.

The connection is opened with a timeout=5.0 as well, and each context exits by committing then closing. If a pragma cannot be applied the code logs a warning and continues rather than failing the request. Rows come back as sqlite3.Row, so columns are read by name.

WAL writes a sessions.db-wal and sessions.db-shm sidecar next to the database. They are normal, and SQLite checkpoints the WAL back into the main file on its own. If you copy the database for a backup, copy all three files together.

The sessions table

The sessions table is the heart of the store. Its base shape is created on import by _ensure_db(); later columns are added by migrations (see below). A session carries both the chat conversation and, for recorded sessions, the transcript.

id
Primary key. A UUID for user sessions, or the fixed sentinel __cron_inbox__ for the pinned Scheduled Reports inbox.
title, custom_title, generated_title
The display title plus two flags recording whether the user renamed it or the app auto-generated it.
created_at, updated_at
ISO-8601 UTC timestamps. The list query sorts by updated_at DESC, and an index backs that sort.
segments
JSON array of transcript segments (text plus speaker) for recorded sessions. Empty [] for chat-only sessions.
chat_history
JSON array of chat messages. Holds prompt roles (user, assistant) and UI-only cron_event rows (see below).
speaker_names
JSON map of speaker id to a human-assigned name for diarized transcripts.
workspace_path
The folder this session is bound to, if any. The path is read from the DB, never the client, when opening a workspace externally.
latched_config
JSON snapshot of the config fields frozen for this session on its first turn (see the session latch).
pinned, archived
Sidebar flags. Pinned floats to the top; archived hides from the default list.
compaction_count
How many times history compaction has run for this session, surfaced in the UI so a long conversation shows it is being trimmed.
goal, goal_state
The WS-E goal loop's target text (goal) and its progress as a JSON blob (goal_state), added additively via _MIGRATED_SESSION_COLUMNS and surfaced in _row_to_dict() as goal / goalState.

Reads go through _row_to_dict() (full detail) or _row_to_summary() (list view). Both use a _safe_col() helper that returns a default when a column is absent, so a row written before a migration still deserializes cleanly.

The session latch

Settings can change mid-conversation, but you do not want the model or effort level to switch out from under an in-flight chat. The latch solves this. On the first turn, latch_session() in server/infrastructure/config.py snapshots a fixed set of fields and reuses that snapshot for every later turn in the session. Changing Settings afterward affects new sessions, not this one.

The latched fields are:

  • default_chat_model and chat_models - which model runs, and the enabled set.
  • chat_model_meta - per-model metadata used to render and price the choice.
  • effort_level and brief_mode - the thinking budget and response length.
  • permission_mode - the approval posture for the turn.
  • bedrock_region - the region the request is routed to.

The live snapshot lives in an in-memory dict (_latched_sessions) guarded by a lock, and the same values are persisted in the latched_config column so a session survives a restart with its choices intact. When a session is deleted, its snapshot is dropped via unlatch_session() as part of the delete cascade.

Latching happens at the very start of the turn, before the prompt is built. See Chat & tool orchestration for where it sits in the request flow.

chat_history and cron_event rows

chat_history holds more than what the model sees. Alongside user and assistant messages, a background scheduled task can append a cron_event row so its result renders inline in the conversation that created it. Those rows are UI-only, and two rules keep them from causing trouble:

  1. They never enter the prompt

    visible_chat_history() filters chat_history down to the prompt roles (user, assistant) before the Bedrock messages array is built. A cron card you see in the chat does not inflate the token bill.

  2. They are capped

    At most 50 cron_event rows are kept per session (MAX_CRON_EVENTS_PER_SESSION). When the cap is exceeded, the oldest are dropped on append. The durable per-run record still lives in cron_runs, so the history drawer never loses data when the inline cap trims.

Reactive compaction is the other force acting on chat_history. When a prompt grows too long, the oldest message pairs are summarized or dropped and the turn retries; compaction_count is incremented so the UI can show it. See Chat & tool orchestration for the compaction path.

The companion tables

Five other tables share the same database file, each created by a numbered migration rather than on import.

TableGrainHolds
session_costsOne row per model turnsession_id, turn_number, model, input / output / cache-read / cache-creation tokens, cost_usd, and api_duration_ms. Backs the cost aggregates and spend caps.
cron_runsOne row per scheduled-task firerun_id, job_id, job_name, session_id, a status lease (running / ok / failed), timestamps, duration_ms, the result text, and the computed next_run.
agent_tasksOne row per background tasktask_id, kind (shell / agent / workflow), session_id, title, a status lease with owner_pid, and kind-specific fields. The unified background-task registry (migration 007).
notificationsOne row per notify_userAn id, session_id, source (chat / agent / cron), title, message, status, and a read_at stamp. The durable log behind the header bell (migration 008).
workflow_runsOne row per workflow runrun_id, name, session_id, a status lease, the phase list, and a token / cost_usd / budget_usd ledger for the ultracode workflow runtime (migration 009).

The status lease is a recurring pattern: cron_runs, agent_tasks, and workflow_runs each write a running row that a boot-time reconcile flips to a terminal state (failed / interrupted / stale) if it is still open after a restart, so nothing hangs forever; notifications is instead the durable record behind the header bell. Each table is indexed for its hot query - session_costs by session_id and created_at, cron_runs by (job_id, started_at DESC), and agent_tasks / notifications / workflow_runs by session and status. See Cost tracking & budgets and Cron / scheduled tasks for how each is written and read.

Migrations

The schema is not frozen. server/migrations/runner.py runs once during the app lifespan (called from server/main.py) and applies any pending migration modules in order. Each module in server/migrations/ is a file named NNN_description.py that exports a VERSION, a DESCRIPTION, and a migrate(conn) function.

The runner discovers those modules, compares their versions against a schema_version tracking table, and applies only the ones not yet recorded. Each migration runs in a transaction: on success its version is inserted into schema_version and committed; on any exception the runner rolls back and stops, so a half-applied migration never leaves the database in an unknown state.

VersionModuleWhat it does
001001_add_session_costs.pyCreates the session_costs table for per-turn cost tracking.
002002_add_session_metadata.pyAdds workspace_path, compaction_count, and latched_config columns to sessions.
003003_backfill_session_timestamps.pyBackfills empty updated_at / created_at so old rows sort and render correctly.
004004_purge_orphaned_session_data.pyOne-time sweep of orphaned session_costs / tasks rows left by pre-cascade deletes.
005005_add_session_flags.pyAdds the pinned and archived flags to sessions.
006006_add_cron_runs.pyCreates the cron_runs table for per-job scheduled-task run history.
007007_add_agent_tasks.pyCreates the agent_tasks table (unified background-task registry) with session and status indexes.
008008_add_notifications.pyCreates the notifications table (durable notify_user log) with unread and session indexes.
009009_add_workflow_runs.pyCreates the workflow_runs table (ultracode workflow runtime) with session and status indexes.

Column-adding migrations wrap each ALTER TABLE in a guard so a re-run over a database that already has the column is a no-op, while genuine errors still propagate. SQLite requires one ALTER TABLE per column, which is why those migrations loop.

A migration that raises stops the whole runner, so the app will fail to finish startup rather than serve against a partial schema. Migrations should be idempotent and forward-only; there is no automatic down-migration.

Saves, locks, and the merge

Two writers can touch the same session at once: the frontend saving the whole session, and a background cron fire appending a cron_event. A naive full save from the frontend would overwrite the cron row that landed a moment earlier. Two mechanisms prevent that.

  1. A per-session lock

    Every writer takes an asyncio.Lock keyed by session id (_lock_for()). The save endpoints, the beacon save, append_message(), and the delete cascade all hold the same lock, so a read-modify-write can never interleave with another writer for the same session.

  2. A UPSERT that merges backend rows

    A session save is a single UPSERT (INSERT ... ON CONFLICT(id) DO UPDATE). Before writing, _upsert_session() reads the existing chat_history, pulls out the backend-owned UI-only rows, and appends any (by timestamp) that the frontend payload does not yet include. So a cron card written between the last hydrate and this save survives the round-trip.

The cap on cron_event rows is re-applied after the merge, so merging can never push the count over 50. Two save paths share this logic: the normal PUT /api/sessions/{id} and a navigator.sendBeacon POST used on page unload. Deleting a session is also lock-guarded and cascades: it removes the parent row and session_costs, re-homes cron jobs and history to the inbox, and clears the per-session in-memory caches.

The Scheduled Reports inbox

A scheduled task can outlive the session that created it. If that session is deleted or archived, the firing must still land somewhere. ensure_cron_inbox() creates a fixed, pinned session with id __cron_inbox__ and title "Scheduled Reports" that catches orphaned runs. It is a fixture, not a user session: the delete endpoints refuse to remove it, so orphaned cron runs are never stranded.
