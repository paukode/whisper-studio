Cron / scheduled tasks

Whisper Studio can run a prompt on a wall-clock schedule and stream the result back into the conversation that created it. This page covers the schedule union, the APScheduler runtime, the run-history lease, and how a firing lands as a card in your chat.

Two modules carry the design. server/cron_schedule.py is the pure schedule layer (timezone resolution, the discriminated schedule union, labels, and trigger construction); server/cron_scheduler.py is the runtime (lifecycle, execution, delivery, and routes). Run history lives in server/cron_history.py. Click a box in the diagram to trace a firing from schedule to chat card.

The schedule union

A job's schedule is a discriminated union with three variants, all normalized and validated by validate_schedule() in server/cron_schedule.py. Bad input raises at create time rather than failing silently at fire time.

TypeShapeBehavior
interval {type:"interval", seconds:1800} Repeats every N seconds (minimum 10). Anchor-based: a resumed job fires N seconds after its last run, not immediately. every_minutes is accepted and converted.
cron {type:"cron", hour:9, minute:0, day_of_week:"mon-fri", tz:"America/New_York"} A timezone-aware wall-clock time. hour and minute may be a single value or a comma list ("9,16" for twice a day); day_of_week accepts *, ranges (mon-fri), lists (mon,thu), or a single day.
at {type:"at", run_at:"2026-07-06T09:00:00", tz:"..."} A one-shot run at a specific instant. run_at must be in the future, and the job auto-disables after it fires (see below).

Where the timezone comes from

When a cron or at job omits tz, the default zone is resolved by _default_tz_name(): the cron_timezone config value if set, otherwise the host system zone. The system zone comes from $TZ, then the /etc/localtime symlink (macOS and Linux), then UTC as a last resort, with no third-party dependency. Because cron triggers carry a real ZoneInfo, "daily at 09:00" fires at the intended local hour and stays correct across DST. A bad IANA name is rejected eagerly at create time.

Every variant flows through schedule_label(), which produces the human string reused by the tools, the API, the panel, and the chat cards ("weekdays at 09:00 · America/New_York", "every 30 min", "once on Jul 06, 2026 at 09:00"). The countdown to the next run is projected by compute_next_run() straight from the schedule, so the UI stays accurate even when the scheduler is not running.

The runtime

init_scheduler() runs once at app startup. It captures the server event loop (so background threads can dispatch coroutines later), reconciles any interrupted runs, imports the legacy results file, then starts an APScheduler AsyncIOScheduler and arms every enabled job. Jobs themselves live in data/cron_jobs.json, mutated under a lock via an atomic temp-file write so a concurrent toggle, delete, and create can never last-write-wins over each other. APScheduler is an optional dependency: if it is not installed, cron is simply disabled and the rest of the app runs normally.

When a job fires, _make_runner() stamps last_run and bumps run_count under the file lock, then hands off to _execute_cron_prompt() on a daemon thread. That function:

  1. Guards against overlap

    An in-progress set (plus APScheduler max_instances=1) ensures the same job never runs twice at once, covering the run-now vs scheduled-fire race.

  2. Opens a run lease

    cron_history.start_run() writes a cron_runs row with status running. This lease is what lets a crash be detected later.

  3. Runs the job prompt

    The prompt is sent to Bedrock inside a short tool-use loop. A per-job Anthropic model override takes precedence: when the job's model field resolves to a configured Anthropic Bedrock id (validated by _validate_job_model(), accepted by cron_create / cron_update and PUT /api/cron/{id}), that model runs; otherwise the loop falls back to the configured Haiku model, then Sonnet, then the first configured model. The system prompt marks the run as a background cron monitor and passes the global prompt rules through append_rules().

  4. Delivers and closes the lease

    The final text is pushed to the originating session and finish_run() closes the lease with ok or failed, the duration, and the next run time.

A one-shot at job disables itself (setting enabled=false and stamping fired_at) before the execution runs, so a crash in the fire-then-persist window cannot re-deliver it on the next restart. A run still marked running for more than 15 minutes at boot is reconciled to failed (interrupted).

Run history

server/cron_history.py persists every run to the cron_runs table in the shared sessions.db (WAL mode, concurrent reads plus a single writer). Each run carries a lease: written as running when the job fires, updated to ok or failed when it finishes. On startup, reconcile_stale() marks any lease older than the 15-minute stale cutoff (STALE_RUN_SEC) as failed, so a run interrupted by a restart is closed instead of hanging forever.

Old runs are pruned per job to a configurable ceiling (default 200, from cron_max_runs_per_job). A previous flat data/cron_results.json is imported once via import_legacy() then retired: init_scheduler() renames it to cron_results.json.imported so it is never imported twice.

server/cron_history.py
Lease-based run store: start_run, finish_run, list_runs, recent_runs, reconcile_stale.
cron_runs
Table keyed by run_id, indexed on (job_id, started_at DESC); carries status, timestamps, duration, text, and next run.

Chat integration

Cron output does not live in an inbox by itself; it flows back into the chat. When a job carries a session_id, a firing calls _emit_cron_event(), which now delegates to the shared emit_session_event() in server/tasks/events.py - the unified session-event service that both appends a cron_event row to that session's history and publishes the same payload live to the event bus as a {type:"cron_event"} event so any open SSE stream renders it immediately. (The same service backs the task_event role.) The row shows up as a compact cron_event card in the conversation.

A running job is no longer opaque. Alongside the final card, server/cron_run.py publishes live progress through emit_progress() in server/cron_events.py: TeamProgressEvent-shaped frames (started / turn_start / tool_call / tool_result / completed / failed / stopped) go onto the event bus as {type:"cron_progress"} and are forwarded to the session SSE as team_progress, so the run renders as a live per-tool log in the existing TeamReportCard/AgentCard fold.

A cron run that calls notify_user also writes a durable source='cron' row to the notifications table (migration 008), so a report delivered while you were away is still waiting behind the header bell.

These cards are UI-only. They are filtered out by visible_chat_history before the next prompt is built, so background firings never inflate the token bill or confuse the model. If the owning session was deleted or archived, _resolve_target_session() redirects the card to a pinned "Scheduled Reports" inbox instead of dropping it, and a session delete cascades to on_session_deleted(), which disables and flags the job as orphaned (never hard-deleting it) and repoints its run history to the inbox.

A cron_event is emitted on more than just firings: cron_created, cron_updated, and cron_deleted events also surface in the session so the lifecycle is visible where you scheduled it. See how these events are drained alongside chat frames in Chat & tool orchestration.

Endpoints and tools

The same logic backs both the assistant's tools and the cron panel's REST API. Claude gets cron_create, cron_update, cron_run, cron_list, and cron_delete; the panel calls the routes below (all under the /api/cron prefix).

RouteMethodPurpose
/api/cronGET / POSTList all jobs (with schedule label, next run, run state, and system timezone), or create one.
/api/cron/previewPOSTValidate a schedule and return its label and next run without persisting anything.
/api/cron/{job_id}GET / PUT / DELETEFetch a job with its recent runs, edit it, or delete it.
/api/cron/{job_id}/historyGETList a job's run history.
/api/cron/{job_id}/runPOSTFire a job once, off its schedule (409 if it is already running).
/api/cron/{job_id}/stopPOSTCooperatively stop an in-flight run of this job (404 if the job is unknown, 409 if it is not running).
/api/cron/{job_id}/togglePATCHFlip a job's enabled state and re-arm or remove its trigger.
/api/cron/runs/recentGETNewest runs across all jobs; drives the sidebar unread badges.

Editing a schedule clears the interval anchor (last_run is reset) so the new cadence does not fire off a stale timestamp, and re-homing an orphaned job to a new session clears its orphaned flag. Panel-created jobs with no active chat session land their results in the Scheduled Reports inbox rather than being rejected.

The stop route is a cooperative signal, not a kill. stop_cron_job() calls request_stop() in server/cron_events.py, which sets a per-job threading.Event that the run loop checks at round boundaries and between sequential tools; an in-flight tool wait polls in one-second slices, so the run winds down and exits with status stopped within about a second of the request.
