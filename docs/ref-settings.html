Settings & config keys

Every field in the in-app Settings panel, and the exact config.json key it writes. Use this page as a lookup when you want to hand-edit config or understand what a toggle actually changes.

Whisper Studio has no separate config UI to learn: the gear icon opens a Settings panel whose fields map one-to-one onto keys in config.json. Everything below is resolved by server/infrastructure/config.py, which merges several layers in priority order (see Config layers). If you prefer to edit JSON directly, the key names in the second column are what you write.

Most of these fields are also latched per session: the model, region, effort level, brief mode, and permission mode are snapshotted when a chat turn first starts, so changing them mid-conversation never disrupts an in-flight turn or invalidates the prompt cache. The new value applies to the next new session. See Persistence for the latch details.

Settings-UI fields

Open the panel with the gear icon in the top bar. Each row is one control:

SettingWhat it does
Bedrock Region The AWS region every Claude and GPT call is routed to. It must match a region where the model you selected is actually enabled, or Bedrock returns an access error. Defaults to us-east-1. See Region pitfalls.
Default Model The model new sessions start with: Haiku 4.5, Sonnet 4.6, Sonnet 5, Opus 4.6 / 4.7 / 4.8, Fable 5.0, GPT-5.4 / 5.5, or on-device Gemma when local mode is on. The picker only shows models valid for the current Model Mode (cloud hides on-device models and vice versa). Default is opus4.8. Fable 5.0 requires the data-retention consent flow before it can be selected. Full catalog on Model backends.
Model Mode Where indexing and retrieval run: cloud (all on Amazon Bedrock), local (all on-device), or hybrid (pick a backend per capability). This also drives which chat models the picker shows. Changing it prompts a reload so the app re-initializes in the new mode. See Model modes.
Feature Flags Per-feature on/off toggles for gated behavior (hybrid search, reranker, query rewrite, prompt caching, and others) without editing config.json by hand. Changes persist immediately. See Feature flags.
Effort Level The reasoning budget the model is given, from lowest to highest: low, medium, high, extra (also accepted under its alias xhigh), max, and ultracode. Higher effort trades latency and cost for deeper thinking. Default is high. The levels a model exposes depend on its tier: Opus 4.8 and Fable expose the full ladder up through ultracode; Sonnet and older Opus cap at max; GPT-5.x runs none through max; Haiku and the local models have no effort control. The value is clamped to the nearest level the selected model supports when you switch models. ultracode is a session mode rather than a plain level - it sends the top reasoning effort plus dynamic-workflow orchestration. The old auto value is retired and is coerced back to a real level.
Permission Mode How the assistant is gated when it wants to act: default, auto, plan, acceptEdits, bypassPermissions, or dontAsk. Default is default (reads run freely, writes ask). See Permission modes and Permissions & approvals.
Tavily API Key Enables the web-search and web-fetch skills. Left empty by default, which disables web search entirely. The API never returns the raw key back to the UI; you only see a masked hint. Prefer the environment variable path (see Tavily key).
Cost Limits Two spend caps in USD: a per-session cap and a per-day cap. When a running turn would exceed either, it stops with an error. 0 means unlimited (the default for both). See Cost tracking & budgets.

Each UI field above writes one or two keys into config.json. This is the mapping, with default values from server/infrastructure/config.py:

Older configs used local_mode (a plain on/off boolean) before model_mode existed. Both are still read: if a config has local_mode set but no explicit model_mode, the loader treats it as "local" so your on-device models stay visible. An explicit model_mode always wins.

Model mode and backends

model_mode picks where the index and retrieval pipeline runs; the backends object is consulted only in "hybrid" mode, where each capability can point at a different engine. Unset capabilities fall back to the cloud backend.

backends.embed
cohere or qwen3 - the embedding model for the index.
backends.rerank
cohere or qwen3 - the reranker applied to retrieved passages.
backends.ner
haiku or gliner - entity extraction for the GraphRAG entity graph.
backends.index_llm
haiku, local, or none - the model that summarizes and enriches indexed content.

These map onto the same per-capability picks that setup.sh --hybrid pulls the on-device weights for. See Model backends & modes and Index / GraphRAG for how each capability is used.

Feature flags

The feature_flags key is a small namespace of booleans that gate optional behavior without a code change. Toggling one in Settings persists it immediately. Under the hood the toggle is surgical: it patches only that one boolean inside the feature_flags object and leaves every other byte of config.json untouched, so it never reflows your formatting or flattens the model catalog. Flags cover features such as hybrid search, the reranker, query rewrite, and prompt caching. The live state of each flag is exposed to the UI through the config endpoint so the panel can show what is currently on.

Several flags added with the ultracode program default on:

FlagDefaultWhat it gates
progressive_toolstrueProgressive tool disclosure: advertise a core tool set plus session-activated tools, with everything else discoverable via tool_search from a compact index. Cuts roughly 60-75% of tool-schema tokens per turn.
goal_looptrueGoal loop + completion gate: a session goal (set with /goal) auto-continues the turn until a cheap evaluator judges it achieved, capped at 8 blocks. See The ultracode workflow runtime.
cron_verifytrueVerify a cron run actually achieved its prompt before pushing an "ok" status.
preview_toolstrueAssistant-controllable browser preview tools (start/stop/click/fill/eval/screenshot); they only surface once Playwright and Chromium are installed.

Unlike the latched fields, feature flags take effect right away rather than waiting for a new session, because they change server behavior rather than the cached prompt.

Transcript condensation keys

The map_reduce_summary object tunes how an oversized transcript is condensed before the summary skills run. It has no Settings-UI field, so edit config.json (or the config.example.json template) directly. Every size below is in characters (roughly 4 characters per token), and any malformed numeric value falls back to its default rather than failing the chat turn.

Sub-keyDefaultMeaning
enabledtrueMaster switch. When false, transcripts are never condensed and pass through unchanged.
threshold_chars600000Condense only above this size (about 150k tokens). Below it the transcript passes through with no extra LLM calls.
chunk_chars150000Per-chunk size for the cloud map step (about 37k tokens).
local_chunk_chars40000Per-chunk size in local mode (about 10k tokens; fits the 16k on-device window).
overlap_chars800Overlap between adjacent chunks so nothing is lost at a boundary.
map_max_tokens1200Output cap for each per-chunk extract.
max_chunks40Cap on the number of chunks; the splitter re-chunks coarser before hitting it.
max_output_chars480000Cap on the aggregate condensed output in cloud mode (about 120k tokens), so the reduce turn does not overflow.
local_max_output_chars24000Tighter aggregate cap for the smaller local window.
engine"auto"Which model runs the map step. "auto" follows the model mode (cloud uses Haiku, local uses the on-device model); force "haiku" or "local" to override.

See Transcript summarization for how the map-reduce pipeline uses these values.

Permission modes

The permission_mode key controls how the assistant is gated when it wants to run a tool. You can also switch it inline with the Mode dropdown in the chat input. Per-tool rules under Settings > Permissions override the mode for specific commands.

ValueBehavior
defaultReads run freely; writes ask for approval.
autoA classifier decides per-tool, per-call (configured in the Auto Mode tab).
planRead-only. The assistant plans but never executes.
acceptEditsFile edits auto-approve; deletes and shell commands still ask.
bypassPermissionsEverything runs without prompts.
dontAskReads auto-allow; writes are silently denied.

The auto mode reads three companion keys - auto_mode_allow, auto_mode_soft_deny, and auto_mode_environment - which the Auto Mode tab manages for you. See Approval & permissions for the classifier and nonce flow.

The Tavily key

Web search is off until a Tavily key is present. You can set it two ways, and the environment variable always wins:

  1. Environment variable (preferred)

    Export TAVILY_API_KEY in your shell. It overrides config.json:tavily_api_key and is re-read on every config load, so a rotated key applies without a restart and never touches disk.

    echo 'export TAVILY_API_KEY=tvly-...' >> ~/.zshrc

  2. Settings field

    Paste the key into the Tavily API Key field. It is stored in config.json. The config endpoint never returns the raw value - only a masked hint like tvly...abcd - so the UI can show that a key is set without exposing it.

See Web research to put the key to work and Environment variables for the full env list.

Config layers

A resolved config is the merge of up to four layers, lowest priority first. Higher layers override lower ones, and nested objects merge key-by-key (except the model catalog, which replaces wholesale).

#LayerSourceScope
1Defaultsbuilt into server/infrastructure/config.pyFallbacks for everything.
2Userconfig.json at the repo root (gitignored)Your global preferences for all sessions.
3Project.whisper/settings.json in the workspace rootPer-project overrides for a safe subset of keys.
4Environmentshell env vars (e.g. TAVILY_API_KEY)Secrets and deployment overrides; wins over everything.

config.json is the User layer. It is gitignored and created on first run when setup.sh copies config.example.json over it. A project's .whisper/settings.json overrides your user config, but only for a project-safe subset of keys (region, model, effort, brief mode, permission mode, the auto-mode lists, cost caps, and feature flags) - global-only secrets like the Tavily key are not accepted there. Environment variables override both.

Separately from that config subset, .whisper/settings.json may also carry an optional hooks key: arbitrary project hook code from the cloned repo. It is treated as untrusted and stays inert until the exact hook set is explicitly trusted (a SHA-256 of the canonical hooks, mirroring trusted skills), so a checked-out repo cannot run hook code on its own. See the Security model for the project-trust flow.

Secrets hygiene. config.json is gitignored, but it sits on disk in plain JSON. Treat it like any credential store: do not sync it to untrusted backups, rotate keys after sharing your screen, and prefer the env-var path for the Tavily key.

Region and edit pitfalls

  • Region must be non-empty. A blank bedrock_region would build a malformed endpoint and fail every Bedrock call, so the loader coerces blank or missing values back to us-east-1 and strips stray whitespace. The Settings UI and the config API both refuse to persist a blank region or default model.
  • Region has to match the model. Enabling a model in one region and pointing bedrock_region at another produces an access error. If chat fails with an access or model-ID error, this mismatch is the usual cause.
  • Unknown default model falls back. If default_chat_model names a key that is not in the catalog, the loader falls back to opus4.8 (or the first available model), so resolution never dead-ends.
  • Prefer the UI or a surgical edit for the model catalog. The rich chat_models shape carries per-model metadata; a naive re-save can flatten it. To add or change a model, edit config.example.json (the canonical template) rather than hand-flattening the live file.
