Model modes

Choose where indexing and retrieval run: cloud, hybrid, or fully local. One dropdown decides whether embeddings, the reranker, and entity extraction reach out to Amazon Bedrock or stay entirely on your laptop.

You want to index a private repo but keep every byte on your machine. Or you want the leanest possible install and are happy to let Bedrock do the heavy lifting. Model mode is the single switch for that trade-off, and this tutorial walks through picking the right one.

The goal

Decide where the indexing and RAG stack runs. When you index a folder for semantic search, three capabilities do the work: an embedder that turns text into vectors, a reranker that reorders the top hits, and an entity extractor that pulls names and relationships for the entity graph. Model mode is the one setting that says whether those run in the cloud, on-device, or a mix.

Model mode is not your chat model. The model you talk to (Haiku, Sonnet, Opus, GPT, on-device Gemma) is chosen in the chat toolbar and is a separate decision. Model mode (Settings → Model Mode) controls only the indexing and retrieval backends. You can run local indexing while chatting with cloud Claude, or the reverse.

Cloud (default)

The default. Embeddings and reranking go to Cohere on Amazon Bedrock, and entity extraction uses Claude Haiku. Because nothing runs on-device, the install skips roughly 16 GB of model weights, so it is the leanest setup and the fastest to get running. Pick this when you are happy for indexing to use Bedrock (and to be billed for it) and you want the smallest footprint.

Hybrid (per capability)

Hybrid lets you choose a backend for each capability independently, stored under the backends key in config. The capabilities are embed (cohere or qwen3), rerank (cohere or qwen3), ner (haiku or gliner), and index_llm (haiku, local, or none). Any capability you leave unset falls back to the cloud backend. This install pulls the on-device weights too, so both sides are available to mix. Pick this when you want, say, local embeddings for privacy but the cloud reranker for quality.

Local (fully on-device)

Everything runs on your laptop: Qwen3 for embeddings, the Qwen3 Reranker, and GLiNER for entity extraction. No Bedrock calls are made for indexing or retrieval, so nothing about your indexed folders leaves the machine. The cost is disk and memory: this mode pulls around 16 GB of weights and runs them locally. Pick this when the data you index must never touch the cloud.

What stays local in every mode

Two things do not depend on the mode you choose:

  • Transcription is always on-device. Whisper and Parakeet run on your machine in every mode, including cloud. Audio never leaves the laptop regardless of the model mode. See Voice & meetings.
  • Each embedder keeps its own index. The cloud and local embedders produce different vectors, so they store separate indexes. Switching mode never rebuilds or invalidates the other one, which is convenient (nothing is lost) but means a folder indexed under one backend is not searchable under the other until you re-index it there.

If you switch to local and a folder suddenly returns no results, it is almost always because that folder was indexed under the cloud embedder. Re-index it once under the new mode and it will search there. The old index is still intact if you switch back.

A note on regions

Cohere embed and rerank are pinned to us-east-1. Your chat model can run in any region you set as the Bedrock Region, so cloud indexing works even when your chat traffic goes elsewhere. You do not need to move your chat region to us-east-1 to use cloud or hybrid mode; the indexing calls target us-east-1 on their own.

Changing the mode

You can set the mode at install time or change it any time from the app.

  1. At install (optional)

    Pass a flag to the setup script to choose up front. Cloud is the default and skips the on-device download; hybrid and local pull the ~16 GB of weights so they are ready to use.

    bash setup.sh --cloud    # default: indexing on Bedrock, lean install
bash setup.sh --hybrid   # per-capability; pulls on-device weights too
bash setup.sh --local    # everything on-device; pulls all weights

  2. In the app

    Open Settings (the gear icon) and go to the Model Mode tab. Switch between cloud, hybrid, and local there. In hybrid, this is where you pick a backend per capability.

  3. Confirm the reload

    Changing the mode prompts a reload so the app re-initializes in the new mode (it has to load or release the on-device models). Accept it, and the indexing stack comes up on the backends you chose.

If you installed with --cloud (the default), the on-device weights were never downloaded, so switching to hybrid or local will fetch them on first use. Expect a one-time delay of a few minutes and a few gigabytes of disk while that happens.

Which mode should you pick?

Pick Cloud if

You want the smallest install and the fastest start, you are fine with indexing running on Bedrock, and your indexed data is not sensitive enough to require staying local.

Pick Local if

The folders you index must never leave the machine. You accept ~16 GB of weights and some local compute in exchange for zero Bedrock calls during indexing and retrieval.

Pick Hybrid if

You want to tune per capability, for example local embeddings for privacy plus the cloud reranker for quality. It carries both sets of weights so you can mix freely.

Still not sure?

Start on Cloud. It is the default, needs no download, and you can switch to hybrid or local later without losing your existing cloud index.

The config behind it

If you prefer editing config.json directly, these are the keys the Settings tab writes:

model_mode
One of cloud (default), hybrid, or local. Resolved in server/infrastructure/model_mode.py. It also drives which chat models the picker shows: cloud hides on-device models, and local hides cloud ones.
backends
Consulted only in hybrid mode. A per-capability map with keys embed, rerank, ner, and index_llm. Unset capabilities fall back to the cloud backend.

For how these settings are read at runtime and how the backend is selected per capability, see Model backends & modes.
