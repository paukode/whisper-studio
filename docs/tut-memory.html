Memory & WHISPER.md

Give Claude a memory that survives past the current thread, and a place to write down the standing facts about your project so you never repeat them.

Out of the box, each chat starts fresh: Claude knows only what is in the current thread. Turn on memory and that changes. Claude quietly records the facts worth keeping, writes a running summary of the conversation, and reads both back on later turns. Add a WHISPER.md to your workspace and it gets the project context you would otherwise re-explain every time.

There are three separate mechanisms here, and it helps to keep them straight from the start: workspace memory (durable facts that carry across sessions), session memory (a structured summary of the thread you are in), and WHISPER.md (a file you write by hand). The diagram traces how they meet in the prompt.

On by default, in two tiers

Memory is on out of the box. The underlying flags, auto_memory for the persistent memory and extraction and session_memory for the per-session summary, both default on, so everything below happens automatically without you calling any tools or running any commands. If you would rather turn it off, flip the Memory row in the "+ More" menu of the chat toolbar, or type /memory off.

Persistent memory has two tiers, so there is always somewhere for a fact to go, workspace or no workspace:

  • Global memory is cross-workspace and stays alive even in plain chat with no workspace open. It holds facts about you: preferences, role, and general feedback on how you like Claude to work.
  • Project memory is scoped to the open workspace. It holds workspace-specific facts such as goals, deadlines, and repo references.

Project memory lives under a slug derived from the workspace folder name, so two different projects keep separate memories and never bleed into each other. Global memory sits apart from all of them, shared across every chat.

Workspace memory: durable facts

Workspace memory is a small set of Markdown files that persist across every session in a project. They live in data/memory/<workspace-slug>/ and are pulled into prompts on demand. Each file carries YAML frontmatter (a name, a one-line description, and a type) followed by a short Markdown body. An index file, MEMORY.md, lists them one line each.

Not everything is worth remembering. The extraction agent is deliberately selective, and it saves into a closed set of four types:

TypeWhat it captures
userWho you are: preferences, role, expertise.
feedbackCorrections and guidance on how you want Claude to work.
projectProject context, goals, and deadlines not already in the code.
referencePointers to external systems and resources.

It deliberately does not save things that are already recoverable from the codebase: architecture and code patterns (read the code), git history (read the log), or one-off debugging fixes (the fix is already committed). The point is to remember what only you can tell it, not to duplicate the repository.

How memory is recalled

Memory is read back on demand, not dumped wholesale into every prompt. On a new query the system scans the memory files for a manifest. If there are five or fewer files, it simply loads them all. If there are more, it runs a quick Haiku side-query that ranks the manifest against your current question and picks up to five relevant files. That keeps the prompt lean: only what is likely to matter is injected. On-device (local) turns skip the ranking side-query to stay fully offline and just take the most recent memories instead.

Session memory: a running summary

Session memory is a different animal. Rather than durable cross-session facts, it summarizes the conversation you are in right now, so a long thread stays coherent without the model re-reading every earlier turn. It is a single Markdown file with four fixed sections:

SectionContents
## GoalsWhat you are trying to accomplish this session.
## DecisionsKey choices made along the way.
## ContextImportant background that has been established.
## BlockersIssues hit or problems still unresolved.

The summary is written incrementally by a background agent and injected into later turns wrapped in a <session-memory> block. Because it is a compact digest rather than the raw transcript, a thread that has run for hours can resume without paying to re-read everything.

When it updates

The update runs as a background task after a turn, and only once the conversation has grown enough to be worth re-summarizing. Cloud sessions wait for both a large size threshold and some tool activity before updating; on-device sessions summarize far more often (every few turns) and on the local model itself, since local chats often have no tools to gate on. Either way it never blocks your reply, and if the summary fails it is simply skipped rather than disrupting the chat. For the exact thresholds and the injection format, see the architecture page.

Session memory and workspace memory are stored separately. Workspace memory carries facts between projects' sessions; session memory is a digest within one thread. You do not have to choose, both run at once when memory is on.

WHISPER.md: project context you write

Everything above is automatic. WHISPER.md is the opposite: it is a file you author, dropped in your workspace root, for the standing facts about a project that you want Claude to have on every turn. If you have used the Claude Code CLI, it is the same idea as CLAUDE.md. Unlike recalled memory, which is selected per query, WHISPER.md is loaded on every chat turn.

Good things to put in it:

  • How to run, build, and test the project (the commands you always forget to mention).
  • Conventions you care about: style rules, naming, directories that are off limits.
  • Domain vocabulary and the shape of the codebase, so Claude does not have to re-derive it.
  • Standing instructions: "always branch before committing", "never touch the generated files".

You can edit it two ways:

  1. The in-app Project Memory dialog

    Open the Project Memory dialog and edit WHISPER.md in place. This is the quickest path and keeps you in the app.

  2. Any editor

    It is just a file in your workspace root, so the Monaco editor in the workspace IDE, or any external editor, works fine. Save and the next turn picks it up.

Keep WHISPER.md short and stable. It is read on every turn, so it costs input tokens each time. Put durable, always-true facts here and let the automatic memory system handle the things that change from session to session.

The memory tools

Claude does not only read memory passively. During a turn it can operate on the store directly through four tools, so it can look something up mid-task or record a fact the moment you state it:

ToolWhat it does
memory_readRead a memory file from the workspace store to recall stored information.
memory_writeCreate or overwrite a memory file (with name, description, and type), and update the MEMORY.md index.
memory_listList memory files with their type, description, and modification date.
memory_deleteDelete a memory file from the store.

These are just tools, so they follow the same rules as any other. If you want to see the full catalogue and how tool permissions work, see Agent tools and Permissions & approvals.

Viewing and curating memory

You do not have to dig through files on disk to see what Claude has remembered. Open the Memory Viewer from the "+ More" menu in the chat toolbar, under Memory files. It lists both tiers side by side and lets you read and edit any file, delete ones that no longer apply, and promote a project memory up to global when a fact turns out to matter everywhere. The MEMORY.md index is read-only, since it is regenerated for you after every change.

Memory also announces itself as it works. When the background extractor saves or removes something after a turn, a small "Memory updated" toast appears, so you know the store changed without having to open the viewer.

Try it

The fastest way to see memory work is to tell Claude something durable, start a fresh chat, and check that it remembers:

  1. Check memory is on

    It is on by default; confirm the Memory row in the "+ More" menu shows On (or type /memory status).

  2. State a preference

    Tell Claude something worth keeping, for example how you like commit messages written, or which package manager the project uses.

  3. Keep working, then start a new session

    Have a normal exchange so the background extractor has something to save, then open a fresh chat in the same workspace.

  4. Check the recall

    Ask a question that touches that preference. Claude should apply it without being reminded, because the fact was recalled from workspace memory.

Peek at data/memory/<workspace-slug>/ and you will see the files it wrote, each with its frontmatter and body, plus the MEMORY.md index.

Where it all lives

data/memory/<workspace-slug>/
Durable project (workspace-scoped) memory: topic files plus the MEMORY.md index.
data/global_memory/
Global (cross-workspace) memory: user and feedback facts that apply everywhere, incl. plain chat.
<workspace>/WHISPER.md
Project context you write; loaded on every chat turn.
server/memory/
The memory subsystem: extraction, recall, session summaries, and tools.
