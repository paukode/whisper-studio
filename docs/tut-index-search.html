Index & semantic search

Attaching one file lets Claude read one file. Indexing a folder lets Claude search the whole thing and answer from the passages it finds, citing them, without re-reading every file.

You have a folder of CVs, invoices, notes, or a whole codebase, and you want to ask questions across all of it at once. Build an index for the folder, select it in chat, and your answers get grounded in the specific passages that matter, with links back to the source files.

The goal

Search across a whole workspace instead of one attached file. Once a connected folder is indexed, Claude can run semantic search over it and answer from the retrieved passages rather than opening files one by one. The result is faster, cheaper (fewer file reads), and cited: every answer that leans on the index ends with a Sources list of file links.

The loop is: build the index once, keep it fresh as files change, then ask as many questions as you like with the folder selected.

Build an index for a folder

Indexing is driven from the workspace index controls in Settings. You point it at a connected folder and start a build.

  1. Open the workspace index controls

    Go to Settings and find the workspace index section. It lists the folders you have indexed and lets you add a new one. If you have not connected a folder yet, do that first from the workspace IDE.

  2. Start the build

    Pick the folder and start indexing. Whisper Studio walks the tree, turns each indexable file into text, chunks it, and embeds the chunks. Common noise directories like .git, node_modules, venv, and build output are skipped, as are dotfiles.

  3. Watch the status

    A status shows build progress as files are processed, and reports the indexed stats when it finishes, for example the number of files and chunks, and how many changed this run. A build runs one folder at a time, so a manual build and the daily refresh never collide.

Builds are incremental. On the first run every file is embedded; after that, a cheap size-and-modified-time check skips unchanged files, and only new or edited files are re-embedded. Files that have been deleted from the folder are dropped from the index. Re-running a build is therefore fast and safe to do whenever your files change.

The index covers more than plain text and code. Spreadsheets are indexed as a schema plus sample, rich documents (PDF, Word, PowerPoint, HTML, EPUB) are converted to text, images are OCR'd, and audio or video is transcribed locally, so a folder of mixed file types all becomes searchable. See Attachment extraction for the per-type routing.

Ask a question, grounded in the folder

Once a folder is indexed, it becomes selectable as a search target in chat. With one or more indexes selected, Claude searches them first and grounds its answer in what it retrieves.

  1. Select which folders to search

    The composer remembers a per-session choice of which indexed folders to search. By default a new session searches every indexed folder; you can narrow it to one. The selection is saved per session, so it survives a reload.

  2. Ask in plain English

    Type a question and press Enter. Your question (and, for a follow-up, a little of the prior turn) is embedded and searched against the selected indexes before the model is called.

  3. Read the answer and its sources

    The matching passages are injected into the prompt as the source of truth. Claude answers from them and ends with a Sources section, one link per file it drew on. The links point at the exact file and line range so you can open the original.

A worked example over a folder of CVs:

Which candidates have Kubernetes experience, and for how long?

Instead of re-reading every CV, Claude searches the index, pulls the relevant passages from the CVs that mention Kubernetes, and answers from them with a Sources list linking each CV.

The grounding chip

When a turn is grounded in the index, a grounding chip appears over the streamed response. It tells you the search actually ran and what it touched: how many folders were searched and how many passages were retrieved. If the chip shows zero passages, nothing in the selected folders scored above the relevance floor, and Claude falls back to its normal tools rather than inventing a citation.

The chip is your signal that the answer is grounded, not guessed. If you expected a folder to be searched and the chip does not show it, check that the folder is both indexed and selected for this session.

The entity graph

Indexing does more than embed text. As it processes files, it extracts named entities (people, organizations, topics, and so on) from each chunk and collapses case, spacing, and punctuation variants into a single canonical node, so you get one bubble per person or topic rather than five spellings of the same name. Which entity types it looks for is picked automatically per file from its extension, so a CV and a source file get different profiles (business vs code). The local extractor is GLiNER by default; a workspace can switch to the heavier GLiNER2 model from the Index settings panel, and in cloud mode Claude Haiku does the extraction.

Files and entities that co-occur become related, and you can view the result as a graph overlay: nodes for entities, edges for the files and entities that appear together. It is a map of what your folder is actually about and how its pieces connect.

The graph is not just for browsing. At query time, after the vector search finds the closest chunks, retrieval takes one hop along shared entities to pull in related passages, so an answer can surface a relevant chunk that never contained your exact search words but is linked to one that did. Those related passages appear under a "Related passages" heading in the grounding block.

Each backend keeps its own index

Where indexing and retrieval run is set by model mode (cloud, hybrid, or local), and the embedding model differs by mode:

Because different embedders produce incompatible vectors, each embedding backend keeps its own index. Switching model mode never rebuilds the other mode's index, and it also means a folder indexed under one backend is not searchable under another until you index it there. If you switch modes and a folder stops turning up in searches, re-index it once under the new backend.

If a freshly indexed folder returns nothing after a mode switch, it is almost always because the index belongs to the other backend. Re-index the folder once in the current mode and it will be searchable again.

Tips that make this smoother

Index once, ask often

Building is the one-time cost. After that, every question over the folder is a fast search, and re-builds only touch changed files.

Narrow the search

If you have several indexed folders, select just the one you mean for a session so the grounding stays focused on the right corpus.

Trust the chip

The grounding chip tells you how many folders and passages the answer used. Zero passages means the answer was not grounded in the index.

Attach vs index

For a single file you just want to ask about, an attachment is simpler. Index when you want to search across a whole folder. See Working with documents.
