Anatomy of a chat turn

One request, followed all the way through: from the keystroke that starts it to the SQLite row that outlives it. This is the sequence-level companion to Chat & tool orchestration, which covers the same machinery component by component.

Where the pipeline page describes the parts, this page describes the order. Read it top to bottom and you have walked a complete turn. Click a box in the diagram to trace a path, for example click tool_use? to see the branch that leads into the tool loop and back.

The sequence at a glance

A turn is a single POST /api/chat whose response is a long-lived SSE stream, not a one-shot JSON body. Tokens, tool activity, and control events all arrive over that one connection as they happen. The high-level shape is:

  1. The user types and clicks Send

    The SPA adds the user bubble, marks the session streaming, and opens a fetch to /api/chat.

  2. The server builds the turn

    It filters the history, assembles the system prompt, and calls Bedrock as a streaming upstream.

  3. Chunks stream back

    Text becomes tokens; a tool_use stop reason enters the tool loop and continues.

  4. The turn ends and persists

    [DONE] closes the stream, the SPA commits the assistant message, and PUTs the session to SQLite.

The rest of this page expands each of those into what actually runs.

1. The request leaves the browser

When you press Send, src/hooks/useChatStream.ts binds the owning session (so a mid-stream tab switch never redirects where the answer lands), appends your message, sets the streaming flag, and issues the POST. The JSON body carries everything the server needs to construct the turn:

FieldWhat it is
questionThe text you typed (a continuation sends an empty string).
historyPrior user/assistant turns, capped to the last 40 entries. UI-only rows are already filtered client-side.
model / effort_level / verbosityThe selected model and its response controls.
session_idWhich conversation this turn belongs to.
attachment_idsAny files dropped into the composer, resolved server-side into the prompt.
approved_tool_resultPresent only on an approval continuation. Its presence is what makes the server resume a paused turn instead of starting a fresh one.

Only one stream may be in flight per session. A re-send aborts this session's stream but leaves other sessions streaming; the server also returns 409 if a session already has a live stream (a stale slot auto-reclaims after a short timeout). Up to three sessions can be active at once. See Frontend architecture.

2. The server builds the turn

Inside chat_endpoint() (server/chat/routes.py), before the model is ever called, the endpoint prepares the two inputs the model reads: the message array and the system prompt.

  • Latch the session config. latch_session() freezes the model, effort level, and related settings on the first turn, so a mid-conversation Settings change never disrupts an in-flight session.
  • Filter the history. visible_chat_history() (server/infrastructure/sessions.py) drops UI-only rows, most importantly cron_event rows, before the Bedrock messages array is built. Those rows persist on disk and render in the chat, but they never enter the model's context.
  • Assemble the system prompt. resolve_system_prompt() composes base rules, the workspace WHISPER.md, selected memories, and per-session memory, split into cache-aligned static and dynamic parts when prompt caching is on.
  • Resolve @file mentions and attachments inline into the user message, and add index grounding if the session selected workspace indexes.

visible_chat_history is not a security boundary. It is a token-cost and prompt-cleanliness control. A cron firing every ten seconds would grow the prompt indefinitely without it, so cron output is stripped before the prompt is built, not for safety but for economy. The client also filters these rows before sending, so the server filter is defence-in-depth against wasted wire bytes, not against an attacker.

3. Bedrock streams, the loop turns

The Anthropic path calls invoke_model_with_response_stream and reads the upstream SSE. Content blocks are parsed as they arrive, and each maps to an event the client folds into the UI. Two block types drive the turn:

  • A text chunk is yielded straight to the client as a token. The SPA appends it to the streaming bubble in real time (src/hooks/chatStream/sseStream.ts).
  • A tool_use block means the model wants to act. The endpoint collects the tool calls, dispatches them, and feeds the results back so the model can continue. This is the loop-back edge in the diagram.

The turn keeps looping (invoke, stream, act on the stop reason, invoke again) until the model stops asking for tools and the completion gate lets it end. On end_turn the endpoint first runs run_completion_gate (server/goals/gate.py): the WS-I Stop hooks always run, and a WS-E Haiku goal evaluator runs when the goal_loop flag is on with an active goal. If a Stop hook blocks or the goal is judged not achieved, the gate injects [completion gate] feedback and the loop continues; only an unblocked gate reaches [DONE], bounded by goal_max_consecutive_blocks (default 8). For the full round-by-round state machine (budget checks, retries, reactive compaction, parallel batching), see Chat & tool orchestration.

What the client sees on the wire

The SPA switches on a typed SSE payload for each frame. A representative slice:

EventEffect in the UI
textAppends a token to the live assistant bubble.
thinkingStreams an adaptive-thinking block (shown collapsed).
skill / skill_input / skill_resultA tool call started, its parsed input, and its output, rendered as a live tool trace.
approval_requestRaises the approval card and pauses on your consent (see below).
usageUpdates the token counters and cost estimate.
[DONE]Signals the stream is complete.

4. The tool branch: two ways to run

When the model asks for a tool, one of two paths runs, and which one depends entirely on the current permission mode and the tool's risk.

Allow-listed: run it now

A read-safe or already-permitted tool runs immediately through its executor (server/executors/), gated by the sandbox wherever it touches the shell. Its tool_result is appended to the messages and the loop invokes Bedrock again. No user interaction, no pause.

Needs consent: pause on a nonce

A tool that requires approval takes the guarded path:

  1. Pause and stash state

    The turn's continuation state is stored in _paused_sessions[session_id] (in-process), keyed so the exact tool call can be resumed later.

  2. Emit an approval request

    The server issues a server-held nonce and yields an approval_request event describing the action, category, and preview.

  3. Wait for your decision

    The approval card appears. When you click Approve, the SPA calls POST /api/approval/execute (server/approval/router.py) with the action and payload; the server validates and runs it, then reports a truthful outcome.

  4. Resume the turn

    The SPA re-enters /api/chat with an approved_tool_result carrying the real result. The server pops the paused state, fills in that tool_use_id, and the loop continues as if the tool had run inline.

The approval path is the only route that bypasses command validation, and it is gated by a nonce the server holds. The client cannot fabricate consent: there is no client-asserted "approved" boolean the endpoint trusts. The nonce lives on the server, so a real, server-mediated Approve is the single legitimate way past _validate_command. See Approval & permissions and the security model.

5. The turn ends and is persisted

When the model returns end_turn and the completion gate allows the turn to end, the endpoint emits [DONE] and the SSE connection closes. On the client, readSSEStream returns its accumulated result and the hook commits the final assistant message atomically with clearing the streaming flag, so there is never a frame where neither the streaming bubble nor the final message is on screen.

The SPA then persists the conversation with PUT /api/sessions/<session_id> (src/api/sessions.ts), which the server turns into a SQLite UPSERT of chat_history. That save runs under a per-session asyncio lock with merge logic, so background cron_event rows appended by the daemon between renders are never clobbered by the frontend's write. See Persistence and Cron.

Turns that do not simply end

Three common variations do not reach the plain end_turn path:

  • A pending approval leaves the turn paused. The stream still closes with [DONE], but the assistant message carries its tool traces and the approval card, and the real end of the turn is the continuation described above.
  • An ask_user_question collapses one or more question calls into a single tabbed card; the card is the response, and your answer re-enters the endpoint as a continuation, exactly like an approval.
  • The completion gate refuses to end the turn. A Stop-hook block, or an active goal the evaluator judges not-yet-achieved (the goal_loop flag), re-injects feedback and loops again instead of ending. These decisions surface over SSE as goal_eval, stop_hook_block, and goal_cap_reached frames. See The ultracode workflow runtime.

A worked path

Putting it together, a "fix this file" turn that needs a write approval runs like this:

  1. Send

    You type the request and Send. POST /api/chat opens the SSE stream.

  2. Build + invoke

    The server filters history, builds the prompt, and invokes Bedrock. Explanatory text streams back as tokens.

  3. tool_use

    The model emits a ws_write_file tool call. Under the current mode it needs consent, so the turn pauses on a nonce and an approval_request is emitted.

  4. Approve

    You click Approve. POST /api/approval/execute runs the validated write; the SPA re-enters /api/chat with the real approved_tool_result.

  5. Continue + finish

    The model reads the result, streams a confirmation, and returns end_turn. [DONE] fires.

  6. Persist

    The SPA commits the assistant message and PUTs the session to SQLite.
