Web research

Ask for current information and get an answer with sources. Claude decides when to reach for the web, runs a search, reads the best pages, and streams back a normal answer with footnoted links. You do not call a tool by hand.

Claude's own knowledge stops at its training cutoff, so anything that happened recently (a launch, a price, a score, a policy change) is a job for the web. When you ask a question like that, Claude can call two tools: web_search to find current results and web_fetch to read the readable text of specific pages. This tutorial shows how to trigger that, what you see while it runs, and how to check the sources.

You need a Tavily API key for web search. Without one, web_search is unavailable and everything else in the app still works. Setting the key takes two minutes. See Installation (step 3) and Configuration.

Before you start

Web search runs through Tavily, a search API whose free tier covers personal use. You supply the key one of two ways:

  • Environment variable (recommended). Export TAVILY_API_KEY in your shell. The env var overrides any stored value, so rotation is one export away. 
    echo 'export TAVILY_API_KEY=tvly-...' >> ~/.zshrc
source ~/.zshrc
  • Settings UI. Click the gear icon and paste the key into the Tavily API Key field. If both the env var and the Settings value are set, the env var wins.

The seeded config.json leaves tavily_api_key empty, so web search is off until you add a key. web_fetch (reading a specific URL) does not need Tavily, so pasting a link and asking about it works even before you configure a key.

Ask a question that needs the web

There is nothing to enable in chat. Just ask, and phrase it so it is clear you want current facts (and, if you care, ask for sources).

  1. Open a chat and type your question

    Anything time-sensitive works. For example:

    What did Anthropic announce at their last developer
conference? Cite sources.

  2. Claude decides whether to search

    If the answer is stable and well within its knowledge, Claude answers directly. If it needs fresh facts, it calls web_search on its own. You do not pick the tool.

  3. Watch the searching badge

    A small inline indicator appears (a short searching the web... chip) while the tool runs, then disappears. Web search and web fetch are read-only tools, so no approval card interrupts you.

  4. Read the cited answer

    The answer streams in like any other, but with footnoted links to the pages Claude relied on. Follow a link to check the source yourself.

Nudge Claude toward the web when it matters: phrases like "as of today", "the latest", or "cite sources" make the current-info intent explicit. If you want a specific page read, paste the URL and say "summarize this".

The two web tools

Research uses two distinct tools with different jobs. Both are read-only.

ToolWhat it doesNeeds Tavily?
web_searchSearches the web for current, up-to-date information and returns the top results.Yes
web_fetchFetches and reads the readable text of a specific web page, including any URL you paste.No

A typical research turn chains them: web_search finds candidate pages, then Claude calls web_fetch on the most promising ones to read them in full before it answers. When you paste a URL directly, Claude usually skips straight to web_fetch. For the full tool catalogue, see Agent tools.

Reading a page you paste

You do not have to ask an open-ended question. Paste a link and ask about it directly, and Claude reads that exact page with web_fetch:

Summarize https://example.com/some-long-article and pull
out the three main claims.

Because web_fetch does not touch Tavily, this path works even if you have not set a Tavily key yet. Fetched pages are also sanitized before rendering: a prompt-injected page cannot run scripts in the app. See the security model for how web_fetch output is treated as untrusted.

Skills that may trigger alongside

Research often pulls in related skills without you asking for them by name. Each shows up as a small inline indicator when it fires:

Analyze Document

When you upload a paper or report instead of (or alongside) searching, Claude parses it locally and reasons over the text. See Working with documents.

Translate

When a relevant source is not in English, Claude can translate the passage it cites so the answer stays readable.

Web Search / Web Fetch

The two tools above, surfaced as skills. Web Search needs the Tavily key; Web Fetch reads URLs you paste.

Check what was searched

You are never asked to trust a citation blindly. The inline search and fetch indicators are clickable: open one to see the trace, meaning the query that was run and the results (or page text) that came back. That is your audit trail for a claim.

  • Click the searching the web... chip to see the exact web_search query and the results Claude got.
  • Open a web_fetch step to see which page was read.
  • Follow the footnoted links in the answer to read each source directly.

Under the hood, these are ordinary tool calls in the streaming chat loop. The chat pipeline streams a tool event as each search or fetch starts, which is exactly what the inline indicator renders.

Why research never asks for approval

Web search, web fetch, and document parsing are all read-only. They cannot change your files, run commands, or alter git state, so no approval card appears no matter which permission mode you are in. The worst case for a research turn is a wrong answer, not a harmful action, which is why this journey stays friction-free. Approvals only enter the picture once Claude can write, run, or commit, as in the workspace.

Troubleshooting

SymptomLikely cause and fix
Claude answers from memory when you wanted fresh factsAsk more explicitly ("as of today", "cite sources"), or paste a specific URL so it uses web_fetch.
No search happens at all, everTavily key is missing. Set TAVILY_API_KEY or fill the Settings field, then reload. See Configuration.
Search worked yesterday, fails nowRotated or expired key. The env var overrides config.json, so re-export it and restart your shell.
A pasted URL will not loadThe page may block automated fetches or be behind a login. Try a public mirror or paste the relevant text directly.
