Tool dispatch & executors

When the model asks for tools, two small modules take over: one decides how to run the batch safely, the other decides where each call goes. This is the layer the chat loop calls once per round.

The design keeps a clean seam. server/tool_router.py is pure dispatch: it maps a tool name to its handler and returns a result, with no state of its own. server/tool_executor.py owns the lifecycle around that dispatch: batching for parallelism, permission checks, denial tracking, plan-mode blocking, hooks, approvals, and result post-processing. Click a box below to trace one batch, from the model's request to a persisted result.

Two layers, one seam

Splitting dispatch from lifecycle keeps each file readable and makes adding a tool a one-line change in most cases. The router never has to know about permissions; the executor never has to know how a given tool is implemented.

route_tool

Pure mapping in server/tool_router.py. Given a tool name and input, it dispatches to the right handler and returns (output, side_effects). Side effects are SSE payload dicts; a special _pause side effect signals the stream to stop and wait.

execute_tool_batch

Lifecycle in server/tool_executor.py. It partitions the batch, runs pre-execution checks, fires hooks, calls route_tool, and hands completed ToolState objects to process_tool_results for approvals, budgeting, and SSE formatting.

Partitioning for parallelism

A single assistant turn can request several tools at once. Running them all in parallel would be unsafe (two writes could race), and running them all serially would be slow (three independent reads should not wait on each other). So partition_batches() walks the calls in order and groups consecutive concurrent-safe tools into one parallel batch, while every unsafe tool becomes a batch of one that runs on its own:

[read, read, write, read, write]
  -> [[read, read], [write], [read], [write]]

Order is preserved. A parallel batch is awaited with asyncio.gather; a solo batch is awaited directly. Because grouping only merges adjacent safe tools, a write always acts as a barrier: nothing after it starts until it finishes.

What counts as concurrent-safe

The partitioner takes an is_concurrent_safe classifier. For registry executors that answer comes from per-executor metadata declared at registration in server/executors/__init__.py:

Metadata flagMeaningDefault
read_onlyThe executor only reads; no side effects.False
concurrent_safeSafe to run in parallel with other tools.False
destructiveCan cause irreversible changes.False

The defaults are fail-closed. A tool with no registered metadata is treated as not read-only, not concurrent-safe, and not destructive, so an unknown or newly added tool serializes by default rather than racing. You opt into parallelism explicitly with @register_executor(name, concurrent_safe=True).

The per-tool lifecycle

Inside a batch, each tool runs through _run_one, which guards it with several checks before dispatch. The tool's input is cloned three ways so nothing leaks across boundaries: the original stays intact for the Bedrock message history, one clone is passed to hooks (observed, never mutated), and one clone is passed to execution (which handlers may mutate).

  1. Sibling abort

    If an earlier command tool (ws_run_command or run_python) already failed this batch, subsequent non-safe tools are skipped with a "prior command failed" note rather than acting on broken state.

  2. Plan-mode block

    In plan mode, write tools (ws_write_file, ws_create_file, ws_edit_file, ws_delete_file, ws_run_command, ws_merge_worktree) are blocked before they run, and a plan_blocked event lets the UI offer an upgrade dialog.

  3. Denial tracking

    If the user has denied a given tool MAX_AUTO_DENIALS (2) times this session, it is auto-skipped with a message pointing to permission settings. This check is bypassed under bypassPermissions so a stale count from an earlier mode does not linger.

  4. Hooks fire

    The call runs through the blocking-hooks engine (server/hooks/engine.py run_hooks): one PreToolUse pass covers in-process plugins, user shell hooks, and trusted project shell hooks. A PreToolUse hook can not only deny the call but also rewrite its input (updatedInput replaces the call input before dispatch). A plugin denial carrying findings emits a security_blocked event; a plain shell/project denial emits the newer hook_blocked frame. On success, PostToolUse fires and may return additionalContext that is folded back into the tool result the model reads next; on an exception, PostToolUseFailure fires. A separate Stop gate (check_stop_hooks) runs at end of turn across chat, sub-agents, and cron. The old server/infrastructure/hooks.py is now a deprecated shim.

  5. Dispatch

    Only now is route_tool called. Any exception is caught, logged, and returned to the model as a [Tool Error] string, so one failing tool never crashes the turn. Errors from command tools cascade to later writes via the sibling-abort flag above.

Permissions and the approval gate

Most consent logic lives one step later, in process_tool_results. Tools that need approval return a string prefixed with [WS_APPROVAL] carrying JSON; the prefix is purely internal. The executor looks up the registered ApprovalSpec for the action and calls resolve_static_decision() (server/security/permissions.py), whose precedence is:

  1. bypassPermissions - everything allowed, no prompts.
  2. Trusted-skill scripts - a trusted skill running its own bundled command.
  3. Session approvals - category-level pre-approvals from the frontend (for example {"write": "allow"}).
  4. Custom rules, then dontAsk, then acceptEdits (writes only).
  5. auto - defer to the classifier (below).
  6. Otherwise, ask.

An allow runs the action inline through the same ApprovalSpec executor the UI's Yes button uses, so there is a single execution path. A deny returns a "blocked by user" result. Anything else emits an approval_request event (optionally with a risk explanation) and pauses the turn: the stream stops, and after the user confirms, the frontend re-enters /api/chat with the real tool_result.

When a batch pauses on approval, Bedrock still requires a tool_result for every tool_use in the preceding message. The executor fills the remaining calls with "[Not executed]" placeholders so the next request's message array stays well-formed, then re-issues them after the approval resolves. See Approval & permissions.

Auto mode and the classifier

In auto mode, when nothing static resolves the decision and a Bedrock model is available, classify_tool_call() asks a fast model (Haiku, falling back to Sonnet) to judge the call and returns allow or ask. The classifier is only invoked for actions that actually need approval; read-only tools never reach it, and the offline local path stays Bedrock-free because model_id is empty there.

Results back to the loop

Once a batch completes, process_tool_results turns each ToolState into a Bedrock tool_result and a set of SSE events. Along the way it handles a few special output shapes signalled by internal string prefixes:

[WS_APPROVAL]
Needs consent; resolved through the precedence chain above.
[WS_WORKSPACE_PROMPT]
A write fired with no connected workspace; a folder picker is shown and the turn pauses.
[WS_PREVIEW_IMAGE]
A screenshot; returned as an Anthropic image content block (never budget-truncated) so the model sees pixels, not base64 text.

Ordinary string outputs are run through budget_fn to cap oversized results, a truncated preview streams to the SPA as a skill_result event, and the full output becomes the tool_result content the model reads on the next round.

The tool catalogue

Whisper Studio's full tool pool runs on the order of 80-plus core tools plus the skill-backed tools - grouped by purpose: workspace & files, git, LSP, web, memory, tasks & scheduling, sub-agents, skills / notebooks / MCP, config & interaction, and live preview. The interactive chat loop no longer ships all of those schemas every turn. With progressive tool disclosure (feature flag progressive_tools, default on) it advertises only a curated core set plus whatever this session has activated; every other tool costs a single one-line entry in a deferred index inside the cached system block, loadable on demand via tool_search. That trims roughly 60-75% of tool-schema tokens per turn. The full ~8K-token pool now survives only off the interactive path - cron and the agent runtime still send every schema (assemble_tool_pool defaults to progressive=False). See Progressive tool disclosure below, and note the pool is still deduplicated and mode-filtered upstream in the chat pipeline.

Cloud Claude is now trimmed automatically by progressive disclosure (core plus session-activated tools); on-device model modes can trim the exposed set further with the local Tools dropdown. The dropdown is no longer the only mechanism that shrinks the advertised set. For the complete, grouped list of every tool the assistant can call, see the Agent tools reference.

Progressive tool disclosure

On the interactive chat loop the advertised pool is partitioned rather than sent whole. The split is computed by partition_pool in server/chat/tool_partition.py and wired through assemble_partitioned_pool (server/chat/tool_pool.py) in the chat pipeline:

CORE_TOOLS
A curated always-advertised set (~34 names) in tool_partition.py: workspace primitives, the discovery tools (tool_search can never be deferred - it is the way back), the todo and background-task tools, orchestration, and the workflow / CI tools when present.
Session activation
Tools this session has pulled in via tool_search or history replay, tracked in server/chat/tool_activation.py and appended after the core prefix in activation order.
Deferred index
Every remaining tool becomes one line (build_deferred_index, server/chat/tool_index.py) in the cached system block - a name plus its first sentence - loadable on demand.

Ordering is cache-stable: the core prefix keeps the catalogue's sorted order and a core_count breakpoint is placed on it, so a new activation invalidates only the appended tail of the tools cache block, not the core prefix. The flag progressive_tools defaults on and applies to the interactive chat loop only; cron and the agent runtime still receive the full pool (assemble_tool_pool defaults to progressive=False).
