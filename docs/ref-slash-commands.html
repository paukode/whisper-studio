Slash commands

Every / command available in the chat input. Type / at the start of an empty composer to see the full list inline, with argument hints and live autocomplete.

Slash commands are shortcuts the composer intercepts before anything is sent to the model. They tune the session (model, thinking depth, response length), run side tasks (a subagent, a one-off side question), or drive the UI (open Settings, clear the view, rename the session). Most take effect instantly and confirm with a small toast; a few open a dialog. They are handled client-side in src/hooks/useSlashCommands.tsx, with the list itself defined in src/components/chat/chatInputConstants.ts.

A command only fires when your message begins with /. Anything that is not a recognised command is treated as a normal chat message, so a stray slash never eats your text. Run /help at any time for the same reference inside the app.

Model & reasoning

These four commands change how Claude thinks for the rest of the session. They mirror the controls under the composer, so a command and a click stay in sync.

/model

/model <name> switches the model for every subsequent turn in this session. The valid names come from your configured model list, so they track whatever you have enabled, for example haiku, sonnet, opus, or a pinned version like opus4.8. On success you get a toast confirming the version label ("Model set to Sonnet 4.6"). Pass an unknown name and the composer replies with the exact usage string listing the models you can pick.

Switching to a model with a stricter data-retention policy routes through a consent screen first, so the change only lands after you confirm. See Model modes for the cloud, hybrid, and local backends behind each name.

/effort

/effort <level> sets thinking depth for models that expose it (the Opus tier). The accepted levels are low, medium, high, extra, max, and ultracode. The raw API name xhigh is accepted as an alias for extra. ultracode is the top mode: it pairs the highest reasoning effort with dynamic-workflow orchestration (see The ultracode workflow runtime). Each model advertises its own set of levels, so the validator is model-aware:

  • Ask for a level the model supports and it is applied, with a toast naming the new level.
  • Ask for a valid level the current model does not expose and it is clamped to the nearest lower one, with a note explaining the adjustment.
  • On a model with no effort levels at all, the command reports that and does nothing.

/plan

/plan toggles plan mode: a read-only stance where Claude investigates and proposes a plan without writing files, running commands, or taking other write actions. Run it again to leave plan mode. It is the same dial described in Permissions & approvals, so any write step still surfaces an approval card once you leave plan mode.

/verbosity

/verbosity <level> sets the response length, the same three-level control that sits under the composer. Pass brief for short, direct answers, normal for balanced detail, or detailed for thorough, expansive ones (low, medium, and high work as aliases). A toast confirms the level you set; call it with no argument to see the usage string.

/memory

/memory on and /memory off flip cross-session memory, and /memory status reports whether it is currently on. Memory is what lets Claude carry durable facts between conversations; see the Memory & WHISPER.md guide for what gets remembered and how to curate it.

/goal

/goal <text> sets a session goal that the goal loop keeps working toward: after each turn a cheap evaluator judges whether the goal is met, and if not the turn auto-continues (capped so it cannot run away). A goal banner appears above the composer while one is active. /goal clear ends it. Both need an open session, and a bare /goal just shows its usage hint. The loop is gated by the goal_loop feature flag; see Feature flags.

Side tasks & agents

These run work alongside the main conversation without derailing it.

/btw

/btw <question> asks a quick side-channel question that does not enter the main transcript. It sends only your question plus a few recent messages for context, streams the answer into a small popup, and leaves the conversation history untouched. Use it for a fast aside ("btw what does this acronym mean") when you do not want to spend the main thread's context on it. With no question, the command just shows its usage hint.

/subagent

/subagent <task> spawns a focused one-shot subagent that works in the background with the full tool loop: web search and fetch, workspace access, code, and any enabled MCP tools. It does not block the composer, so you can keep chatting while it runs. Its live progress renders in a report card inside the chat, and its final answer is posted there when it finishes. Press Stop on the card to cancel a run in flight. For coordinating several agents at once, see Sub-agents & teams.

/workflow

/workflow <saved-name> launches a saved workflow by name: a deterministic script from the ultracode workflow runtime that orchestrates agents in the current session. The run starts immediately and renders a workflow card in the chat that tracks its phases, and it does not block the composer. It needs an open session, and if no saved workflow matches the name you get an error toast prompting you to check that it is saved. See The ultracode workflow runtime for how workflows are authored, previewed, and resumed.

Files & tools

Colon-style commands open an inline picker as you type after the colon, so you can search rather than remember exact names.

/file: and @file

/file:<path> references a workspace file so Claude sees its contents inline. There are two behaviors depending on whether you add a question:

/file:path
With no trailing question, the file is attached to the composer as a chip (uploaded in the background) and you keep composing.
/file:path a question
With a question, your message and a working indicator appear immediately while the file is fetched, converted, and uploaded in the background, then the turn sends.

The same file reference is also available as an @file: mention from anywhere in the message, alongside @skills: and @mcp:. Documents are extracted to text (large ones are capped) and images become vision blocks; see Attachment extraction for the type-aware pipeline. Supported inputs include images, PDF, Office docs, audio, video, code, and text.

/skills: and /mcp:

/skills: browses the available skills and opens the Skills view. /mcp: browses the tools exposed by your connected MCP servers. Both open a searchable submenu as you type after the colon, which is the discoverable way to see exactly what is wired up in the current session.

The colon matters. /file:, /skills:, and /mcp: are submenu commands: the argument comes right after the colon, and the picker dismisses once a space begins the message body. See Agent tools for the full catalogue those tools come from.

Conversation & session

Manage the transcript and the session record.

/export

/export downloads the current conversation as a plain-text file (chat-export-<timestamp>.txt), with each message prefixed by its role and separated by rules. It reads the messages currently in view, so pair it with a fresh session for a clean transcript.

/clear

/clear wipes the visible chat so you can start fresh in the same tab. It clears the on-screen messages only; the stored session still holds its history in SQLite. To remove a session for good, delete it from the sidebar instead.

/rename

/rename <new title> sets a custom title for the current session and marks it as user-chosen, so the automatic title generator will not overwrite it later. Everything after the command word becomes the title. With no title, you get the usage hint.

/workspace

/workspace opens the native folder picker to connect a workspace directory, which is what unlocks the file tree, editor, terminal, and the file-aware tools. The workspace IDE tutorial walks through what becomes available once a folder is connected.

Utilities & diagnostics

App-level helpers that do not touch the model.

/theme

/theme <name> switches the interface theme immediately and remembers your choice. Accepted values include auto, dark, light, and the accessibility variants such as dark-high-contrast, light-high-contrast, dark-daltonized, and light-daltonized. auto follows your system's light or dark preference.

/notify

/notify requests browser notification permission (or tells you the current state if it is already granted or blocked). With notifications enabled, long-running work like a subagent or a scheduled task can alert you when it finishes.

/settings

/settings opens the Settings panel. It is the keyboard-first way to reach model defaults, API keys, and the rest of the fields documented in Settings & config keys.

/doctor

/doctor runs an automated diagnostics pass and shows the results in a dialog: a set of checks (dependencies, AWS access, and related environment health) each marked as ok, warn, or fail. It calls the /api/doctor endpoint served by server/doctor.py. Run it first whenever chat is not responding or a model call errors.

/ci

/ci [branch] watches the branch's latest GitHub Actions run to a terminal state, streaming its status into a card in the chat. /ci autofix [branch] goes further: it reads the failing logs, posts a diagnosis, and emits an approvable workflow that fixes the problem and then runs verification, using the ultracode workflow runtime. Both need an open session and the GitHub CLI (gh) installed and authenticated; without it the command reports that CI is unavailable. See The ultracode workflow runtime.

There is no /hooks command. Blocking hooks (PreToolUse, PostToolUse, and the Stop gate) are managed in the Settings panel, not from the composer. See Settings & config keys and the Security model for how project hooks are trusted.

/help

/help opens a dialog listing every command grouped by category (General, AI & Model, Workspace, Settings, Debug) - the AI & Model group is where /workflow, /ci, and /goal appear - each with its icon and one-line description. It is generated from the same command list this page documents, so it always matches your build.

Names track your config. /model and /effort validate against the models and effort levels you actually have enabled, so the exact options you see may differ from the examples here. When in doubt, run the command with no argument and read the usage hint it returns.
