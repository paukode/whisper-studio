Frontend architecture

The whole UI is one React 19 single-page app served from the same local origin as the API. It is built around one idea: each chat session is a self-contained runtime that keeps streaming, recording, and saving whether or not you are looking at it.

Where the chat pipeline is the backend state machine, this page is its client. Three pieces carry most of the weight: AppShell.tsx (layout plus startup), the session runtime registry in src/stores/sessionRuntimes.ts, and the useChatStream hook that consumes Server-Sent Events. Click a box in the map to trace where data flows.

The shell and the startup sequence

src/components/layout/AppShell.tsx is the top-level component. It renders the four-region layout (sidebar, chat, an optional transcript pane, the workspace IDE, and a right-side dock) and it owns the one-time boot sequence. There is no separate layout engine: every visible split is a ratio-flex Splitter, so no pane is ever pinned to a pixel width and surviving panes redistribute through flexbox when you open or close one.

The shell also mounts a persistent bottom status strip, src/components/layout/AppStatusBar.tsx (WS-H), that reads only from existing stores to show the active model, effort, permission mode, git branch with dirty and ahead/behind counts, a live context-window meter, per-turn tokens and cost, and a running background-task pill. The Header carries its own running-count pill that opens the BackgroundTasksPanel.

On mount, the shell fires the startup fetches in parallel:

  • Load config and models. loadConfig() reads the merged runtime config; loadModels() fetches the model list only. On-device weights are deliberately not eager-loaded at startup, so launch stays fast and memory stays free until you actually pick a local model. See Model backends & modes.
  • Load data retention, skills, and MCP servers. loadDataRetention(), loadSkills(), and loadMCP() populate the settings store; the global tool store is then filled via fetchSkills() and fetchMCPTools() so the skills dropdown and autocomplete have something to filter.
  • Restore sessions. loadSessions() lists saved sessions, then the persisted currentSessionId is restored and rehydrated through switchSession(). A stale persisted id is dropped so the next session prompt starts fresh. This is what survives the reload-mid-chat case (a Vite HMR cascade that forces a full reload).
  • Check workspace status. A GET /api/workspace/status recovers a previously connected repo after a refresh, flipping the UI back to the connected state and reopening the workspace panel.

The shell also wires the global keyboard shortcuts (Cmd/Ctrl+, for Settings, Cmd/Ctrl+B to toggle the sidebar, Cmd/Ctrl+K for the command palette) and the ESC kill switch, which is armed only while a stream is running or subagents are alive so Esc keeps its normal meaning everywhere else.

Per-session runtimes

The registry in src/stores/sessionRuntimes.ts is the heart of parallel sessions. Every live session owns a RuntimeEntry: its own chat store, its own transcription store, its own long-lived cron/notification EventSource, an in-flight chat AbortController, and self-saving subscriptions.

The design rule is simple: nothing is cleared on a switch. The UI binds to the active session's stores and re-points on switch; background sessions keep streaming, keep receiving cron events, and keep saving themselves. This is why you can start a long answer in one session, switch to another, and come back to find the first one finished.

Two different caps, both set to 3, do two different jobs. MAX_LIVE_RUNTIMES (in sessionRuntimes.ts) is how many runtimes stay resident in memory. MAX_ACTIVE_SESSIONS (in useChatStream.ts) is the user-facing ceiling on how many sessions may be busy (streaming, mid-approval, or recording) at once.

Eviction is LRU over idle, hydrated sessions. maybeEvictIdle() flushes a save first, then drops the least-recently-used runtime and closes its channels. isEvictable() guarantees it never evicts the current session, the recording owner, a streaming session, one that is mid-approval, or one still hydrating. Because busy sessions are never evicted, the cap is soft: the registry may briefly exceed three until one goes idle.

Access is through a small set of getters and hooks:

getChatStore(sessionId)
The imperative getter stream code uses. Bind it once at send time so the stream keeps writing to its owning session regardless of switches.
useActiveChatStore(selector)
The reactive hook UI components use. Built on useSyncExternalStore, so a session switch swaps subscription and snapshot in a single commit.
countActiveSessions(excluding?)
Counts busy sessions for the parallelism ceiling; viewing idle sessions is never counted.
dropRuntime(sessionId)
Teardown: aborts the in-flight stream, closes the EventSource, and unsubscribes.

Each runtime also carries its own cron EventSource to /api/sessions/{id}/events, so a background session still receives its cron events and raises an unread badge on its sidebar row. Stale streams (after a laptop wake or a network restore) are recycled on online and tab refocus so leaked connection slots do not starve new /api/chat requests.

The store layer

State is Zustand v5. Two kinds of store live side by side. Per-session factories (createChatStore, createTranscriptionStore) are instantiated once per runtime, so each session's messages and segments are fully isolated. Singletons are shared app-wide:

StoreKindOwns
chatStoreper-sessionMessages, streaming buffers, tool traces, pending approvals, team reports.
transcriptionStoreper-sessionDiarized segments and speaker names for that session's recording.
sessionStoresingletonThe session list, currentSessionId, switching, titling, and saves.
workspaceStoresingletonFile tree, editor tabs, and dirty state for the connected repo.
uiStore, layoutStore, dockStoresingletonPanel visibility, persisted split fractions, and the right-side dock panels.
settingsStoresingletonConfig, model list, selected model, effort level, verbosity, data retention.
toolStore, taskStore, recordingStore, cronUnreadStore, subagentStoresingletonSkill/MCP catalogue, the live task list, the shared mic, cron badges, and running subagents.
backgroundTaskStore, workflowStore, ciStore, goalStoresingletonThe background-task registry + runningCount (WS-A), workflow runs keyed by run_id (WS-D), CI watches keyed by task_id (WS-J), and the active goal + last verdict per session (WS-E).

Zustand v5 selector rule. Never return a fresh object or array from a selector. A new reference on every render trips Zustand's equality check and drives a React error #185 render loop that passes CI but crashes at runtime. Select primitives and derive composite values with useMemo in the component.

Chat streaming

The useChatStream hook in src/hooks/useChatStream.ts owns the send path. Its central contract is bind once, never let go:

  1. Resolve the owning session

    send() reads currentSessionId once (creating a session if there is none), then does every store access through getChatStore(activeSessionId). A mid-stream session switch changes nothing about where this stream writes.

  2. Check the parallelism ceiling

    Activating a session beyond MAX_ACTIVE_SESSIONS gets a toast, not a queue. Re-sends and approval continuations in an already-active session always pass. Any in-flight stream for this session is aborted first.

  3. Build the request body

    The user message is appended, history is capped to 40 entries, and UI-only rows (role='cron_event') are filtered out before the wire. The body carries the question, transcript, model, effort level, verbosity, session id, attachment ids, and the selected search indexes.

  4. POST and consume

    A fetch('/api/chat') streams the response into readSSEStream. A 409 means the server's same-session guard is holding the slot; the hook surfaces the server's guidance as a toast instead of stacking two streams.

  5. Commit atomically

    On stream end, streaming state is cleared and the final assistant message (text, tool traces, team reports, usage, any artifact or plan) is added in a single store update, so there is never a frame with neither the streaming bubble nor the final message.

Routing SSE events

src/hooks/chatStream/sseStream.ts holds readSSEStream, the parser. It reads the response body, splits on newlines, strips the data: prefix, and validates each payload against a Zod schema (SSEEventDataSchema) before dispatching it to the owning session's store. Validation failure logs loudly but still passes the raw payload through, because some event types are intentionally loose. The last 200 events are kept on window.__lastSSE for DevTools.

Each event type routes to a store action:

EventEffect
textAppended to the streaming buffer via appendStreamToken.
thinking / thinking_start / thinking_stopDrives the adaptive thinking block and its elapsed timer.
skill / skill_input / skill_resultStarts, fills, and completes a live tool trace; also refreshes the Git panel and file tree after mutating tools.
approval_requestEnqueues a pending approval, or auto-accepts/denies it from session memory.
user_questionCollapses parallel ask_user_question calls into one tabbed card.
team_progress / team_resultsFolds live sub-agent activity into a turn-local team report.
usageUpdates token counters and the cost estimate; also carries context_used/context_max for the status-bar context meter.
groundingRecords how many folders were searched and passages injected this turn.
cron_eventAppends a live cron card that renders identically to the persisted, resumed one.
hook_blocked / stop_hook_feedback / stop_hook_blockSurfaces a blocking hook's veto (a Pre/Post/Stop gate) and its feedback into the turn.
goal_eval / goal_cap_reachedUpdates goalStore with the completion-gate verdict and attempt counter for the active goal.
workflow_preview / workflow_startedRenders the ultracode workflow phase-preview approval card, then its inline run card.

Three more frames route on a different channel. task_event, ci_progress, and ci_result arrive on the per-session EventSource in src/stores/sessionRuntimes.ts - not the chat POST stream - and feed backgroundTaskStore and ciStore, so a detached job keeps updating even when its session is not the one on screen.

An approval is not a dead end. sendApprovalContinuation POSTs the accept or deny back to /api/chat and feeds the continuation response straight back into readSSEStream; the two functions are mutually recursive, which is why they share one module. The ESC kill switch stops the viewed session's stream and every running subagent synchronously, and a wasKillFinalized guard stops the read loop from resurrecting an answer after a kill.

Rendering safety

All Markdown passes through src/utils/sanitizeHtml.ts before it reaches the DOM. renderMarkdownSafe is the single path that produces HTML for React's raw-HTML escape hatch: markdown -> marked.parse -> DOMPurify.sanitize. DOMPurify runs with USE_PROFILES: { html: true }, which permits standard elements but strips <script>, on* handlers, and javascript: URLs.

This matters because a web_fetch can pull a prompt-injected page into the model's context. Without sanitization, a payload like <img onerror> would run inside the SPA's same-origin context. A separate sanitizeHtml helper covers already-HTML content that never went through marked, such as a .docx converted for the document viewer. See the security model.

The rest of the glue

  • Providers. The app is wrapped with TanStack Query (for cached, non-streaming server reads) and a theme provider that mirrors the light/dark toggle you see in these docs.
  • Validated API client. The client in src/api/client.ts validates responses with Zod schemas, so a shape drift shows up as a clear error rather than a silent undefined deep in a component.
  • Reference-counted recording. The chat-input mic and the meeting recorder share a single MediaStream through the recording controller, so two features can hold the microphone without fighting over the device. See Transcription & diarization.
