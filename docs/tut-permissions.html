Permissions & approvals

One dial decides how much Claude can do on its own: from a read-only planner that never touches your disk, to a hands-off mode that runs everything. Under it sit approval cards you click, per-tool rules you write, and a small set of guards that never turn off.

When the model asks to use a tool, that call does not just run. It is resolved to one of three outcomes first: allow (run now), ask (pause and show you a card), or deny (refuse). What decides the outcome is your permission mode plus any per-tool rules you have set. Click a box below to trace where a call goes.

The one control: the Mode dial

The single dial for all of this is the Mode dropdown in the chat input, right where you type. There is no separate permissions screen to open before you can start; you change how much latitude Claude has mid-conversation, per keystroke if you like. The setting is stored server side in data/permissions.json and applies to the tool loop on the very next turn.

Two shortcuts do the same thing without leaving the keyboard: the /plan slash command toggles read-only plan mode, and per-project overrides live in <workspace>/.whisper/settings.json. Everything on this page also applies to sub-agents and scheduled jobs, since they run through the same gate.

The six modes

Pick the row that matches how much you trust the current task. The default is a good starting point: you can read anything, but any change to your files or system waits for a click.

ModeBehaviour
defaultReads run freely; writes ask for approval. The everyday setting.
autoReads run freely; a classifier decides per-tool, per-call whether a write is safe enough to run or should ask. Tune it under the Auto Mode tab in Settings.
planRead-only. Claude investigates and proposes a plan but never executes. File writes, deletes, commands, and merges are hard-blocked before they can dispatch.
acceptEditsFile edits and new files auto-approve; deletes and commands still ask. Good for a focused editing session.
bypassPermissionsEverything runs without prompts. Fastest, least guarded. Use only in a throwaway or fully trusted workspace.
dontAskReads auto-allow; writes are silently denied. Claude keeps working on read-only tasks and simply cannot change anything, without stopping to ask.

Reads never prompt, in any mode. Listing files, reading a file, grep, and glob are treated as read-safe and always run (even under dontAsk). The distinction that actually matters is read vs write: a "write" is anything that changes files, runs a command, or alters Git or config state. That category is what each mode loosens or tightens.

A note on plan mode

Plan mode is the strictest and the one people reach for most before a risky change. The core write tools (ws_write_file, ws_create_file, ws_edit_file, ws_delete_file, ws_run_command, and worktree merges) are blocked before dispatch, so there is no card to approve and nothing can slip through. Claude can still read your whole project and hand you a plan; you switch to default or acceptEdits when you are ready to let it act. Toggle it fast with the /plan command.

Approval cards: what "ask" looks like

When a call resolves to ask, the turn pauses and an approval card appears inline in the chat, right where the tool would have run. The card shows the exact thing about to happen, not a vague label, so you are approving a concrete action:

Card kindShows
DiffA line-by-line diff of the file edit, before and after. File-write cards also offer an Undo that restores the pre-write contents.
CommandThe exact shell command and the directory it will run in.
ListThe set of paths a bulk action would touch (for example a delete).
TextA plain summary for anything that does not fit the above.

Each card carries a short risk hint and, when several actions are queued, a "N more pending" counter. You have four buttons:

  1. Yes

    Approve just this one call. The turn continues from exactly where it paused.

  2. Yes, all <category>

    Approve this and every later call in the same category (for example, all writes) for the rest of the session. Handy once you have seen a few and trust the direction.

  3. No

    Deny this one call. Claude sees the denial and can adjust or ask something else.

  4. Block <category>

    Deny this and every later call in that category for the session, without asking again.

These session-wide "allow all" and "block all" choices sit above the mode: they are the first thing the resolver checks after bypassPermissions, so once you have said "yes to all writes" you will not be re-prompted even in default.

The approval is enforced server side, not in the browser. A risky action carries a one-time nonce and the turn stays paused until the client confirms it; the old client-asserted "user_approved" flag was removed. You cannot approve something the server did not gate, and a page cannot fake an approval. See Approval & permissions for the nonce mechanics.

Per-tool rules: override any mode

Modes are broad. When you want a specific, always-on decision for a specific tool or command, write a per-tool rule under Settings › Permissions. Rules are checked before the mode falls through to its default, so they win over whatever the dial says (with one exception, below). Each rule targets a tool by name, matches its input, and pins an action:

FieldMeaning
toolThe tool name to match, or * for any tool.
pattern / prefixA glob (pattern) or a literal starts-with (prefix) matched against the tool's key field: the path for file tools, the command for shell, the branch for git, and so on.
actionOne of allow, ask, or deny.

For example, to let Claude run any git command without prompting but always confirm deletes:

{ "tool": "ws_run_command", "prefix": "git ", "action": "allow" }
{ "tool": "ws_delete_file", "pattern": "*",     "action": "ask"   }
{ "tool": "ws_delete_file", "pattern": "*.log", "action": "deny"  }

Rules are evaluated top to bottom and the first match wins, so order matters. The Settings panel warns you when a broad earlier rule shadows a narrower later one so you can reorder before it surprises you.

Resolution order, top to bottom: bypassPermissions (absolute) → a trusted skill script → your session-wide "all this session" choice → your custom rules → the mode's own default. So bypassPermissions overrides even a deny rule, but every other mode respects your rules first.

The guards that never turn off

Modes and rules control the consent layer: whether you are asked. Underneath them sit two guards that apply no matter which mode you pick, including bypassPermissions. They are the real security boundary; the layers above are convenience.

Everything runs sandboxed

Every shell command and every Python run is wrapped in the macOS kernel sandbox (sandbox-exec), which enforces filesystem and network policy at the OS level. Approving a command chooses to run it; it does not unwrap the sandbox around it.

Destructive patterns are blocked

A command validator refuses a small set of clearly destructive shapes before they can run at all: rm -rf / at an absolute path, mkfs, dd to a device, redirects into /etc/, and hidden command substitution like $(...) or backticks. These are denied regardless of mode.

The validator also refuses access to sensitive paths and files, strips control characters and deceptive Unicode whitespace, and caps how many commands you can chain in one call. It is defence-in-depth: it catches the easy cases without being what stops a determined attacker. That job belongs to sandbox-exec. And of course, reads never prompt because they cannot change anything.

Which mode should you use?

default
Most of the time. You watch each change land.
plan
Before a big or unfamiliar change: get the plan first, then switch and let it act.
acceptEdits
A long editing pass where you trust the edits but still want a beat before deletes and commands.
auto
You want fewer prompts but not zero; let the classifier wave through the obviously safe writes.
dontAsk
A read-only exploration where you never want Claude to write, and never want to be interrupted saying so.
bypassPermissions
A scratch workspace you do not mind. Never on anything you care about.

How this fits the pipeline

Permission checks happen inside the tool loop, not before the turn. Each round the model may ask for tools; the executor partitions them, evaluates each against your mode and rules, and for anything that resolves to ask it issues a nonce, pauses, and waits. This is why an approval can appear mid-answer and why continuing after one picks up exactly where it left off. See Chat & tool orchestration for the surrounding state machine.
