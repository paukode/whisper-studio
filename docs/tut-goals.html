Goals & autopilot

Most of the time you just type a message. But when a task has many steps and you want Whisper Studio to keep going until it's genuinely done — not stop halfway — you give it a goal. This page shows you which to reach for, and how to set a goal so it runs to completion.

There is no mode to switch and nothing to configure. You decide how much autopilot a task deserves by how you ask: a plain prompt for something quick, /goal for something that should run until it's finished, or a workflow for wide, parallel work. Click a box in the diagram to see when each fits.

Three ways to ask

They are not different products — they are how much you want Whisper to push on its own before it stops and hands back to you.

Reach forWhenHow
A plain prompt A question, or a small, well-scoped change. One turn or a short tool loop is enough. This is the default — don't over-reach. Just type and press Enter. See Your first chat.
A goal A multi-step task you want run to completion — "keep going until this actually works" — where the model might otherwise stop as soon as it has done something. /goal <text>. A completion gate checks each turn and continues until the goal is met (capped). See below.
A workflow Wide, parallel, or adversarially-verified work: audit twenty files, generate five designs and judge them, migrate a whole directory. Ask in ultracode mode, or /workflow <saved-name>. The model writes a script that fans out subagents; you approve the preview. See the workflow runtime.

A rule of thumb: if you'd normally have to nudge with "keep going" or "you're not done yet", set a goal instead and let the completion gate do the nudging. If you'd want several independent things done at once, that's a workflow.

Give it a goal

A goal is a one-line description of what "done" means for this stretch of work. Whisper keeps it in view and, at the end of every turn, asks itself whether the goal is met yet.

  1. Set the goal

    Type /goal followed by the objective, for example /goal make the login page pass tsc, eslint, and its tests. A goal banner appears above the composer so you always see what it's working toward.

  2. Give it the first instruction

    Then prompt as normal — "start on the failing test" — or just say "go". The goal is context for every turn, not a replacement for your first message.

  3. It keeps going until it's done

    When the model stops, the completion gate runs a quick check instead of ending the turn: a cheap evaluator (and any Stop hooks) decide whether the goal is achieved. If not, it feeds back what's missing and the turn continues — so you don't have to keep typing "keep going".

  4. Clear it when you're done

    /goal clear ends the goal. It also clears itself naturally: the loop is bounded by a consecutive-block cap (default 8) so a stuck task can never run away, and a confident "can't do this" ends the turn rather than looping.

Goals shine when "done" is checkable. "Make the build pass" or "all tests green" gives the evaluator a real signal — the verify-change skill runs the actual gate and reports VERIFY PASS / VERIFY FAIL, which the evaluator weighs above any confident-sounding summary. Phrase your goal as an outcome it can verify.

Full-stack & UI work

This is where a goal earns its keep. For a change that touches the interface, "verify" isn't just running tests — Whisper drives a live browser preview on its own: it starts your dev server, loads the changed page, reads the console for errors, clicks through the changed flow, and takes a screenshot, then decides pass or fail. And because a goal isn't "done" until that passes, it keeps re-checking the page and fixing until the UI actually renders and behaves — you never have to say "now check it in the browser".

So phrase a UI goal as something it can see: "the login form submits and shows the dashboard" or "the settings toggle persists after a reload", not "make the form nicer". You'll want the permission mode set the way you like first, since a goal may run for several turns editing and reloading. The mechanics — and the two caveats (the browser tooling has to be installed, and parallel UI checks each spin their own dev server) — are in Verifying the UI in the loop.

When a goal is overkill

Don't set a goal for a plain question or a one-line edit. A goal adds a small evaluator check at the end of each turn, which is wasted effort when there was never any doubt the task was done. Keep goals for genuinely multi-step, "run until it holds up" work. And you can always steer mid-goal — your next message is still just a message; the goal is the backdrop, not a cage.

Scaling past one worker

A goal keeps one line of work going. When the work is naturally parallel, two other tools take over:

Sub-agents & teams

Spin off a background helper with /subagent, or let Whisper run a small team — each with its own tool loop.

Ultracode workflows

For wide or adversarial work, the model writes a deterministic script that fans out many subagents under a budget, and you approve it before it runs.
