Requirements

Everything you need in place before installing: the operating system, four command-line tools, an Amazon Bedrock account, an optional web-search key, and the disk and memory each model mode asks for.

Whisper Studio targets macOS. Most of the heavy lifting (the venv, Node, ffmpeg, ripgrep, the model weights) is done for you by setup.sh, so this page is short. Budget 15 to 25 minutes for a first install, most of it spent downloading dependencies and model weights.

Platform. Whisper Studio is a macOS app. Transcription uses the MLX runtime (Apple Silicon), the sandbox is sandbox-exec(5), and setup.sh leans on Homebrew for system packages. There is no supported Windows or Linux path today, though the Python backend itself is largely portable.

Required command-line tools

You need exactly four tools present before you run the setup script. Everything else the app depends on is installed for you (see What setup.sh installs below). Open Terminal and run each check command; if it prints a version, you are set.

ToolWhy you need itCheck command
GitClone the repository (and used at runtime for the workspace Git integration and skill/plugin cloning).git --version
Python 3.10+The FastAPI backend. It uses match statements, asyncio.TaskGroup, and modern type syntax, so 3.10 is a hard floor.python3 --version
HomebrewInstalls the rest (ffmpeg, ripgrep, and Python or the AWS CLI if they are missing).brew --version
AWS CLIAuthenticates to Amazon Bedrock for the Claude chat responses. You run aws configure once.aws --version

If Homebrew is installed but Python or the AWS CLI is not, setup.sh can install them for you. Python 3.10+ is the one gate the script enforces up front: if your python3 is older or missing, it offers to install python@3.12 via Homebrew (in an interactive terminal) and persists it on your PATH. The fastest way to get all four in one line:

# Install Homebrew first if `brew --version` fails:
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
# then install Git, Python, and the AWS CLI:
brew install git python@3.12 awscli

You do not need to install Node.js. The Installation guide has copy-paste blocks for Homebrew, Python, and AWS if any check above fails. Come back here for the hardware and AWS specifics.

What setup.sh installs for you

The point of the required-tools list being so short is that setup.sh provisions everything else into an isolated venv/, so your system Python and global toolchain stay untouched. On a fresh run it:

  • Creates the venv/ and installs the Python dependencies from requirements.txt.
  • Installs ffmpeg and ripgrep via Homebrew (ffmpeg decodes audio for Whisper; ripgrep backs the workspace search tools).
  • Installs Node.js into the venv via nodeenv (the version is pinned in .nvmrc), then runs npm install and builds the frontend. You never install Node yourself.
  • Fetches the Playwright Chromium build used by the live-preview browser tools.
  • Downloads the model weights for your chosen model mode into models/.

Homebrew is therefore doing double duty: it is one of the four things you install by hand, and it is what the script reaches for to fill in ffmpeg, ripgrep, and (if needed) Python, Git, or the AWS CLI. On a machine without Homebrew, the script warns and some of these steps are skipped. See Installation for the full run-through and the setup flags.

Amazon Bedrock access

Chat responses come from Claude on Amazon Bedrock, so you need an AWS account with Bedrock access and the Claude model you want enabled in your region. Two things trip people up here, both region-related:

  1. Enable model access. In the Bedrock console, request access to the Claude models you plan to use. Access is granted per model, per region.
  2. Match the region. Whatever region you set with aws configure must be the same Bedrock Region you enter in Whisper Studio's Settings, and the model must be enabled there. A mismatch shows up in chat as You don't have access to the model with the specified model ID.

Configure credentials once:

aws configure

The full walk-through, including how to create access keys, is in the AWS CLI configuration docs. Running aws configure installs nothing on the AWS side; it only stores your keys and default region locally.

No enabled model, no chat. Transcription and every local workspace tool run without AWS, but until a Claude model is enabled in your region and your credentials can reach it, the chat panel returns AccessDeniedException. If that happens after install, run /doctor in the chat input for an automated dependency and AWS-access check.

Optional: a Tavily key for web search

The assistant's web-search skill is powered by Tavily. It is entirely optional: without a key the web_search and web_fetch tools are unavailable, and everything else (chat, transcription, documents, the workspace IDE) works normally. The free tier covers personal use.

If you want it, sign up at app.tavily.com, copy the key, and either export it in your shell or paste it into the Settings UI. The environment variable takes priority when both are set, which makes rotation a one-line change:

echo 'export TAVILY_API_KEY=tvly-...' >> ~/.zshrc
source ~/.zshrc

See Configuration for how the environment, project, and user config layers stack.

Hardware, by model mode

The disk and memory you need depend on which model mode you install in. The mode decides where indexing and retrieval run and, crucially, which model weights get pulled onto your machine. It is separate from the chat-model choice in the toolbar. You pick a mode with a flag at install time (--cloud, --hybrid, --local) or change it any time under Settings, so this is not a permanent commitment.

ModeWhat runs on-deviceOn-device weights pulledFootprint
Cloud default Only transcription. Embeddings, reranking, entity extraction, and chat all run on Bedrock. None beyond the transcription models. Lean. The smallest install.
Hybrid Whatever you choose per capability. You pick a backend (cloud or on-device) for each of embeddings, reranking, and entity extraction. The full on-device stack is pulled so the on-device options are available to select. Same download as Local (~16 GB); you decide at runtime what actually runs.
Local Everything on-device: chat (Gemma), embeddings and reranking (Qwen3), entity extraction (GLiNER), and transcription. Fully offline. The whole stack: Gemma, Qwen3 embed and rerank, GLiNER. ~16 GB of weights, plus the RAM to load the ones you run.

The rule of thumb: Cloud is the cheap install (nothing but the transcription models lands on disk), while Hybrid and Local both pull roughly 16 GB of on-device weights. The difference between Hybrid and Local is not what gets downloaded; it is what you let run. Hybrid pulls the weights so you can mix per capability; Local commits to running everything locally.

Transcription is always on-device. In every mode, the Whisper and Parakeet speech models run locally and audio never leaves the laptop. They are pulled on install (or on first record) regardless of which mode you chose, so even a Cloud install downloads them. See Transcription & diarization.

A few practical notes on the on-device weights:

models/
Where every weight lands. Downloads are idempotent, so re-running setup skips anything already present.
Memory
The ~16 GB figure is disk. On-device chat with Gemma additionally needs enough free RAM to hold the model while it runs; the smaller index models (Qwen3, GLiNER) are lighter.
Apple Silicon
The on-device runtimes (MLX for transcription, llama.cpp with Metal for Gemma) are built for Apple Silicon. On-device chat is a macOS-arm64 feature.

Not sure which mode?

If you are just getting started, or you want current answers via Bedrock and a small install, start with Cloud (the default). Reach for Local only when you want everything to run offline and you have the disk and memory to spare, and for Hybrid when you want to keep chat on Bedrock but run some of the index stack on-device. The full trade-offs, including which backend handles what, are on the Model modes page.
