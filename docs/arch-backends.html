Model backends & modes

Whisper Studio can send a chat turn to three very different runtimes: on-device Gemma, OpenAI GPT-5.x over Bedrock, or the default Anthropic Claude path. The choice is made from one thing (the model key you picked in the toolbar) and it happens before any tokens stream.

Every chat turn enters the same endpoint in server/chat/routes.py, but not every turn talks to the same provider. Two of the three backends are self-contained side paths that return their own SSE stream and exit early. Click a box in the diagram to trace where a given model key ends up.

One key, three paths

The toolbar model key is latched on the first turn of a session, so switching models in Settings never disrupts an in-flight conversation. Once the system prompt and messages are assembled, the endpoint asks two yes/no questions in order, and the first "yes" wins:

  1. Is it a local model?

    server/chat/routes.py calls local_chat_response(...) (server/local/route.py). If is_local_model() matches the key (any key prefixed local_, such as Gemma), it returns a StreamingResponse and the turn runs entirely on-device.

  2. Is it an OpenAI model?

    Otherwise the endpoint calls openai_chat_response(...) (server/openai_bedrock/route.py). If is_openai_model() matches (its chat_model_meta carries provider == "openai_bedrock"), the turn is served in OpenAI request/response format over Bedrock.

  3. Otherwise: the Anthropic path

    If both bridges return None, the default Claude path continues in the endpoint: Bedrock streaming plus the full agentic tool loop, retry wrapper, reactive compaction, and per-turn cost tracking.

The early return is load-bearing. Both bridges must be reached before compaction, because compaction itself calls Bedrock (via Claude). A local or OpenAI turn must never touch that. Each side path is therefore self-contained: it builds its own prompt, runs its own tool loop, and yields its own SSE frames.

The Anthropic path (default)

This is the path most turns take, running Claude Haiku, Sonnet, or Opus. It is where the streaming loop from Chat & tool orchestration lives, so the backend-specific parts are the Bedrock client and the resilience wrappers around each round.

The shared Bedrock client

All Bedrock traffic goes through a process-wide singleton boto3 bedrock-runtime client, one per region, cached in server/chat/infra.py. A fresh client per call multiplies boto3's connection pool by the number of concurrent agents and can exhaust the macOS default file-descriptor limit (256), so the cache exists to keep sockets bounded (the pool is capped at 32 connections). Changing the region in Settings clears the cache so the next call rebuilds it.

The streaming call is invoke_model_with_response_stream. Chunks arrive as content_block and message events, which the loop parses into text, thinking, and tool_use blocks as they stream.

Retry and reactive compaction

Each streaming invocation is wrapped by invoke_stream_with_retry in server/infrastructure/bedrock_retry.py. The wrapper classifies every Bedrock error and reacts accordingly:

Error kindBehavior
Throttling (429)Retried with exponential backoff, up to 5 attempts (THROTTLE_MAX_RETRIES).
Other transient (for example, service unavailable)Retried up to 3 attempts (MAX_RETRIES).
Non-retryableRaised immediately, no retries.
Prompt too longRaised immediately as PromptTooLongError so the caller can compact.

Backoff is capped exponential: 500ms, 1s, 2s, 4s, up to a 10s ceiling. When the wrapper raises PromptTooLongError, the endpoint runs reactive compaction: it drops the oldest message pairs, calls compact_messages_with_claude(...), and retries the round. If the conversation is still too long after that, it surfaces an error asking you to start a new session. (Reactive compaction is the mid-stream safety net; the endpoint also compacts pre-emptively when history grows large. See Chat & tool orchestration.)

Cost tracking

Because the Anthropic path is the only one that bills a cloud provider per token, it is where cost tracking runs. Every round logs input, output, and prompt-cache token counts and an estimated USD cost (via estimate_cost), which feeds the session and daily spend caps. If a cap is exceeded, the turn stops with an error. See Cost tracking & budgets.

On-device Gemma

The local backend (server/local/) runs a quantized Gemma model with llama.cpp, entirely offline. Its design goals are isolation and a small prompt, since a laptop has far less headroom than Bedrock.

server/local/route.py
The bridge the endpoint calls. Detects a local key, and for a streaming turn returns stream_local_chat(...); for an approval resume it calls resume_local_chat(...) against the paused local turn.
server/local/stream.py
The streaming tool loop for Gemma. A simpler, non-streaming variant exists alongside it.
server/local/runtime.py
Model definitions (local_gemma, local_gemma_coder), the single-worker executor thread, and the model load/generate/unload lifecycle.

Some deliberate choices set the local path apart from the cloud one:

  • One executor thread. All llama.cpp work (load, generate, unload) runs on a single-worker thread, so the model is created and used on the same thread. This mirrors the on-device speech backends.
  • A lean system prompt. The cloud system prompt is tool-heavy and workspace-heavy (roughly 2 to 2.5K tokens the local model cannot use), so build_local_system_prompt(...) drops it and keeps only project and memory context. This speeds prefill and frees context space.
  • A configurable context window. Gemma supports up to 256K tokens, but the default (and floor) is 16K, because with tools on the tool-pool prompt alone is around 12K tokens. You can raise the window live from the chat-input context-window slider, which reloads the model at the new size. The WHISPER_LOCAL_N_CTX environment variable overrides the default at startup.
  • A shrinkable tool set. The Tools dropdown sets a scope of off, core, core_web, or all. Advertising fewer tools makes the prompt smaller and prefill faster.

The local path reuses the same executor and approval pipeline as the cloud loop, so an on-device turn is gated by the same permission modes and approval flow. It just never touches AWS.

Summarising a huge transcript reuses the resident model. Condensing an oversized transcript runs a fast per-chunk extraction (the map step), and because only one on-device model is resident at a time, that step runs on the model you are chatting with (its key is threaded into the condense call), falling back to whatever model is currently resident (loaded_key()) and then the first downloaded model - rather than loading a fixed model and evicting the one you are chatting with. So the map step does not thrash the model cache. See Transcript summarization.

OpenAI on Bedrock

The OpenAI backend (server/openai_bedrock/) runs GPT-5.x models. They are invoked through Bedrock but in OpenAI request/response format (the Responses API shape), which is why they are a separate provider rather than a variant of the Anthropic path.

server/openai_bedrock/route.py
The bridge the endpoint calls. Detects an OpenAI key and returns stream_openai_chat(...), or resume_openai_chat(...) for an approval resume.
server/openai_bedrock/runtime.py
Backend detection (is_openai_model), region resolution, and the effort and verbosity mappings below.

Two GPT-5.x-specific controls are mapped from the app's own settings:

  • Reasoning effort. reasoning_effort_for(...) maps the app's effort label to OpenAI's reasoning effort. On GPT-5.4/5.5 the ladder tops out at xhigh, so the app's extra, max, and ultracode all collapse to it. GPT-5.6 and up add a max tier above xhigh: when _supports_max_effort(...) is true the top-tier labels (max / ultracode) ride it, while only extra always collapses to xhigh.
  • Verbosity. verbosity_for(...) resolves the response verbosity (low / medium / high): a per-request override wins, then the model's configured default, otherwise medium.

GPT-5.x follows instructions literally, so a prompt like "think for N seconds" can turn into an actual wait. To avoid that, the OpenAI path drops the sleep tool from the pool and appends a short note to the (Claude-tuned) system prompt telling the model to interpret timing phrases as "reason briefly." The default region for this provider is us-east-2, overridable per model.

The "self-contained side path" framing above describes the top-level chat turn. The agent and subagent runtime is provider-agnostic too (WS-C): server/agents/providers/ holds base, anthropic, and openai adapters, and get_adapter(model_key, model_id) returns a ProviderTurn in canonical Anthropic shape, so a subagent can run on GPT-on-Bedrock as well as on Claude (with effort propagation and shared AGENT_CALL_CONCURRENCY). See The ultracode workflow runtime.

Where the model list comes from

The cloud model catalogue is config-driven, resolved in server/infrastructure/config.py and read through the thin accessors in server/chat/infra.py:

Config keyWhat it holds
chat_modelsMap of model key to the Bedrock model id.
chat_model_metaPer-model metadata: label, thinking mode, supports_tools, is_local, provider, and effort levels.
default_chat_modelThe key used when no model is selected.
bedrock_regionThe AWS region for the shared Bedrock client.

Local models are the exception: they are defined in code (server/local/runtime.py) and detected by their local_ key prefix, so a local turn needs no config lookup to branch. For every field the Settings UI writes, see Settings & config keys.

Data-retention consent

One piece of model metadata needs its own path: requires_data_retention. Mythos-class models such as Fable 5.0 only run when the AWS account's Bedrock data-retention mode is provider_data_share, so selecting one routes through a consent screen (src/components/chat/dataRetentionConsent.tsx) before the model is used. The setting is account-wide and managed just-in-time around model selection rather than left on permanently: acknowledging the screen enables it, and switching back to a non-retention model offers to return the account to zero retention (mode none).

Model backend is not model mode

This is the distinction that trips people up. The chat model (which backend runs your turn) is one setting, chosen in the toolbar. The model mode (cloud / hybrid / local) is a different setting entirely, and it controls the indexing and retrieval stack, not the chat model.

Chat model (toolbar)

Selects the backend for the conversation itself: Claude, GPT-5.x, or on-device Gemma. This is the decision chain on this page.

Model mode (cloud / hybrid / local)

Selects where the RAG stack runs: embeddings, the reranker, and entity extraction for the workspace index. It does not change your chat model.

So you can, for example, chat with cloud Claude while your index embeddings run locally, or run on-device Gemma while nothing about your indexing changes. The two settings are independent. For a hands-on tour of the mode dial, see the Model modes tutorial; for what each capability actually runs in each mode, see Index / GraphRAG & search.
