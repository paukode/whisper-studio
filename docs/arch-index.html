Index / GraphRAG & search

A local semantic index of your files: incremental embeddings, a lightweight entity graph, sqlite-vec KNN, an optional cross-encoder reranker, and a one-hop GraphRAG expansion that feed grounded, cited answers back into chat.

There are two flows and both live in server/index/pipeline.py. The build flow turns a folder into vectors, entity nodes, and edges. The query flow turns a question into ranked passages. Click a box in the diagram to trace how a file becomes storage, and how a question becomes grounding.

The build flow

build(ws_path) is serialized per workspace with a lock, so a user-triggered build and the daily scheduled refresh can never race the same SQLite index. It walks the tree, decides what changed, extracts and chunks each changed file, embeds it, extracts entities, optionally extracts typed relations, and writes everything atomically per file.

Walk and change detection

The walker in _iter_files skips anything not worth indexing: .git, node_modules, venv / .venv, __pycache__, dist, build, .next, models, and any dot-directory. For each remaining file the pipeline runs a two-stage change gate so a re-index is cheap:

  1. Size + mtime gate

    If the recorded size and modification time both match the manifest, the file is skipped without even opening it.

  2. SHA1 content hash

    If the gate misses but the SHA1 hash matches, the file's timestamps are touched (so the next run hits the cheap gate) and it is not re-embedded.

  3. Re-extract and re-embed

    Only a genuine content change reaches extraction. Files that vanished from disk are dropped in a deletion pass at the end.

A build can be cancelled cooperatively (an Event checked between files). On cancel the deletion pass and the completion stamp are skipped, so a partial index is never mistaken for a finished one and unvisited files are not wrongly dropped.

Extraction, by type

_extract_text routes each file to the right extractor, reusing the same code as attachment extraction:

File typeHow it becomes text
Text and codeDecoded directly (binary and oversized files are skipped).
Spreadsheets (CSV / XLSX)Small ones via MarkItDown; large ones via a bounded, streamed schema-plus-sample so a huge grid never loads into memory.
Rich docs (PDF / DOCX / PPTX / HTML / EPUB)MarkItDown, with OCR for PDFs. Inlined base64 images are stripped so they do not pollute the embedding.
ImagesOCR (Apple Vision, falling back to Haiku).
Audio and videoLocal transcription (see Transcription).

A file that yields no usable text still records an empty (zero-chunk) manifest entry, so the next daily refresh skips it via the cheap gate instead of re-OCR-ing it forever.

Chunking

chunk_text (server/index/chunker.py) splits text into line-anchored, structure-aware chunks. It prefers to start a chunk at a block boundary (a markdown heading or a code definition), budgets each chunk to about 400 tokens with a 64-token overlap, and keeps every chunk comfortably under the embedder's 512-token window so nothing gets truncated. Each chunk carries its 1-based start and end lines, which flow all the way through to the citation links in a grounded answer.

Optionally, a contextual header is prepended before embedding (the folder's chunk_context setting): the file path (free, offline) or an LLM-written situating line. Changing that mode forces a full re-embed, because the vectors themselves change.

Embedding

Chunks are embedded in batches by embed_documents (server/index/embedder.py). The backend depends on the active model mode:

Qwen3-Embedding-0.6B
The local default. 1024-dimensional, L2-normalized vectors on transformers + torch (MPS on Apple Silicon, CPU fallback). Pooling takes the last token's hidden state; queries get an instruction prefix, documents do not. Loaded lazily and unloaded after a build so an index run does not hold roughly 1 GB resident alongside the chat model.
Cohere Embed v4
The cloud path, on Amazon Bedrock (1536-dimensional). Uses input_type to distinguish queries from documents instead of an instruction prefix.

The MPS path can intermittently emit a non-finite (NaN) vector. A persisted NaN silently makes a document unfindable and breaks sqlite-vec, so the embedder re-computes any bad row (up to three retries) and zeroes it only as a last resort. There is also a repair_nonfinite_vectors repair pass in the store for indexes built before this guard existed.

Entities and relations

Each chunk's entities are extracted by extract_entities (server/index/extractor.py). Distinct entities become graph nodes; two nodes that co-occur in a chunk are linked. Crucially, edges are derived (two nodes are connected if they share a chunk) rather than stored, so an incremental delete of a file's chunks can never leave a stale edge weight behind.

Entities (NER)

GLiNER (zero-shot, local) returns the spans it believes match at a 0.5 confidence threshold; a workspace can opt into the heavier GLiNER2 model instead from the Index settings panel. In cloud mode, Claude Haiku extracts entities instead. The label set is no longer fixed: an entity-type profile (business vs code) is auto-picked per file from its extension, so a CV and a source file are asked for different kinds of entity. Names are stripped and deduped case-insensitively so one entity maps to one node.

Typed relations (optional)

When enabled per workspace, an LLM extracts typed edges (works_at, cites, depends_on) via Haiku or on-device Gemma. The LLM may only connect two names from the already-extracted entity list, so it cannot invent nodes, and each relation carries a 1 to 5 strength score.

After the walk, dedupe_entities collapses case, spacing, and punctuation variants of the same name into one canonical node (a lexical pass, plus an optional semantic-embedding pass that catches "Postgres" / "PostgreSQL"), so the graph shows one bubble per person or topic. A salience filter excludes generic category words ("function", "service", "API") and very high-fanout entities from co-occurrence edges, so boilerplate does not clutter the graph.

Storage layout

Everything lives under storage/index/<name>-<hash8>/, one readable directory per workspace, managed by server/index/store.py. The index is keyed per embedding backend, so switching between the Qwen3 and Cohere backends uses a separate DB and does not force a rebuild of the other. The schema holds files, chunks, entity nodes, node-to-chunk links, typed relations, and a small meta table.

vec BLOB (in chunks)
Vectors are stored as float32 BLOBs and are the source of truth. Because they are L2-normalized, cosine similarity is just a dot product.
vchunks (sqlite-vec)
An indexed KNN table, rebuilt from the BLOBs whenever a monotonic write-generation counter advances, so it self-heals after any change without threading sync through every write path. Available on the Homebrew-Python venv that setup.sh builds.
numpy fallback
When the interpreter cannot load SQLite extensions (the python.org framework build), search falls back to a cached brute-force numpy matrix. Same ranking, since both compare normalized vectors.
fts_chunks (FTS5)
A BM25 keyword index over chunk text and file path, rebuilt on the same write-generation signal, so a filename token (like buddy in buddy.json) is matchable even when the content never says it.

The query flow

query(ws_path, text, k) returns three lists kept deliberately separate so a multi-index caller can fuse them itself:

  • Dense matches. The question is embedded (with the instruction prefix) and the top-k cosine neighbours are fetched via store.search (sqlite-vec, or the numpy fallback).
  • Keyword hits. A BM25/FTS5 leg (behind the rag_hybrid_search flag) catches exact terms (filenames, ids, codes) that dense embeddings miss.
  • Related passages. One GraphRAG hop: store.expand returns chunks that share entity nodes with the dense matches, ranked by how many they share.

Retrieval grounding

retrieve_grounding is what chat calls when the user has selected indexes to search. It runs the query across every selected index, fuses the rankings, and returns a cited Markdown block. The fusion and filtering steps are careful about precision:

  1. Global dense ranking

    Cosine is comparable across indexes, so dense matches from every index are ranked together in one list rather than by per-index rank.

  2. Reciprocal-rank fusion

    The dense and keyword lists (and, for a follow-up, a conversation-contextualized query) are fused by reciprocal rank, so a keyword or filename hit with near-zero cosine is promoted in instead of being lost under the score floor.

  3. Optional rerank

    Behind the rag_reranker flag, a cross-encoder rescores the top candidates by judging each (question, passage) pair directly. On device this is Qwen3-Reranker (P("yes") from the final-token logits); in cloud mode it is Cohere Rerank on Bedrock. Best-effort: failure keeps the fused order.

  4. Dedup, cap, floor

    Byte-identical passages collapse, each document contributes at most 3 chunks, and matches below a 0.15 cosine floor are dropped as noise.

  5. Auto-merge

    Parent-document retrieval: once a small, focused file has a chunk in the top matches, its remaining chunks are spliced in (bounded by a budget, and never for a large document like a book), so a CV's full history grounds the answer rather than just its header.

The returned block instructs the model to treat the passages as the source of truth, answer directly from them, and end with a Sources section of path:lines links. Line-range citations come straight from the chunk boundaries recorded at build time.

How chat uses it

Grounding is injected during turn assembly: relevant passages are prepended to the user message, and a grounding SSE frame surfaces a chip ("grounded in N folders / M passages") in the UI. When a turn is grounded and the strict_rag flag is on (the default), the workspace file and search tools are suppressed for that turn, so the model answers from the injected passages instead of re-crawling files it has already indexed.

Grounding runs on a dedicated single-thread executor with a timeout (extended on a turn's first, cold reranker load). If it times out or fails, the turn answers without grounding rather than blocking, and no chip is shown.
