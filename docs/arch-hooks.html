Blocking hooks

Hooks let you gate what the assistant does. A hook can deny a tool call, rewrite its input, add context to a result, or refuse to end a turn — deterministically, in your own code, before the model ever sees the outcome. They run across every path: chat, sub-agents, and cron.

The engine lives in server/hooks/: schema.py (the hook definition, outcome, and matcher semantics), engine.py (evaluation), config_loader.py (loading + project trust), and routes.py (the /api/hooks API). Click a box in the diagram to trace one tool call through the gates.

The three events

A hook is registered for one event. Three are gates the model cannot route around:

EventWhenWhat a hook can do
PreToolUseJust before a tool executesDeny the call (the model gets the reason instead of a result) or rewrite its input via updatedInput.
PostToolUseJust after a tool returnsFold additionalContext into the result the model sees — e.g. a lint summary after a write.
StopWhen the turn is about to endRefuse to end: return a reason and the turn continues. Bounded by MAX_STOP_BLOCKS_PER_TURN so it can't loop forever.

A hook is a HookDef (server/hooks/schema.py): an event, a shell command, a matcher (default *; exact name, a pipe|list, or a /regex/ against the tool name), a timeout, an enabled flag, and an on_error policy (ignore or block). Its source is user, project, or plugin.

Two phases, one contract

run_hooks(event, payload) evaluates an event in two phases. Phase 1 runs in-process hooks — Python plugins and the orchestrator's own gate (this is where the goal completion gate registers its Stop handler); a control dict here can short-circuit the whole event. Phase 2 runs shell hooks: user hooks first, then trusted project hooks, each filtered by its matcher against the tool name.

A shell hook receives the event payload as JSON on stdin (run_sandboxed gained input_data for exactly this, plus env_extra for the WHISPER_* variables) and communicates back through both its exit code and its stdout:

  1. Exit code

    0 passes. 2 blocks — stderr becomes the reason the model sees. Any other code, or a timeout, is handled by the hook's on_error (ignore lets the call through; block stops it).

  2. Structured stdout

    A JSON object {decision, reason, updatedInput, additionalContext} is parsed if present: decision:"deny" blocks with reason; updatedInput rewrites a PreToolUse tool input; additionalContext is appended after a PostToolUse.

env_extra is passed via the subprocess environment (env=), not prepended to the command. A command prefix would turn any hook that begins with a shell construct (if, for, case) into a syntax error whose exit code 2 reads as DENY — silently blocking every tool call. That was the single biggest bug the hooks review caught.

Wired across every path

The gates are not chat-only. PreToolUse and PostToolUse fold into the tool executor that chat, sub-agents, and cron all share, so a deny or a rewrite applies uniformly. The Stop gate (check_stop_hooks()) runs at the end of every turn in all three streaming loops. Sub-agents and cron runs honor PreToolUse too — cron bridges the check onto the server loop.

server/hooks/engine.py
run_hooks(event, payload), the two-phase evaluator, check_stop_hooks(), and dry_run() for the Test button.
server/hooks/schema.py
HookDef / HookOutcome, matcher semantics, and config normalization (v2 keyed-by-id; the v1 bare list still loads).
server/hooks/config_loader.py
User hooks.json plus project hooks in <workspace>/.whisper/settings.json — the latter inert until trusted.

Project trust

Hooks are code that runs on your machine, so a hook that arrives inside a cloned repository is a supply-chain vector. Project hooks are therefore inert until you approve them: the panel shows each project hook's command before you trust it, and a SHA-256 of the trusted content is recorded so an edit re-arms the prompt (the same trust model as skills and saved workflows). User hooks — the ones you wrote in your own config — are trusted by definition.

The API and the panel

The Settings Hooks panel is backed by the /api/hooks routes; the assistant does not create hooks itself (there is no hook tool and no /hooks command — hooks are a user-owned guardrail).

RouteMethodPurpose
/api/hooksGET / POSTList hooks (grouped by event); create one.
/api/hooks/{hook_id}PUT / DELETEEdit or remove a hook.
/api/hooks/testPOSTDry-run a hook against a sample payload (the Test button) without wiring it live.
/api/hooks/project/approve · /revokePOSTTrust or untrust the current workspace's project hooks.
/api/hooks/exportGETExport the effective hook config.

Live blocks surface in the chat: a denied call renders a hook_blocked card and a refused turn renders stop_hook_feedback. The older server/infrastructure/hooks.py is now a deprecated shim that delegates to this engine.
