Cost tracking & budgets

Every chat turn logs exactly what it spent. Those rows roll up into session and daily totals, surface prompt-cache telemetry, and gate the next round against your spend caps - all locally, in SQLite.

Cost tracking lives in server/costs/, a small self-contained package: tracker.py (pricing, logging, aggregation, cache telemetry, the API) and budget.py (cap enforcement). The chat pipeline calls into both. Remaining-capacity tracking is no longer a costs concern: it moved to the token-based context meter in server/chat/loop_hints.py (see below). Click a box below to trace the flow from a completed turn to the Costs panel.

Pricing: a per-model rate table

Cost is not measured, it is estimated from token counts and a per-model price table. estimate_cost(model_key, input_tokens, output_tokens, cache_read_tokens, cache_creation_tokens, cached_in_input) returns a USD figure. The rates live in _MODEL_PRICING in server/costs/tracker.py as USD per 1M tokens, with four explicit rates per model: input, output, cache-read, and cache-write (prompt-cache creation).

There is deliberately no fallback bucket. An unknown model key bills $0 and logs an error rather than silently inheriting another model's rate, so a missing entry is loud instead of quietly wrong. Add new models to the table explicitly.

Two cache conventions, one function

Anthropic and OpenAI count cached tokens differently, and estimate_cost handles both through its cached_in_input flag:

  • Anthropic (cached_in_input=False, the default): input, cache-read, and cache-creation are three disjoint token buckets, each billed once at its own rate.
  • OpenAI (cached_in_input=True): the cache-read tokens are a subset of the reported input tokens, so the function subtracts them out and bills the cached portion once at the cache-read rate and the remainder at the input rate, never both.

See prompt caching for how cache reads and writes are produced in the first place.

Logging a turn

When a streaming round ends, its message_delta carries the round's token usage. The chat loop calls record_turn(session_id, turn_number, model, input_tokens, output_tokens, cost_usd, ...) to persist one row into the session_costs table in storage/sessions.db. One turn, one row.

Cost logging never breaks a chat. record_turn wraps its write in a try/except and only logs on failure, so a cost-logging error can never crash the turn that produced it. The same is true of the read-side aggregates, which return an empty result rather than raising.

Each row records the model, the four token counts, the estimated cost, the API duration, and a UTC timestamp:

ColumnMeaning
session_id, turn_numberWhich session and round this row belongs to.
modelThe model key that ran (drives which pricing row applied).
input_tokens, output_tokensNon-cached prompt tokens in, generated tokens out.
cache_read_tokens, cache_creation_tokensPrompt-cache reads and writes for the round.
cost_usdThe estimate_cost result for the round.
api_duration_msHow long the Bedrock call took.
created_atUTC timestamp, defaulted by SQLite; drives the daily rollups.

The table is created by server/migrations/001_add_session_costs.py with indexes on session_id and created_at. When a session is deleted, its cost rows are deleted with it. See Persistence for how the SQLite store is managed overall.

Aggregations

The logged rows are read back through a handful of query helpers in server/costs/tracker.py:

get_session_summary(session_id)
Totals for one session: turn count, total input/output tokens, total cache reads, total cost, and total API duration.
get_session_costs(session_id)
The per-turn breakdown for a session, ordered by turn number.
get_model_breakdown(session_id="")
Per-model turns, tokens, and cost, ordered by spend. Pass a session id to scope it, or leave it empty for a global breakdown across all sessions.
get_daily_costs(days=30)
Daily totals for the last N days, grouped by DATE(created_at) - the data behind the bar chart.
get_today_total_cost()
The single number the daily budget check compares against: today's total cost in UTC.

Cache telemetry

get_cache_stats(session_id="") aggregates prompt-cache usage - globally or for one session - into a small block: read_tokens, write_tokens, a hit_rate (cache reads over total prompt tokens), and est_savings_usd (what the cached reads would have cost at the full input rate minus their cache-read rate, summed per model). It is the live "is caching actually working" surface, and it rides on GET /api/costs/summary as a cache object.

Tracking remaining context

Remaining-capacity tracking answers a different question than the cost aggregates. It is not "how much have you spent" but "how much room is left in this conversation before it has to compact". This moved out of the costs package into the token-based context meter in server/chat/loop_hints.py: every round's message_start usage already reports the true prompt size, so note_prompt_tokens() records it directly against the model's window - no character-count estimate:

  • The true prompt size is input + cache_read + cache_creation tokens from message_start. note_prompt_tokens() stores the latest value per session and context_estimate() reads back (context_used, context_max).
  • The window defaults to DEFAULT_CONTEXT_MAX (200,000 tokens) and can be overridden per model via context_window_for() and the context_windows config map.
  • When the real prompt crosses COMPACT_NUDGE_FRACTION (0.8) of the window, should_nudge_compaction() fires once so proactive compaction can trigger on token truth rather than a character guess.

Caching no longer fools the meter. The old character-based forecast had to add the cached read and creation tokens back by hand to see the true context size. Now every round's message_start usage already reports the true prompt size and note_prompt_tokens() records it directly, so there is nothing to correct for. See Chat & tool orchestration and compaction.

Budgets: stopping before you overspend

Budgets are a hard gate, checked before each round in the chat loop. check_budget(session_id) in server/costs/budget.py reads two caps from config.json and returns either None (you are within budget) or a BudgetExceeded object describing which cap was hit:

max_session_cost_usd
Ceiling on one session's total cost. Compared against get_session_summary. 0 means unlimited.
max_daily_cost_usd
Ceiling on today's total cost across all sessions. Compared against get_today_total_cost. 0 means unlimited.

The session cap is checked first, then the daily cap. Because either limit set to 0 means "no limit", cost enforcement is off by default and you opt into it in Settings.

What happens when a cap is hit

When check_budget returns a BudgetExceeded, the chat loop does not start another Bedrock call. Instead it:

  1. Emits a budget_warning event

    An SSE frame carries the human-readable message plus the structured budget_kind (session or daily), budget_limit, and budget_current so the SPA can render a precise notice.

  2. Writes the warning into the transcript

    A short [Budget exceeded] text line is streamed so the halt is visible in the conversation, not just a silent stop.

  3. Ends the turn cleanly

    It sends [DONE] and returns. The turn stops rather than erroring, so nothing is left half-streamed.

The message tells you exactly how to proceed - start a new session, raise the relevant cap, or wait until tomorrow for a daily reset. You can also clear today's records manually with POST /api/costs/reset-daily.

Settings and the Costs panel

The two caps are set in the Settings UI (and persisted to config.json); see Configuration and Settings & config keys. Live spend is surfaced in two places in the SPA:

Live in the chat

Each turn's usage SSE event carries the running token counts, the per-turn estimated_cost_usd, and context_used / context_max, so the current turn's cost and remaining context update as tokens stream.

The Costs panel

A dedicated panel reads the aggregate endpoints to show per-model totals, a daily bar chart, today's total, and the configured caps - the whole spend picture at a glance.
