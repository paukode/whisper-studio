Attachment extraction

When you drop a file into chat, one type-aware router turns it into Markdown the model can read. PDFs, DOCX, spreadsheets, images, and audio/video each get their own handler, with an OCR fallback for scans, careful sizing for vision, and on-device transcription for media.

Everything an attachment goes through lives under server/extract/, fronted by the upload endpoint in server/attachments.py. The goal is simple: give the model clean Markdown for text, readable text for scanned pages, a compact sample for huge tables, and image blocks sized so fine print survives. The router

The heart of the pipeline is convert_document(content, ext, filename) in server/extract/__init__.py. It runs MarkItDown first to convert almost any format to Markdown, then dispatches by file extension to a specializer:

ExtensionHandlerWhat it does
.pdfextract_pdf (server/extract/pdf.py)Keep the text layer, or OCR the pages if the PDF is scanned.
.xlsx / .xlsextract_xlsx (server/extract/sheet.py)Keep the full table when small; sample it when large.
.csvextract_csv (server/extract/sheet.py)Same size-aware sampling as spreadsheets.
everything elseMarkItDown outputDOCX, PPTX, HTML, EPUB, JSON, XML, code, plain text.

This function is blocking (it touches disk and CPU-heavy libraries), so the upload endpoint calls it through asyncio.to_thread and never stalls the event loop. Images and audio/video take their own handlers, covered in Images and vision sizing and Audio and video below.

MarkItDown first

Whatever the extension, convert_document writes the bytes to a temp file and runs MarkItDown().convert() once. That single call handles the long tail of office and web formats, so the specializers never re-parse from scratch. For PDFs and spreadsheets, the MarkItDown text is passed straight into the specialized handler, which decides whether it is good enough to keep or whether it needs to fall back to OCR or sampling.

MarkItDown often raises or returns nothing on scanned PDFs and odd files. That is expected. The failure is logged and swallowed; PDFs still get an OCR pass, and any other file surfaces [No content extracted] rather than crashing the upload.

Scanned PDFs

MarkItDown (via pdfplumber and pdfminer.six) only reads an embedded text layer. A born-digital PDF yields plenty of text; a scanned, image-only PDF comes back nearly empty. So extract_pdf counts the non-whitespace characters MarkItDown produced:

  • If the text has 100 or more non-whitespace characters (_SCANNED_TEXT_THRESHOLD), the PDF is treated as born-digital and the MarkItDown text is returned as-is.
  • Below that threshold, the PDF is treated as scanned. Its pages are rasterized with pypdfium2 and sent to OCR.

Rasterization is deliberately bounded so uploads stay responsive:

_RENDER_SCALE = 2.0
About 144 DPI. A good balance of OCR accuracy against render time.
_MAX_OCR_PAGES = 30
Only the first 30 pages are rendered and OCR'd. If the document is longer, the output notes that OCR was limited to the first 30 pages.

pypdfium2 was chosen partly because it is license-clean (BSD/Apache). If it is unavailable for any reason, the handler returns the (empty) MarkItDown text rather than failing.

OCR engine priority

OCR lives in server/extract/ocr.py and tries engines in a strict order, designed so the normal path is fully on-device and never spends Bedrock tokens without reason:

  1. Apple Vision, always first

    Native macOS OCR via the ocrmac package (Vision framework). Zero model download, near-zero RAM, fast on printed text, fully on-device and private, no Bedrock cost. Results come back as (text, confidence, bbox) tuples, which the handler sorts top-to-bottom then left-to-right (Vision's bounding boxes use a bottom-left origin) so reading order is preserved.

  2. Bedrock Claude Haiku, only as a fallback

    If Apple Vision fails or is unavailable and AWS credentials resolve locally, the pages are sent to Haiku's vision model with a prompt to transcribe them into clean Markdown. Up to 20 images go out in a single invoke, so a scanned PDF is usually one request.

Credentials are checked without a network call (boto3.Session().get_credentials()), so a machine with no AWS setup simply never reaches the Haiku branch. If Haiku is tried and returns an access-denied or missing-credentials error, that fallback is disabled for the rest of the process so uploads do not keep hammering a denied endpoint.

OCR is best-effort by design. If neither engine yields text, the handler returns an empty string (or [No text recognized]) instead of raising, so a bad scan never fails the whole upload.

Large spreadsheets

A wide sheet with thousands of rows can blow past the context window on its own. So server/extract/sheet.py keeps MarkItDown's full Markdown table only while it is small. Once that output exceeds 60,000 characters (_LARGE_CHARS), the handler swaps in a compact schema and sample instead:

  • Each sheet's title and dimensions, as N rows x M cols.
  • The header row.
  • The first 20 rows (_SAMPLE_ROWS) of data, rendered as a Markdown table.

The injected text opens with a note telling the model it is seeing a sample and can ask for specific rows or columns to see more, which it does through the analyze_document tool (see Agent tools). CSV files follow the same rule via extract_csv, reporting the total row and column counts alongside the sample. If openpyxl or the CSV parser fails, the handler keeps the original MarkItDown output rather than losing the data.

Images and vision sizing

Images take their own branch in the upload endpoint. Two things happen to every uploaded image:

  • OCR to text. ocr_image_bytes (server/extract/image.py) runs the same OCR path as scanned PDFs, so text-only models (the local Gemma build) get readable text pulled out of the image. Vision-capable models still receive the raw image block alongside it.
  • Sizing for vision. _prepare_image in server/attachments.py caps the image's long edge and re-encodes it once, at high quality, so fine text stays legible.

The sizing is tuned around how Claude handles images. Claude internally downsamples anything whose long edge exceeds about 1568px, so Whisper Studio does that once, at high quality, rather than shipping an oversized image the model re-compresses (which softens small text and hurts OCR):

IMAGE_MAX_DIM = 1568
Long-edge cap, Anthropic's recommended maximum. The image is only ever shrunk, never enlarged.
IMAGE_JPEG_QUALITY = 90
High quality so small text stays readable when a JPEG re-encode is needed.
IMAGE_HARD_BYTES
Keeps each image under Bedrock's roughly 5 MB per-image limit; JPEG quality steps down if necessary.

Small, simple images (a PNG screenshot within the size caps) pass through untouched so they stay lossless for the crispest possible OCR. PNG sources stay PNG when that fits under the byte cap, and only fall back to JPEG when the lossless version is too large.

Audio and video

Audio and video are never sent to the model as media. The upload endpoint routes them by extension (is_media_ext - MP3, WAV, M4A, FLAC, OGG on the audio side; MP4, MOV, WebM, MKV, AVI, M4V on the video side) to extract_media in server/extract/media.py, which turns them into a Markdown transcript entirely on-device - no Bedrock call. Only that transcript (plus, for video, a bounded set of frames) is kept; the original media bytes are discarded, which is why audio and video get the higher size cap.

  • Transcription. ffmpeg decodes the audio track to 16 kHz mono and mlx-whisper (large-v3-turbo) transcribes it into timestamped lines. Video reuses the same loader on its audio track, so one code path covers both.
  • Speaker labels. Each segment is embedded with ECAPA-VoxCeleb and clustered by the same agglomerative clusterer that powers live diarization (server/diarization/), so the transcript reads [00:12] Speaker 2: …. Segments too short to embed inherit the previous speaker, and the whole step degrades to a single speaker if clustering is unavailable.
  • Video frames (hybrid understanding). ffmpeg samples keyframes on a single pass - about one every 10 s, stretched on longer clips so no more than 20 frames (_MAX_FRAMES) are ever kept. Each frame's on-screen text is OCR'd with Apple Vision directly (not the escalating engine, so a text-free video never fans out into a Haiku call per frame) and merged into the transcript as timestamped notes. The frames themselves are retained as downscaled JPEGs (1288px long edge, quality 82) so vision-capable models can see what was on screen, while text-only models still get the transcript plus the OCR'd notes.

This is the uploaded-file path. Recording live in a meeting is a different pipeline - audio streamed over a WebSocket with interim drafts and self-healing speaker clusters - covered in Transcription & diarization and the Voice & meetings tutorial.

Heading outline for partial injection

Every converted document also gets a heading outline built by server/extract/outline.py. build_outline(text) scans the Markdown for # through ###### headings and returns both a nested outline and a list of sections, each a character-offset slice keyed by a 1-based number.

This is what lets the chat layer inject a large document by outline plus a lead section rather than a blunt character truncation. When the model wants more, get_section(text, sections, num) returns exactly that section by number. Documents with no Markdown headings (plain text, a flat spreadsheet dump) simply come back with an empty outline. See Chat & tool orchestration for how the injection fits into a turn.

Size gates and storage

The upload endpoint (POST /api/upload) gates by byte size before any conversion runs. Documents and images share one cap; audio and video get a larger one because only their transcript is kept:

MAX_FILE_SIZE = 50 MB
Documents and images. Over this, the upload is rejected with a 400.
MEDIA_MAX_FILE_SIZE = 200 MB
Audio and video. They are transcribed to text (only the transcript is retained), so the byte cap can be higher.

Converted attachments are held in memory keyed by a UUID, with a 3 hour TTL (ATTACHMENT_TTL) so a message's files survive a later regenerate or edit-resend in the same running session. A background loop prunes expired entries. Anything not matched by an image, media, or known-document extension is treated as plain text if its bytes decode as UTF-8, and rejected only when it is genuinely binary.
