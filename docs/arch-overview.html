System overview

Whisper Studio runs as a single process on your machine: one FastAPI server that serves both the API and the React SPA from the same origin. Except for Amazon Bedrock and the optional Tavily and MCP integrations, nothing in the request path is remote.

This page is the map. Everything below the diagram - and every other page in the Architecture section - is a zoom into one part of it. Click any box to trace where its data flows; the downstream path lights up and the rest fades.

The layers, left to right

Read the diagram as a pipeline. A request starts in the browser, crosses one of three transports, and lands in the FastAPI process, which fans out to local runtimes, storage, and - only for chat - the cloud.

Browser (React 19 SPA)

A single-page app renders the chat, the live transcript, and the full workspace IDE. State lives in Zustand stores organized as per-session runtimes (up to three sessions stream at once). All Markdown is sanitized through marked + DOMPurify before it reaches the DOM. See Frontend architecture.

Transport (same origin)

Three channels share http://127.0.0.1:8000: REST for request/response, Server-Sent Events for streamed tokens and session events, and WebSocket for binary audio, the terminal PTY, and LSP. Because the SPA is served by the same process, there is no CORS layer.

FastAPI server (the local process)

The heart of the system. The chat pipeline builds each prompt and drives a streaming tool loop; the tool router and executors carry out tool calls; the agent runtime spawns sub-agents and teams; and the approval + validator layer gates anything that writes.

The A-J program added five more server subsystems the map now covers. A unified background-task framework (server/tasks/, the agent_tasks table, kinds shell/agent/workflow/ci, /api/background-tasks) registers every detached job, including scheduled and sub-agent runs. Blocking hooks (server/hooks/) gate the loop at PreToolUse, PostToolUse, and Stop (see Blocking hooks). A goal loop + completion gate (server/goals/, run_completion_gate) keeps a turn running until a goal verdict clears (see The goal loop). The ultracode workflow runtime (server/workflows/) runs model-authored deterministic JS in a Node-24 vm harness over JSON-RPC into Python; see The ultracode workflow runtime. And CI watch + PR autofix (server/ci/) watches a build and drives it back to green.

Local runtime & storage

Transcription (Whisper / Parakeet), the workspace index, and the sandbox all run on-device. Everything durable lives in a single WAL-mode SQLite database plus your workspace files and the memory store.

External services

Amazon Bedrock is the only cloud LLM. It is called when you submit a message, and now also from detached background work: the goal evaluator (server/goals/evaluator.py), CI diagnose (server/ci/diagnose.py), and workflow agents make their own (often Haiku) Bedrock calls without a fresh submit. Optional MCP servers and Tavily are reached only when a tool invokes them, and CI watch/autofix reaches GitHub through a read-only gh CLI wrapper (server/ci/provider.py) when ci_watch/ci_status/ci_autofix run - a fourth external service, but not an LLM call.

Key invariants

One origin, one process

The SPA is served from the same FastAPI app that exposes the API. Nothing to cross, so no CORS.

Localhost by default

The server binds 127.0.0.1. LAN exposure requires an explicit HOST=0.0.0.0 opt-in.

Transcription is local

Audio never leaves the machine. Only a transcript can become a prompt - and only when you type something.

Bedrock is the only cloud LLM

No telemetry, no proxy, no gateway. The signed request goes straight from the local process to Bedrock.
