The ultracode workflow runtime

Ultracode is the mode where the assistant stops improvising a chain of tool calls and instead writes a program - a deterministic JavaScript orchestration script - that a locked-down Node vm harness runs detached from the turn. The script fans out many subagents with real concurrency, budgets, and a resumable journal, and it runs the same way on Claude and on GPT-on-Bedrock.

The runtime lives in server/workflows/. The model authors a script; a single Node process parses and validates it; the user approves it; then the run executes detached while the Python side enforces the limits and streams progress back as a live card. Why a script instead of a tool loop

An ordinary chat turn is a loop: the model emits one tool call, waits for the result, decides the next call, and so on. That is fine for a handful of steps but it serializes everything and the model re-reasons the whole plan on every hop. Ultracode inverts it. For work that is wide (audit twenty files), deep (generate five designs, judge them, synthesize), or that must be confident (find bugs, then adversarially verify each), the model writes the control flow once as a script. Loops, conditionals, fan-out, and fan-in are ordinary JavaScript; the expensive leaf calls - agent() - run concurrently under a real scheduler. The plan is deterministic and inspectable before a single agent runs.

Ultracode is a tier, not a separate model. It is offered at the top of the effort dial and turns on the workflow tools plus the script-authoring directive (server/prompts/ultracode.py). Simple questions are still answered directly - the directive tells the model to reach for a workflow only when the work is genuinely parallel, comprehensive, or adversarially verified.

The script contract

A workflow script is plain JavaScript (not TypeScript) that must begin with a pure export const meta literal - no variables, calls, or interpolation - so it can be extracted and shown for approval without executing the body. After meta, the body runs in an async context and may await directly.

export const meta = {
  name: 'review-changes',
  description: 'Review changed files across dimensions, verify each finding',
  phases: [{ title: 'Review' }, { title: 'Verify' }],
}

const DIMENSIONS = [{ key: 'bugs', prompt: '…' }, { key: 'perf', prompt: '…' }]

// pipeline: each item flows through all stages independently - no barrier,
// so a 'bugs' finding verifies while 'perf' is still being reviewed.
const results = await pipeline(
  DIMENSIONS,
  d => agent(d.prompt, { label: `review:${d.key}`, phase: 'Review', schema: FINDINGS }),
  review => parallel(review.findings.map(f => () =>
    agent(`Adversarially verify: ${f.title}`, { phase: 'Verify', schema: VERDICT })
      .then(v => ({ ...f, verdict: v }))
  ))
)
return { confirmed: results.flat().filter(f => f.verdict?.isReal) }

The body has a small, fixed set of hooks. Everything else is ordinary JavaScript built-ins.

HookWhat it does
agent(prompt, opts?)Spawn one subagent. Without schema it returns the final text; with a JSON-Schema schema it returns a validated object. opts: label, phase, schema, model, effort, isolation:'worktree', agentType.
pipeline(items, …stages)Run each item through all stages independently, no barrier between stages. Item A can be in stage 3 while B is still in stage 1. The default for multi-stage work.
parallel(thunks)Run tasks concurrently and await them all - a barrier. A thrown thunk resolves to null (filter before use). Use only when you truly need every result together.
phase(title)Start a new phase; subsequent agent() calls group under it in the progress display.
log(msg)Emit a progress line shown above the run's progress tree.
workflow(name, args?)Run another saved workflow inline as a sub-step (one level of nesting; shares the parent's caps and budget).
budget{ total, spent(), remaining() } - the run's USD target, for dynamic loops (while (budget.remaining() > 50_000) …).
argsThe JSON value passed into the run, verbatim - the way a saved workflow is parameterized.

Determinism and the sandbox

The script runs inside a Node 24 vm context created by server/workflows/harness/harness.mjs. The context is deliberately bare: there is no require, process, fs, fetch, or module system - only the hooks above and the standard JS built-ins. On top of that, harness/guards.mjs installs determinism guards so a resume always replays identically:

Math.random()
Throws. Randomness would make a resume diverge - derive variety from agent() results or an index instead.
Date.now() / new Date()
Throw (a Date with explicit arguments still works). The Date.prototype.constructor back-door is redirected to the guarded constructor too.
setTimeout / setInterval / setImmediate
Throw; queueMicrotask is kept (deterministic).

The single hardest boundary is host code execution. Node is launched with --disallow-code-generation-from-strings, so the classic vm escape - reaching the host Function constructor via anAgent.constructor.constructor('return process')() - raises EvalError instead of returning the host process. The harness's own vm.runInContext is unaffected.

Honest caveat: a vm context is not a certified security boundary. The real anchor is defence in depth - the user approves every new script before it runs, the context has no dangerous globals, the child's environment is scrubbed, and codegen-from-strings is disabled. Treat a workflow script the way you treat any code you are about to run: read the preview.

The runtime and its limits

server/workflows/runtime.py is the Python side of the bridge - a WorkflowRun that acts as the JSON-RPC server for the harness and owns the limits the script cannot bypass, no matter what the JavaScript does:

  1. 16-way concurrency

    An asyncio.Semaphore caps live agent() calls. A script may hand parallel() a thousand thunks; only sixteen run at once and the rest queue.

  2. A 1000-agent lifetime cap

    A runaway-loop backstop. The cap is checked before dispatch; exceeding it raises an AgentCap error into the script and marks cap_reached.

  3. A USD budget

    Optional per-run ceiling. Cost accrues when an agent completes, so the budget is re-checked after acquiring a slot as well as before dispatch - that bounds a parallel burst's overshoot to the concurrency window instead of the whole cap.

  4. A resumable journal

    Every phase and agent call is appended to the run's journal; the resume cache is built from it (see below).

Cost is priced per agent against the model that actually ran (a per-agent opts.model override is billed at that model's rate), and rolls up into the run's ledger and the session cost log. A nested workflow() inherits the parent's remaining budget and merges its spend, agents, and cap back into the parent, so nested work can never exceed the outer cap.

The Node harness to Python bridge

The script and the runtime speak line-delimited JSON-RPC over the child's stdio, framed by server/workflows/rpc.py (a 10 MB line ceiling, typed error constants for budget / cap / cancel / depth). Only the calls that need the model cross the boundary: an agent() or nested workflow() becomes a request to Python; every other statement is pure in-sandbox JS. This is what lets the same script run unchanged on either provider - the harness never talks to a model directly.

On the Python side, server/workflows/agent_adapter.py turns one agent() RPC into one run_agent call. It maps the harness opts (schema to structured_schema, effort, model, isolation, agentType) and normalizes the result back to { text, output, usage, status, agent_id }. Because it rides the provider-agnostic runtime, a workflow launched from a GPT-on-Bedrock session spawns GPT subagents, and one launched from a Claude session spawns Claude subagents - structured output is validated on both.

Workflow subagents can reach every session-connected tool - the same tool router and executors the chat pipeline uses - so a workflow agent can read files, run the verify-change skill, search the index, or drive the browser, exactly like a hand-spawned sub-agent.

Journal, resume, and lifecycle

Runs are registered in the workflow_runs table (see Persistence) with an immutable on-disk script snapshot and an append-only journal under data/workflows/runs/<run_id>/. The run is driven detached via the shared async-task machinery; its outcome is published on a per-run event-bus channel (workflow:{run_id}) so a reloaded session re-attaches. On boot, reconcile_stale() flips any orphaned running row to stale - its journal stays resumable.

Resume is the payoff of determinism. Every agent() call is keyed by a call_hash of its prompt and opts (display-only fields like label and phase are dropped). When a run is resumed, the longest unchanged prefix of calls returns cached results instantly - ordered by issue seq, not completion order, so two identical parallel calls replay into the right positions - and only the first edited call and everything after it runs live. Same script and same args means a 100% cache hit; editing one stage re-runs that stage onward.

Approval and trust

A brand-new inline script, or a saved workflow that has not been approved, returns a workflow_preview side-effect rather than running. WorkflowPreviewCard shows the phases as a stepper and the whole script; approving POSTs to /api/workflows/runs and the run begins. A saved workflow only launches immediately once you have approved it (its trust is revoked automatically if the script is later edited), and a resume replays a run you already approved. This preview-then-approve gate is the same mechanism the CI autofix below reuses.

CI watch and PR autofix: ultracode applied

The clearest use of the runtime is the CI loop in server/ci/. After you push, ci_watch follows a branch's GitHub Actions run to its conclusion as a detached background task. When it fails, ci_autofix reads the failing logs, diagnoses each failure into structured findings, and generates a workflow script that applies the minimal fixes and then runs the verify_change skill. That generated script is handed to you through the very same workflow_preview card - so nothing is edited or pushed until you approve it, and once approved it runs under the runtime's budget, journal, and run card. Untrusted CI-log text is embedded as double-encoded JSON and clearly delimited as data in the fix agent's prompt, so a hostile log line cannot inject instructions.

This is the pattern to copy: a subsystem does its own scouting (find the failing checks, read the logs), then emits a workflow for the runtime to execute under human approval. Diagnosis is read-only; the write path is gated by the preview.

How it composes with the rest of the system

Ultracode is the capstone of a larger harness; it leans on the pieces around it rather than duplicating them.

Provider-agnostic agents
Every agent() is a run_agent call, so workflows inherit Claude/GPT parity, usage accounting, effort, structured output, and worktree isolation.
The task registry
A workflow run and a CI watch are rows in the unified background-task registry, so they appear in the tasks panel and the header running-count pill.
Tool disclosure and hooks
The workflow and CI tools surface only in the ultracode tier; workflow subagents still pass through the same Pre/PostToolUse hooks and tool executors.
The verify-change skill
Generated fix workflows end by running the repo's verification gate, which emits a deterministic VERIFY PASS / VERIFY FAIL line - the same skill the goal completion gate keys on.
