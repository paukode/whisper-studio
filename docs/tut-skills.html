Skills

Skills are small, focused capabilities Claude can reach for mid-conversation. The built-in ones load automatically when they fit your request, and you can add your own by dropping a Markdown file into the skills/ directory.

You never have to name a skill. You just ask for what you want, and Claude picks the skill whose description matches. This page shows what ships in the box, how selection works, and how to write and invoke a skill of your own.

What a skill actually is

A skill is a Markdown file with a small YAML header (the frontmatter) and a body. The frontmatter names the skill and describes when to use it; the body is the instructions the model follows once the skill runs. On startup, Whisper Studio reads every file in skills/, turns each one into a tool, and hands the whole set to Claude alongside its other tools. Because a skill is a tool, selection is just normal tool use: Claude reads the descriptions and calls the one that fits.

Skills come in two shapes:

  • Prompt skills. The body is guidance the model reads and applies directly (for example, a house style for meeting notes). No code runs on your machine.
  • Executor skills. The frontmatter names an executor, so invoking the skill runs backend code instead (for example, calling the Tavily API or a sandboxed Python subprocess).

Built-in skills

These load automatically when relevant. You do not enable or invoke them by hand; you just ask.

SkillWhat it doesAsk something like
Web SearchReal-time web results (titles, URLs, snippets) via the Tavily API."What's the latest on X?"
Web FetchPulls the contents of a specific URL into the conversation."Read this page and summarize it."
Run Python sandboxedExecutes Python for calculations, parsing, or quick data work and returns stdout/stderr."Compute the median of these numbers."
Summarize TranscriptTurns the session transcript into one of four fixed styles: action items, meeting notes, key points, or a brief."Give me the action items from this call."
Meeting NotesA fuller, polished write-up: overview, decisions, an owner / action / deadline table, discussion points, blockers, and next steps."Write up minutes for this meeting."
TranslateTranslates the transcript or any text to a target language."Translate this into Kurdish."
Analyze DocumentDeep analysis of an attached file, with optional section fetches for large documents."Compare the two contracts I attached."
Presentation BuilderDrafts a structured slide deck from a topic or source material."Build me a 10-slide deck on this."
AWS Boto3 read-onlyDirect, read-only queries against AWS services via boto3 (list, describe, get)."List my S3 buckets."
Catch UpA plain-language recap for someone who joined late, plus suggested follow-up and clarifying questions."Catch me up on what I missed."

The three transcript skills - Summarize Transcript, Meeting Notes, and Catch Up - work from the live or recorded transcript, which the server supplies for you. A very long transcript is condensed automatically before these skills run, so a multi-hour session still summarizes cleanly; see Transcript summarization.

Web Search needs a Tavily API key. Set TAVILY_API_KEY in the environment (it takes precedence) or the tavily_api_key field in config.json. Without a key, the Web Search skill is unavailable and everything else keeps working. See Environment variables.

Seeing a skill run

When Claude invokes a skill, a small inline indicator appears in the message stream as the call starts, carried on the skill and skill_input SSE events. Click into that indicator to open the trace: which skill ran, the input it received, and what it returned. This is the same tracing you get for any tool call, so a skill never happens invisibly.

Turning skills on and off

Every skill is listed in Settings, Skills, where you can enable or disable each one and read its definition. Disabling a skill removes it from the catalogue, so Claude can no longer pick it. The on and off state is stored in data/skills_config.json, separately from the skill files themselves, so toggling a built-in skill off never edits or deletes it.

Writing a custom skill

A custom skill is one Markdown file in skills/. The frontmatter is the important part: name is how the skill is invoked, and description is what Claude reads when deciding whether this skill fits your request. Write the description the way you would explain the skill to a colleague, since it is the whole basis for selection. The optional triggers line lists example phrases that hint when the skill applies.

Here is a minimal prompt skill. Save it as skills/standup-notes.md:

---
name: standup_notes
description: Turn rough standup jottings into a tidy per-person update with blockers called out. Use when the user pastes standup notes.
triggers: standup, daily update, blockers, what I did, yesterday, today
input_schema:
  notes:
    type: string
    required: true
    description: The raw standup notes to format
---

Rewrite the notes as a clean standup update. For each person, list:
Yesterday, Today, and Blockers. Put anything that sounds like a blocker
under Blockers even if it wasn't labelled that way. Keep it terse.

The steps to add and use it:

  1. Create the file

    Drop your .md file into skills/, or use Settings, Skills, New to write it in the app. Files created in the UI are validated and hot-reloaded, no restart needed.

  2. Confirm it loaded

    Ask Claude to run skill_list, or open Settings, Skills. Your skill should appear with its description and trigger count.

  3. Just ask

    Say something that matches the description ("clean up these standup notes"). Claude selects the skill on its own. You do not have to type its name.

  4. Or invoke it by name

    If you want to force a specific skill, Claude can call skill_invoke with the skill's name and your input.

Selection lives and dies by the description. If a skill is being ignored, sharpen its description to say plainly when to use it, and list a few realistic triggers. If two skills overlap, disable the one you don't want so Claude isn't guessing between them.

skill_list and skill_invoke

Two tools let Claude work with skills directly, rather than relying only on automatic selection:

skill_list
Returns every available skill with its name, description, and triggers. Claude calls this to discover what is on hand before picking one.
skill_invoke
Runs a named skill. It takes a skill_name and an input string, then applies the skill to that input.

You rarely need to think about these. They exist so Claude can be explicit ("let me list the skills, then invoke the right one") when a request is ambiguous or when you name a skill outright.

Folder skills

A skill can also be a whole directory instead of a single file: a folder containing a SKILL.md plus its own scripts, references, and assets. These are loaded whole, and the skill's instructions can point Claude at bundled scripts to run. Folder skills are read-only in the UI (you edit the files on disk), and you can mark one trusted so that when it runs a script from inside its own directory, that specific command is auto-approved. Untrusted folder skills still surface the normal approval card. See Permissions & approvals for what trust does and does not grant.

Sandboxing and guardrails

Skills that touch the outside world run behind the same protections as every other tool.

Run Python is sandboxed

Code runs in a subprocess with a short timeout, inside the OS sandbox. It is meant for quick computation, not for running arbitrary programs against your machine.

AWS Boto3 is read-only

The Boto3 skill is guarded to allow only read operations (list, describe, get). Write actions are the job of a separate, gated path, not this skill.

The real boundary is the OS sandbox (sandbox-exec on macOS); the read-only Boto3 guard and the command validator are defence-in-depth on top of it. For the full picture of what contains a skill and what merely double-checks it, see Sandbox & security and the security model.

A skill's instructions run with Claude's tools. Treat an imported or shared skill like any code you did not write: read its SKILL.md and scripts before you trust it, and leave it untrusted until you have.

Where skills live

skills/
Skill definitions: one Markdown file per skill, plus folder skills as subdirectories.
server/skills.py
Loads the files, parses frontmatter, builds the tool catalogue, and serves the Skills API.
data/skills_config.json
Which skills are disabled or trusted, kept apart from the skill files.
