The workspace IDE

Connect a project folder and Whisper Studio becomes a coding assistant with a body. Claude can read, edit, run, and commit in your repo, while a file tree, Monaco editor, terminal, Git panel, and LSP let you work right alongside it in the same tab.

Everything here is the same one local process you already run. Chat lives on the right; the workspace panel is the IDE on the left. The moment you connect a folder, a set of workspace tools switch on and Claude gains the ability to act on real files, gated by the Mode you choose.

What you get

Connecting a workspace turns the left-hand panel into a full IDE. None of it needs a build step or a separate editor window.

File tree + search

A sorted tree (directories first) with a "Search files..." box that queries the backend and jumps you straight to a match.

Monaco editor

The same editor core as VS Code, with tabs, syntax highlighting, a breadcrumb, and a status bar showing your cursor line and column. Save with the usual shortcut.

Integrated terminal

An xterm.js pane wired to a real PTY over WebSocket. Type commands yourself; output streams back live and follows the app theme.

Git Changes panel

Sits under the file tree. Shows the working-tree status and lets you open any changed file to view its diff without leaving the panel.

Rich file viewers

Beyond code, tabs render images, PDFs, spreadsheets, Word docs, CSV/TSV tables, Markdown previews, and Jupyter notebooks by file type.

LSP: Python + TypeScript

Language servers back the lsp_diagnostics, lsp_hover, and lsp_references tools, so Claude can read real type errors and symbol usages.

Connect a folder

Use the Workspace control to point Whisper Studio at a project directory. Once a folder is connected, the workspace tools become available to Claude and the panel fills in with the tree and (as you open files) the editor.

  1. Open the Workspace panel

    The panel header reads Workspace and carries three buttons: refresh the file tree, disconnect, and collapse. Connect the folder you want to work in.

  2. Browse or search for a file

    Click through the tree, or type in the search box to filter across the whole project. Open a file and it lands in a Monaco tab (or the right viewer for images, PDFs, notebooks, and the like).

  3. Describe what you want

    Switch to the chat and ask in plain English: "add a retry to the fetch helper", "why does this test fail", "rename this component everywhere". Claude does the file work with tools.

Refreshing is automatic. When a tool writes a file (or you approve a change), the panel fires a workspace-refresh so the tree and open tabs pick up the new content. Non-dirty tabs reload in place; tabs with unsaved edits are left alone. The refresh button in the header is a manual override for changes made outside the app.

Point Claude at exact files

You do not have to describe where things live. Reference a file inline with @file:path/to/file and its contents are pulled into the prompt, so Claude reads the real code instead of guessing.

The tools Claude uses

With a workspace connected, Claude works through a dedicated set of tools. Reads run freely; anything that changes state routes through the approval gate described below.

ToolWhat it doesKind
ws_read_fileRead a file from the connected codebase.read
ws_grepSearch file contents with regex (ripgrep), globs, context lines.read
ws_globFind files by glob pattern.read
ws_list_directoryList files and directories.read
ws_edit_fileEdit a file by replacing a specific string.write
ws_write_fileOverwrite or modify an existing file (atomic write).write
ws_create_fileCreate a new file.write
ws_delete_fileDelete a file.write

git_*Full Git: status, diff, log, blame, commit, branch, merge, push, PR.read / write
lsp_*Diagnostics, hover, and references from the language servers.read

There are also worktree tools (ws_create_worktree, ws_diff_worktree, ws_merge_worktree) for keeping a change isolated on its own branch before you merge it back. For the complete catalogue, see Agent tools and the deeper tool dispatch writeup.

Running commands

ws_run_command is how Claude runs your tests, a linter, a build, or a quick script. Two behaviors matter here:

  • Long tasks auto-background. A command that does not finish quickly is moved to the background so the turn is not blocked waiting on it. Claude checks back on the output rather than hanging.
  • It is sandboxed. Every shell command wraps in sandbox-exec, the macOS kernel-level filesystem and network policy. This is not a courtesy check; it is the real boundary (see Sandbox & security).

You can also just use the terminal pane yourself. It is a genuine PTY, so an interactive REPL, a watch process, or a one-off command all behave the way they would in Terminal.app.

How much Claude does on its own

The Mode dropdown in the chat input sets how much autonomy Claude has over your files. The same modes that govern chat govern the workspace.

ModeBehavior in the workspace
defaultReads run freely; writes ask for approval.
acceptEditsFile edits auto-approve; deletes and commands still ask.
autoA classifier decides per-tool, per-call whether to allow, ask, or deny.
planRead-only. Claude plans and reads but never writes or runs anything.
bypassPermissionsEverything runs without prompts.

A sensible progression is to start in plan to see the intended change, switch to default to approve edits one by one, and only reach for acceptEdits once you trust the task. Per-tool rules in Settings can override the mode for specific commands. The full model is in Permissions & approvals.

The approval card

When a mode calls for consent, the guarded action pauses and an approval card appears with the concrete detail: for an edit, the exact diff; for a command, the exact command line. Nothing touches disk until you approve. Behind the card is a nonce-based flow, so a write cannot be slipped through by the client asserting it was already approved.

Hard guards stay on even at the most permissive mode. Shell commands always run inside the macOS sandbox, and a small set of destructive patterns are always blocked regardless of mode. bypassPermissions removes the prompts, not the sandbox. See Permissions & approvals and Sandbox & security for exactly where the line is.

Review the result

After a change lands, you have two ways to see it. The edited file updates in its Monaco tab (or opens as a diff), and the change shows up in the Git Changes panel under the tree. From there you can open a diff, keep editing, and, when you are ready, ask Claude to commit and push, or open a pull request, through the git_* tools.

  1. Inspect the diff

    Open the changed file from the Git Changes panel to review exactly what moved. Nothing here is committed yet.

  2. Adjust if needed

    Edit directly in Monaco, or ask Claude for a follow-up. Your unsaved edits are never clobbered by an auto-refresh.

  3. Commit and push

    Ask Claude to stage and commit (with git_add_commit), then push or open a PR. Each of these is a guarded action, so you approve the commit message and the push.

Disconnecting

The disconnect button in the panel header drops the workspace connection. The workspace tools go away with it, so Claude can no longer read or write files until you connect a folder again. Your session history and transcript are untouched.
