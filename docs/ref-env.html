Environment variables

A short list of shell variables that override stored config and control how the server binds and starts. The environment is the highest-priority config layer, so anything set here wins over the project and user layers.

The environment variables below are the ones you are likely to set: variables that steer binding and startup (HOST, PORT, BACKEND_PORT, NO_OPEN), one that opens up LAN access (WHISPER_TRUSTED_ORIGINS), and one that supplies a secret (TAVILY_API_KEY). Everything else is configured through the Settings UI or config.json. A few lower-level knobs also read the environment (TZ for cron scheduling, plus WHISPER_RG_PATH, WHISPER_SEARCH_TIMEOUT, WHISPER_GLOB_LIMIT, and WHISPER_GREP_LIMIT for the search tools), but the defaults suit almost everyone. See Configuration for how the layers stack.

Binding and startup

The three binding variables shape how the process comes up. The most common way to set them is a one-line prefix in front of the launcher:

HOST=0.0.0.0 PORT=8080 bash setup.sh --no-open

PORT is the backend port. Because the launcher auto-increments when the port is busy, the value you pass is a starting point, not a guarantee. If port 8000 is taken and you did not set PORT, the server climbs to 8001, 8002, and so on. setup.sh then propagates the resolved value into BACKEND_PORT so the Vite dev proxy points at wherever the backend actually landed.

BACKEND_PORT only matters in dev mode (bash setup.sh --dev), where the browser talks to the Vite dev server and Vite proxies API calls to the backend. In production mode (the default), FastAPI serves the SPA and the API on the same origin, so there is nothing to proxy and you can ignore BACKEND_PORT entirely.

Exposing on the LAN

By default the server binds 127.0.0.1, so it is reachable only from the machine it runs on. Setting HOST=0.0.0.0 binds every interface, but that alone does not make the app usable from another device. The Host-header guard rejects any request whose Host is not localhost, 127.0.0.1, ::1, or 0.0.0.0 with a 403 - and a LAN browser sends Host: 192.168.x.x, so every request, the initial page load included, is refused. To actually reach it from another device you must also set WHISPER_TRUSTED_ORIGINS to the host or origin you browse from:

HOST=0.0.0.0 WHISPER_TRUSTED_ORIGINS=192.168.1.50 bash setup.sh --no-open

The API is unauthenticated. Whisper Studio assumes it runs on localhost for a single trusted user. There is no login. When you bind HOST=0.0.0.0, anyone who can reach the port can drive the agent, run sandboxed commands, read your workspace, and spend your Bedrock budget. Do this only on a network you fully control, and prefer an SSH tunnel over raw LAN exposure.

The Origin/Host guard gives partial, not complete, protection here. It rejects cross-site browser requests, so a malicious web page you happen to visit cannot quietly script your local instance through the browser. It does not stop a deliberate non-browser client (curl, a script, another program on the LAN) from calling the API directly, because those clients simply send the expected Origin and Host headers. Treat the guard as a defence against drive-by browser attacks, not as authentication. See the security model for the full boundary picture.

Secrets

TAVILY_API_KEY enables the web-search skill. Reading it from the environment is the recommended path because it keeps the key out of the on-disk config.json and makes rotation trivial:

echo 'export TAVILY_API_KEY=tvly-...' >> ~/.zshrc
source ~/.zshrc

If you also paste a key into Settings, the environment value takes precedence. Without a key set by either path, web search is simply unavailable; chat, transcription, and the workspace tools all work as normal. See Web research for how the skill uses it.

Where the environment fits

Config resolves in three layers, highest priority first. The environment sits at the top, so a value exported in your shell overrides both the project and user layers:

Environment
Shell variables (HOST, PORT, BACKEND_PORT, WHISPER_TRUSTED_ORIGINS, TAVILY_API_KEY, NO_OPEN). Highest priority. Set per-launch or in your shell profile.
<workspace>/.whisper/settings.json
Project layer. Per-project model, region, and permission mode. Edited through the Settings UI.
config.json
User layer, at the repo root and gitignored. Defaults for all sessions. Edited through the Settings UI or by hand.

Only the binding and secret keys have an environment override. Most settings (model, region, effort level, permission mode, feature flags, cost caps) live only in the project and user layers. For the complete field-by-field list, see Settings & config keys.

The environment layer is shell-only. There is no field in the Settings UI for HOST, PORT, BACKEND_PORT, WHISPER_TRUSTED_ORIGINS, or NO_OPEN. To change them, set the variable and restart the server.
