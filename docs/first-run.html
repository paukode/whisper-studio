First run & verify

You ran bash setup.sh and it printed a URL. This page is the short checklist that confirms the three things you actually came for work: chat with Claude, on-device transcription, and the workspace tools.

Whisper Studio is one process serving one origin, so there is nothing to "connect" between a frontend and a backend. What is left is telling the app which AWS region and model to use, then exercising each subsystem once. Work through the steps in order; each one leaves you in a good state for the next.

New machine, nothing installed yet? Do the Installation walk-through first, then come back here. This page assumes setup.sh has already finished and given you a URL.

The healthy-install checklist

Five steps. The first two are one-time configuration; the last three prove each subsystem is alive. If all five pass, your install is healthy.

  1. Open the URL setup.sh printed

    Look at the end of the terminal output for a line with an address, typically http://127.0.0.1:8000. If port 8000 was busy, the script auto-increments, so read the URL it actually printed rather than assuming 8000. Open it in a browser. You should see the Whisper Studio interface, not a blank page. (A blank page usually means the frontend bundle is missing; re-run bash setup.sh, or bash setup.sh --new for a clean rebuild. Note that --new also resets config.json to the template after a timestamped backup, so you would then need to re-add your Tavily key and any custom settings, or restore them from the backup.)

  2. Set your region and model in Settings

    Click the gear icon to open Settings, then fill in:

    • Bedrock Region must match the region where your Claude model is enabled. This is almost always the same region you used in aws configure. A mismatch here is the single most common first-run error (see below).
    • Default Model should start with Sonnet. It is the best balance of speed, cost, and capability while you are confirming things work; you can switch to Opus or Haiku later with the toolbar or /model.
    • Tavily API Key only if you did not export TAVILY_API_KEY in your shell. If the env var is set, the app uses it and this field can stay empty (the env var wins if both are present). Skip this entirely if you do not need web search yet.

    Click Save.

  3. Confirm chat: type "hello"

    In the chat input, type hello and send it. If Claude streams a reply back token by token, chat works and your AWS access is good. If you get an error instead, jump to Common first-run stumbles below; the message text tells you which problem you have.

  4. Confirm transcription: click Record

    Click Record in the top-right. The browser asks for microphone permission the first time; grant it once. Speak a sentence or two. A live transcript should appear with speaker labels attached. The first recording is slightly slower to start because the on-device speech model (Whisper or Parakeet) auto-downloads and loads into memory on first use; after that it is warm. Transcription runs entirely on your machine, so it needs no AWS setup and works even offline.

  5. Optional: connect a workspace folder

    To unlock the file, code, and Git tools, connect a folder with the Workspace control. Once a workspace is connected, Claude can read files, edit code, run commands, and use Git, all gated by your permission mode. You can skip this if you only want chat and transcription; connect a folder whenever you are ready to code.

If steps 3 and 4 both succeed, you have a working install. Chat and transcription are the two independent subsystems, so passing both means AWS, the local model runtime, and the browser permissions are all in order.

Run /doctor for an automated check

Rather than eyeballing each subsystem, you can let the app check itself. Type /doctor in the chat input and send it. This runs an automated dependency and AWS-access check and reports the results inline in the chat.

A healthy result confirms the pieces the manual checklist covers, without you having to record or connect anything:

  • The system dependencies setup.sh installs (such as ffmpeg and ripgrep) are present and found on the path.
  • AWS credentials resolve and Bedrock is reachable in your configured region, so a chat turn can actually reach a model.
  • The configured Default Model is enabled and callable in that region (this is exactly the check that a plain hello would exercise).

If /doctor flags something, its output points at the specific failing check, which is faster than reading a raw exception from a failed chat turn. Run it any time later, too, if chat suddenly stops working after an AWS or config change.

Backend changes do not hot-reload. If you edit a server/*.py file, restart the process before re-running /doctor, or you will be checking the old code. The frontend, in --dev mode, does reload on its own.

Common first-run stumbles

Almost every failed first chat is one of two AWS problems, and the error message tells you which one. Read the text of the error before changing anything.

AccessDeniedException (or "Could not invoke model"). Your AWS user does not have Bedrock permission, or the Claude model is not enabled in the region you selected. Grant the user Bedrock access in IAM, and enable the specific Claude model in the Bedrock console for that region.

"You don't have access to the model with the specified model ID." This is almost always a region mismatch. The model is enabled somewhere, just not in the region Settings points at. Make the Bedrock Region in Settings match the region where you enabled the model (and where aws configure is pointed). This is the same failure the Installation guide's troubleshooting table calls out.

A few non-AWS stumbles round out the set:

SymptomWhat it means and the fix
Blank page in the browser The frontend bundle is missing. Re-run bash setup.sh, or bash setup.sh --new for a clean rebuild. Note that --new resets config.json to the template (after a timestamped backup), so re-add your keys and settings afterwards.
The URL at 8000 does not load Port 8000 was in use, so the script chose another port. Read the URL the script actually printed rather than typing 8000 by hand.
Microphone permission never prompts The browser may have remembered a denial. Re-grant microphone access for the site in the browser's site settings, then click Record again.
Recording misses system audio The default mic does not capture what macOS is playing. Set up a virtual input device; see Voice & meetings.

For anything not listed here, the Installation guide has a fuller troubleshooting table covering prerequisites, network errors, and ports.
