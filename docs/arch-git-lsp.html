Git integration & LSP

Whisper Studio shells out to your real git binary, but never carelessly. Reads run freely; writes stop for approval; a one-second watcher keeps the SPA in sync; and a small proxy tunnels editor traffic to language servers.

The git layer lives under server/git/ and has three jobs: answer questions about the repo (branch, status, diff, log), let the assistant make changes safely behind the approval flow, and tell the frontend the moment something in .git moves. Alongside it, two separate LSP surfaces give the assistant and the editor code intelligence. Click a box in the diagram to trace a path.

Git core

Everything starts in server/git/core.py. It is deliberately plain: synchronous functions, subprocess.run() for git, and direct file reads for the fast paths. Two pieces do most of the work.

Root discovery

find_git_root(start_path) walks up the directory tree looking for a .git entry. It accepts either a directory (a normal repo) or a file (a worktree or submodule, where .git is a gitdir: pointer). The resolved root is normalized to Unicode NFC and cached in a plain dict capped at 50 entries, evicting the oldest key when full. Because the SPA panel, prompt building, and every status call hit this repeatedly, the cache turns a tree walk into a dict lookup.

A companion, find_canonical_git_root(), resolves a worktree back to the main repo identity by following the gitdir: then commondir chain. Two structural checks guard it: the worktree gitdir must be a direct child of <commonDir>/worktrees/, and its gitdir back-link must point at <gitRoot>/.git. Both are required, so a hand-crafted repo cannot borrow another checkout's trust.

The subprocess wrapper

_run_git(args, cwd, timeout=15) is the single choke point for running git. Its hygiene is the point:

  • No credential hang. It sets GIT_TERMINAL_PROMPT=0 and empties GIT_ASKPASS, and closes stdin (subprocess.DEVNULL). A missing credential fails fast instead of blocking the whole process on an invisible prompt.
  • A time budget. The default timeout is 15 seconds (pushes bump it to 30). A wedged network operation cannot stall a chat turn indefinitely.
  • Bounded output. Tool results run through _truncate() (server/git/executor.py), which caps text at roughly 50,000 characters or 200 lines, whichever comes first, and appends a note when it clips. A giant diff will not blow up the token bill.

The git executable itself is resolved once with shutil.which("git") and memoized.

Diffs

server/git/diff.py is the most careful reader in the subsystem, because diffs can be arbitrarily large. It supports four shapes:

ShapeHowNotes
Working tree vs HEADgit diff HEADThe default for the panel and the git_diff tool.
Staged (cached)git diff --cachedWhat a commit would capture right now.
Single file, PR-styleAgainst a merge basefetch_single_file_git_diff() diffs a file against the merge base with the default branch, falling back to HEAD.
Untracked fileSynthetic diffA brand-new file is rendered as one hunk of pure additions.

Size safeguards are baked in as constants: at most 50 files get per-file detail, files over 1 MB are skipped, and each file is capped at 400 lines of hunk content (GitHub's auto-load limit). A cheap --shortstat probe gives accurate totals in O(1) memory first, so the expensive numstat and hunk parsing only run when the change set is small enough to be worth it.

is_transient_git_state() checks for MERGE_HEAD, REBASE_HEAD, rebase-merge/, CHERRY_PICK_HEAD, and similar markers. During a merge or rebase the working tree holds incoming changes rather than your edits, so diffing would be misleading. In those states the diff functions return nothing until the operation finishes.

Git tools

Each git capability is a registered tool in server/git/executor.py. They split cleanly into two tiers by whether they change the repo.

Read tools (no prompting)

These are registered read_only=True, concurrent_safe=True, so they run without an approval card and can batch in parallel with other reads (see Tool dispatch & executors). Most add --no-optional-locks so they never contend with a concurrent git process.

ToolRuns
git_statusgit status (optionally --short).
git_diffWorking-tree, --cached, --stat, or a single-file PR-style diff.
git_loggit log with a limit (default 20), oneline or formatted, optionally per-file.
git_showgit show <ref>, optionally --stat.
git_blamegit blame, optionally scoped to a line range.
git_branch_listgit branch, optionally -a or -r.
git_stash_listgit stash list.

Every one first calls _get_git_cwd(), which returns a friendly error when no workspace is connected or the workspace is not a repo, so the assistant gets clear feedback instead of a raw subprocess failure.

Write tools (approval-gated)

Registered read_only=False, concurrent_safe=False, these do not run their git command directly. Instead they return a [WS_APPROVAL] payload, which surfaces an approval card in the UI. Only after you confirm (or the session is pre-approved for CLI actions) does server/approval/bootstrap.py call the matching do_* helper that runs the real command. Each helper returns a uniform (ok, output) tuple.

ToolAction
git_add_commitStage files (or -A) and commit with a message.
git_pushPush, setting upstream by default.
git_push_prPush the current branch, then open a PR via the gh CLI.
git_create_branchgit checkout -b, optionally from a ref.
git_checkoutSwitch to an existing branch.
git_delete_branchDelete a branch (-d, or -D when forced).
git_mergeMerge a branch, optionally --no-ff.
git_stashpush / pop / apply / drop.
git_cloneClone a URL and optionally connect it as the workspace.

Two of these carry extra guardrails. git_push_pr refuses to run while you are on the default branch and tells you to make a feature branch first. git_clone is the one write tool that does not require a pre-existing workspace, so it validates the URL against a strict allowlist (https://, ssh://, or git@host:path, no shell metacharacters, no file://) and refuses a non-empty destination.

The secret-file guard

Before git_add_commit stages anything, contains_secret_files() in server/git/security.py matches every path against a pattern list and aborts the commit if any look sensitive. The patterns include .env and .env.*, *.pem, *.key, credentials.json, service-account*.json, id_rsa, id_ed25519, and catch-alls like *secret* and *password*. The same module also classifies read-only commands for auto-approval, blocks writes to git config and flags like --no-verify and -i, and warns on destructive shapes such as a force push to main or master.

The secret guard is a safety net, not a boundary. It matches on file names, not contents, so a secret stored under an innocuous name can still slip through. Treat it as defense-in-depth on top of a good .gitignore. See the security model for what is a hard boundary versus a best-effort check.

The change watcher

server/git/watcher.py keeps the frontend honest without polling the whole repo. The GitFileWatcher singleton runs one daemon thread that stats three paths once per second: .git/HEAD, the git config, and the loose ref file for the current branch. When any mtime changes, it marks its lazy cache dirty and fires subscribers.

The subscriber is the SSE endpoint GET /api/git/events (server/git/router.py). It emits one git-changed event immediately with the current branch and head SHA, then one on every subsequent change, coalescing bursts and sending a keep-alive comment every 30 seconds so proxies do not drop the idle stream. This replaced a 15-second frontend interval, so the Git panel refetches only when state actually moved. If HEAD changed, the watcher re-resolves which branch ref to watch, so switching branches keeps working.

The same cache backs the hot read helpers get_cached_branch(), get_cached_head(), get_cached_default_branch(), and get_cached_remote_url(). Because the watcher invalidates on the exact files those values derive from, they stay correct with no timer and no manual invalidation.

Git REST endpoints

The frontend reads git state directly through server/git/router.py (prefix /api/git). These are read and UI-support routes; writes are not exposed as REST here, they go through the approval-gated tools above.

GET /api/git/status
Branch, clean flag, and a per-file status list for indicators.
GET /api/git/changes
Combined status + diff in one round trip, behind a 1-second TTL LRU cache that absorbs the assistant's tool-loop and the panel's auto-poll.
GET /api/git/branch
Current branch, default branch, and whether they match.
GET /api/git/diff
Diff summary (file count, lines added/removed, per-file stats) for the panel.
GET /api/git/show
The HEAD version of a file, for side-by-side diffing in the editor.
POST /api/git/restore
Undo changes to a single tracked file, or remove an untracked one.
GET / POST /api/git/worktrees
List worktrees, add one, or remove one (never the active workspace).
GET /api/git/events
The SSE change stream described above.

Code intelligence: two LSP surfaces

Whisper Studio exposes language-server features in two distinct ways, which is worth keeping straight.

Assistant tools

server/lsp.py (prefix /api/lsp) backs the tools lsp_diagnostics, lsp_hover, and lsp_references. These are pragmatic static-analysis helpers rather than a live language server: diagnostics run pyflakes (falling back to py_compile) for Python and ESLint for JS/TS, hover extracts the symbol under a line and column and shows surrounding context, and references do a bounded workspace grep (capped at 100 matches). They work with nothing installed and get richer as you add tools, which suits the assistant's needs without spinning up a server per call.

The LSP proxy (editor)

The Monaco editor in the workspace is a real language-server client, so server/lsp_proxy.py provides a WebSocket-to-stdio tunnel at ws://.../ws/lsp/{language}?workspace=/path. Each connection spawns a language server subprocess and relays JSON-RPC both ways, translating between plain JSON on the socket and LSP Content-Length framing on the process's stdio. Two servers are wired up:

LanguageServerCommand
Pythonpython-lsp-serverpylsp
JavaScript / TypeScripttypescript-language-servertypescript-language-server --stdio

The proxy checks that the binary is on PATH before accepting, and closes with a helpful install hint if it is missing. On disconnect it terminates the child (then kills it after a 3-second grace) so no orphaned language server lingers. If the servers are not installed, the editor still works; you just do not get inline diagnostics, hovers, or completions there.

On the browser side, the useLsp hook (src/hooks/useLsp.ts) is the language client. When a Python, TypeScript, or JavaScript file is open in a connected workspace it opens the WebSocket, runs the initialize / initialized handshake, and sends textDocument/didOpen plus debounced textDocument/didChange (full-text sync) as you type. It maps publishDiagnostics to Monaco markers, and registers completion and hover providers backed by textDocument/completion and textDocument/hover, translating the 0-based LSP positions to Monaco's 1-based coordinates in both directions. A small status dot in the editor's top-right corner shows whether the server is connecting, connected, or unavailable. Everything is torn down on file switch, language change, or disconnect: providers disposed, markers cleared, and the socket closed after a shutdown/exit. For unsupported languages or when no workspace is connected, the client is a no-op and no socket is opened.

Want the full editor experience? Install the servers with pip install python-lsp-server and npm install -g typescript-language-server. The proxy picks them up on the next connection.
