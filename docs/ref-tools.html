Agent tools

The assistant is agentic: it calls tools to read and change your workspace, search the web, run git, remember things, schedule jobs, and coordinate sub-agents. This page lists every tool, grouped by area, with what each one does and whether it needs your approval.

With progressive tool disclosure (feature flag progressive_tools, default on) the interactive chat loop advertises to cloud Claude only a core set plus the tools this session has activated; every other tool lives as a one-line entry in a deferred index and loads on demand via tool_search. The full pool still goes to sub-agents and cron. On-device Gemma can also gate its set with a per-model Tools dropdown (local only). Tools that change files, run commands, or alter git or config state pause for your approval before they take effect; read-only tools run without prompting.

How tools reach the model

Tool definitions are part of the input the model reads before your message. But the interactive chat loop no longer sends the whole catalogue every turn: with progressive tool disclosure (progressive_tools, default on) only the core and session-activated schemas are sent, while every deferred tool costs a single index line in the cached system block (build_deferred_index) rather than a full schema. That cuts roughly 60-75% of tool-schema tokens per turn, and it applies to cloud Claude too. The ~8K-token full catalogue now describes the non-progressive path (sub-agents and cron), not an interactive cloud turn. For cloud Claude the schemas bill as input tokens (a cost concern, not a speed one); for local Gemma there is no bill, but the extra tokens slow the on-device prefill, which is why you can also shrink the set with the Tools dropdown.

Dispatch itself is a pure name-to-function mapping. Each executor is registered with safety metadata (read_only, concurrent_safe, destructive) that the tool loop uses to decide what can run in parallel and what must pause for consent. Defaults are fail-closed: a tool is treated as unsafe unless it declares otherwise. See Tool dispatch & executors for the mechanics.

Throughout the tables, a ★ Core badge marks a tool in the local Core set (about 11 tools) that the Tools dropdown keeps when you trim the catalogue. This is distinct from the larger progressive-disclosure core set (CORE_TOOLS, ~34 names in server/chat/tool_partition.py) that cloud Claude always advertises: the two overlap but are not the same list. See the Tools dropdown below.

Approval at a glance

Two badges appear in the tables. Read the Approval & permissions page for the full nonce flow and per-tool rules.

read Runs freely

Read-only tools (reading files, searching, viewing git history, fetching pages) run without a prompt. Consecutive read-safe tools can also run concurrently in one batch.

approval Pauses for consent

Tools that write files, run commands, or change git or config state issue a nonce, pause the turn, and wait for you to confirm. The same gate applies to both Claude and Gemma.

ToolWhat it doesApproval

Tasks & scheduling

Three related families. The todo tracker (task_*) tracks the steps of a multi-step workflow within a single turn or session; the background-task registry (task_status/task_output/task_cancel) checks and stops detached shell commands, agents, and workflow/CI runs; cron jobs schedule prompts to run later, whose output streams back into the chat that created them. See Cron / scheduled tasks and the cron tutorial.

The scheduler also exposes cron_update (edit a job's prompt, schedule, or enabled state) and cron_run (run a job once, off its schedule). The three listed above are the ones surfaced in the primary Tasks & scheduling group.

Sub-agents

The assistant can delegate work to independent agents, each running its own full tool loop, and coordinate them as a team. Agent types range from a read-only explore to a test-running verify. See Sub-agents & the event bus and the sub-agents tutorial.

ToolWhat it doesApproval

send_messageSend a message to another agent by ID, or broadcast to all agents.read

Each brief you write for spawn_agent or a team_create member must be self-contained: the agent sees none of the parent conversation, so restate the objective, scope, constraints, and expected output. A spawned agent inherits the session-selected model rather than an agent-type default.

Skill-backed tools

Beyond skill_invoke, every skill installed under skills/ is also advertised to the model as its own top-level tool, named after the skill (dashes become underscores). So the model can call, say, meeting_notes or terminal_run directly, without going through skill_invoke first. See Skills for what each one does and how to add your own.

The default set ships with these skill tools:

  • Command, code, and cloud execution - terminal_run, run_python, aws_cli, and aws_boto3. These run commands, Python, or AWS calls, so they pause for approval like the other command tools.
  • Web - web_search and web_fetch, listed in the Web section above. Read-only.
  • Content and transformation - analyze_document, summarize_transcript, meeting_notes, catch_up, translate_text, presentation_builder, create_program, frontend_design, github_actions, feature_flags_architect, and self_eval. These generate or reshape content and run without a prompt.

Because this set is driven by the installed skills rather than hardcoded, the exact list depends on what is in skills/. Adding or removing a skill adds or removes its tool, so the pool grows with your skill folder.

These tools appear in the catalogue only when ultracode is active (assemble_full_catalog(ultracode=True)), so you will not see them in other modes. workflow_run and ci_autofix return an approval preview card: the generated workflow runs only after you approve it. The goal loop is not a tool. It is surfaced through the /goal command and the GoalBanner UI (there are no goal_* tools), so it is documented in Slash commands, not here.

The Tools dropdown (local models)

Cloud Claude is trimmed automatically by progressive disclosure (core plus session-activated tools), not by this dropdown. For on-device Gemma, the per-model Tools dropdown lets you shrink the catalogue further so fewer schema tokens slow the prefill. The dropdown itself has no effect on cloud models.

ScopeTools declaredUse when
OffNonePlain chat, fastest prefill.
Core setThe ★ Core tools (about 11)Everyday file, code, and git work.
Core + webCore set plus web_search and web_fetchCore work plus internet lookups.
All toolsEverything on this pageFull capability (heaviest, slowest prompt).

See Model modes for how local, hybrid, and cloud selection interact with this dropdown, and Model backends & modes for the routing underneath.
