Development & contributing

How to run the checks, find your way around the codebase, and land a change in the right layer. This page is for people working on Whisper Studio; if you just want to run it, start with Installation.

Whisper Studio is one FastAPI process serving one origin: a Python backend under server/ and a React 19 SPA under src/, built and served together by setup.sh. Get a working install first (see Installation), then use the checks and map below to make changes.

Running tests & checks

Activate the virtualenv that setup.sh created, then run the backend and frontend checks. These four commands are the same gate CI runs:

source venv/bin/activate

pytest tests/ -q      # backend unit + smoke tests
npm run build         # tsc -b + vite build (catches type and build errors)
npm run lint          # ESLint over src/
npm test              # vitest frontend unit tests

Python is linted and formatted with Ruff, a blocking CI gate. Run it before you push so the pipeline stays green:

ruff check .          # lint
ruff format .         # format

Run /doctor in the chat input for an automated dependency and AWS-access check. It is the fastest way to confirm a fresh checkout is wired up correctly.

Conventions

Backend edits need a restart
Backend modules are imported once at startup, so edits to server/*.py do not take effect until the process restarts. setup.sh only restarts if the old process is killed first - lsof -ti :8000 | xargs kill.
Frontend is strict TypeScript + Zod
The SPA uses TypeScript strict mode. SSE event payloads are validated at runtime with Zod schemas in src/types/, so a malformed event fails loudly rather than corrupting state.
Tools have three homes
Tool schemas live in server/agent_tools.py, dispatch in server/tool_router.py, and the implementation in server/executors/. Add a tool by touching all three.
Never reinvent the approval flow
Approval-gated server actions go through server/approval/registry.py, which holds the nonce. Do not build a parallel approval path - see Approval & permissions for why.
Keep files under the size budget
CI fails any server/ or src/ file over 1200 lines. Split a growing module into a focused component rather than adding it to the allowlist.

Contributing

Small fixes: open a pull request. For anything larger, read the architecture overview and the relevant subsystem page first, so the change lands in the right layer and reuses the existing approval, tool, and persistence plumbing instead of working around it. Match the surrounding code - the backend and frontend both have a consistent house style, and CI enforces the linting and file-size rules above.
