Transcript summarization & condensation

A multi-hour meeting still summarises cleanly because the server condenses an oversized transcript with map-reduce before the model ever sees it - in cloud, local, and OpenAI-on-Bedrock modes alike.

The three summary skills work by injecting the whole transcript into the model's context so it can summarise in a single pass. That is the best-quality path while the transcript fits the context window. A long session can outgrow it, and the old choices were both bad: silently truncate (and answer from a partial transcript) or overflow the request (a hard failure). Condensation is the third answer - shrink the transcript first, keeping the material that matters, so the one-pass summary still runs.

Click a box below to trace where the transcript flows. Notice the two-part shape: the map step extracts raw material from each chunk, and the reduce half is the main chat model writing the summary in its normal turn.

The summary skills

Three folder skills turn a transcript into a written summary. None of them run anything destructive - the assistant simply writes the summary. The server supplies the live or recorded session transcript automatically, so these skills take no transcript text from you.

SkillWhat it does
summarize_transcriptSummarises the session transcript in one of four fixed styles - action_items, meeting_notes, key_points, or brief. Takes no text input.
meeting_notesA fuller, polished write-up: overview, decisions, an owner / action / deadline table, discussion points, blockers, and next steps. Also condenses a huge pasted notes argument the same way.
catch_upA plain-language recap for someone who joined late, plus suggested follow-up and clarifying questions.

See the Voice & meetings tutorial for how to invoke a summary from a recording, and Skills for how folder skills are written and loaded.

When condensation fires

Condensation is self-gating. Below threshold_chars (default 600,000 characters, about 150k tokens) the transcript passes through unchanged, with no map step and no LLM calls at all - most conversations never touch the condenser. Only an oversized transcript is condensed, and because that path blocks on real completions it runs off the event loop (run_in_executor) rather than stalling the request.

The condensed result is cached in a bounded LRU (max 8 entries) keyed by the transcript hash plus the config signature. The frontend re-sends the whole transcript on every turn, so identical input is condensed once and reused. A growing live recording changes the hash and re-condenses occasionally, which is intended.

Map, then reduce

The condenser is called once, in chat_endpoint, right after the model is resolved - the single point where the transcript enters the request. It feeds both the "[Transcript so far]" block and the transcript handed to the summary tools, so the first turn never overflows.

  1. Split

    The transcript is cut into overlapping, line-anchored chunks by the same structure-aware chunker the workspace index uses. The count is capped at max_chunks (default 40): the split re-chunks coarser first, then hits a hard cap.

  2. Map: extract, don't summarise

    Each chunk gets one fast, non-streaming extraction call. The prompt pulls out raw material - attendees and speakers, decisions, action items as owner|task|deadline, discussion points, blockers, open questions, and uncertain terms - it does not prose-summarise. Extract-not-summarise is the quality crux: owners, decisions, and numbers must survive the shrink. Cloud fans the map calls out with bounded concurrency to cut latency; local runs them serially, because a single on-device instance cannot do concurrent completions.

  3. Reduce

    The extracts are concatenated in order and prefixed with a NOTE saying they are per-segment extracts of reduced fidelity, then substituted for the raw transcript. The main chat model composes the requested summary from that material in its normal turn - that is the reduce half.

The condenser never returns empty. One failed chunk yields "" and is skipped; if every chunk fails, it falls back to a hard-truncated transcript. The caller always gets usable text. The reduce itself is not a separate LLM call - it happens inside the normal chat pipeline and tool loop.

Every model mode

The map step follows the active model mode. In cloud mode it maps with Haiku (an Anthropic invoke; if the configured Haiku key is missing it falls back to another Claude model, never to a non-Claude id). In local mode it maps with the resident, active on-device model: the local key resolver picks the current chat model, else the currently loaded model (loaded_key()), else the first downloaded model, so a summary never evicts the chat model to load a fixed one, and an is_downloaded guard means a summary never triggers a silent multi-GB download.

The condensed transcript is now threaded into the tool loop on the local and OpenAI-on-Bedrock paths too. Previously only the cloud path passed it through, so summarize_transcript returned "No transcript available to summarize." in those modes at any transcript size. Configuration

The top-level map_reduce_summary block tunes the condenser. All sizes are in characters (roughly 4 characters per token), not tokens. A malformed numeric value coerces to its default rather than crashing a chat turn.

KeyDefaultMeaning
enabledtrueMaster switch. When off, no condensation runs and even an oversized transcript passes through unchanged, which can overflow the request.
threshold_chars600000Condense only above this size; smaller transcripts pass through with no LLM calls.
chunk_chars150000Per-chunk size for the cloud window (about 37k tokens).
local_chunk_chars40000Per-chunk size for local (about 10k tokens, to fit the 16k local window).
overlap_chars800Overlap between adjacent chunks.
map_max_tokens1200Output cap per map extract.
max_chunks40Cap on the number of chunks; re-chunk coarser, then hard-cap.
max_output_chars480000Cap on the aggregate condensed output for cloud (about 120k tokens).
local_max_output_chars24000Tighter aggregate cap for the small local window.
engine"auto""auto" follows the model mode (cloud to Haiku, local to local); or force "haiku" / "local".

The mode-aware one-shot completion, resolve_map_engine, and local-key resolution; raises on hard failure so the condenser can fall back explicitly.

The single condense call site in chat_endpoint, right after the model is resolved.
