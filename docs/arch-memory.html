Memory system

Whisper Studio carries context across turns and across sessions with three distinct stores: per-session memory that summarizes the conversation so far, persistent memory that survives every session in a global and a per-project tier, and a hand-written WHISPER.md for project context. All three are assembled into the system prompt before the model is ever called.

There is no single "memory blob". Each store has its own file layout, its own update cadence, and its own wrapper tag in the prompt. Extraction happens after a turn as a fire-and-forget background task; recall happens before a turn to pick what the model reads. Click a box in the diagram to trace how a store feeds the next turn.

Three kinds of memory

It helps to keep the three straight from the start. They differ in scope, in who writes them, and in how they are wrapped when injected.

Session memory

Session memory (server/memory/session_memory.py) is a single markdown file per conversation with four fixed sections: Goals, Decisions, Context, and Blockers. It lives at data/session_memory/<session_id>.md and is meant to answer one question: if you resume this conversation later, what does the model need to pick up where it left off?

When it updates

After each turn, maybe_update_session_memory() runs in the background. It tracks, per session, the character count at the last update and the message index it last processed, then decides whether an update is due:

  • Character growth. Characters added since the last update must cross a threshold: TOKEN_THRESHOLD_CHARS (200,000 chars, roughly 50K tokens) for cloud models, or LOCAL_TOKEN_THRESHOLD_CHARS (6,000 chars, a few turns) for on-device models.
  • Tool activity (cloud only). Cloud sessions additionally wait for at least TOOL_CALL_THRESHOLD (10) tool_use blocks since the last update. Local chats often have no tools, so the character threshold alone gates them.

When both applicable guards pass, the summarizer runs against the last 30 turns (each message capped for the excerpt), writes the new markdown, and the per-session counters advance. The result is capped: each section stays under MAX_SECTION_CHARS (2,000 chars) and the whole file under MAX_TOTAL_CHARS (12,000 chars), truncated if the model overshoots.

Who does the summarizing depends on the model. Cloud turns use the memory_extractor agent (see Sub-agents); on-device turns run the summary on the local model itself, on its own executor thread, so a local conversation's memory never leaves the machine. Either way, a failed summary degrades to no update rather than disrupting the chat.

How it enters the prompt

On the next turn, get_session_memory_context() reads the file and wraps it:

<session-memory>
## Goals
- Ship the memory docs page and wire it into nav

## Decisions
- Use the real prompt tags, not the outline's placeholders

## Context
- Cloud path uses Haiku for recall; local path stays offline

## Blockers
(none)
</session-memory>

If the file does not exist yet, the wrapper is empty and nothing is injected.

Persistent memory: two tiers

Persistent memory (server/memory/memdir.py and server/memory/recall.py) survives across sessions. It is a two-tier store, and every memory belongs to exactly one tier:

TierLocationAlive whenHolds
Globaldata/global_memory/Always, including plain chat with no workspace openUser preferences, role, cross-project feedback
Projectdata/memory/<workspace-slug>/A workspace is open (slug = sanitized basename of its path)Goals, deadlines, repo-specific references

The global directory deliberately sits outside data/memory/ so it can never collide with a workspace slug. Both tiers share the same on-disk layout:

MEMORY.md
A one-line-per-entry index of that tier's topic files, auto-generated from their frontmatter after every write or delete. Capped at 200 lines and 25 KB when loaded; if it exceeds either limit it is truncated and a warning is appended telling the model to move detail into topic files.
<topic>.md
Per-topic markdown with YAML frontmatter (name, description, type). The type is a closed taxonomy: user, feedback, project, or reference.
.cursor.json
Extraction cursors keyed by session id, one index per session (not a single "last message index"). Persisted in the global tier so a session keeps its place even when it connects or disconnects a workspace mid-life.
.dream_meta.json
Per-tier consolidation bookkeeping (last run time and turn-end count). See consolidation below.

Recall: choosing what to inject

Because the two tiers together can accumulate many topic files, they are not all injected. Before each turn, recall_memory_context() scans both live tiers and chooses which files are relevant:

  1. Scan both tiers

    List the topic files across global and project. If there are none, nothing is injected.

  2. Few files? Take them all

    At or below MAX_SELECTIONS (5) files, every file is returned and no model call is made.

  3. Local turn? Stay offline

    For on-device models, the ranking side-query is skipped and the most recent files (up to 5) are returned. This trades smart ranking for isolation.

  4. Otherwise, rank with Haiku

    A cheap Haiku side-query receives a manifest of file names, types, and descriptions plus the user query, and returns up to 5 filenames as JSON. If it fails, recall degrades to the most recent files.

The Haiku call is deliberately tiny: MAX_RECALL_TOKENS is 256 and it runs with a single retry and short timeouts. It then reads the selected files, always prepends each tier's MEMORY.md index, and tags each entry with a freshness note ("today", "yesterday", or "N days ago"). The whole block is wrapped in <memory-context>.

Extraction: writing new memories

After a turn (when the model is not asking for more tools), maybe_extract_memory() fires in the background. It runs with or without a workspace: with one the agent sees both tiers and routes each write by scope; without one, everything lands in global memory. It is guarded four ways:

  • Feature flag. auto_memory must be enabled. ensure_global_memory_dir() returns nothing when it is off, so the whole pipeline (and recall) short-circuits.
  • Throttle. Extraction runs only every DEFAULT_EXTRACT_INTERVAL (3) turns.
  • Mutual exclusion. If the model already called memory_write this turn, the cursor advances and extraction is skipped, so the model and the agent never fight over the store.
  • Cursor. Cursors are keyed by session id and persisted in the global tier; only messages after this session's saved cursor are considered, then the cursor advances.

When it does run, the memory_extractor agent gets an excerpt of the new messages (last 20, truncated) plus a manifest of existing memories per tier, and is told to save what would be valuable in future sessions. It runs at depth=1 so it cannot recurse into further extraction.

Extraction is opinionated about what not to save: code patterns, git history, and one-off debugging fixes are all derivable from the code itself and are skipped. It stores durable facts instead, such as user preferences and role, working guidance, project goals and deadlines, and pointers to external systems.

WHISPER.md

WHISPER.md is a plain markdown file at the workspace root, loaded on every turn by get_whisper_md_context() in server/skills.py. It is the Whisper Studio analogue of CLAUDE.md: durable project context that you author by hand rather than have the system extract. It is injected inline as a [WHISPER.md - project-specific instructions] block.

Unlike the other two stores, it is fully under your control. The in-app Project Memory dialog is backed by a small CRUD API:

EndpointAction
GET /api/whisper-mdRead the current file (returns content plus whether it exists).
PUT /api/whisper-mdSave new content to the workspace root.
DELETE /api/whisper-mdRemove the file.

Because it is read every turn and never summarized, keep it tight. It is a good home for stable conventions (coding style, deploy steps, domain vocabulary), not for transient task state.

Assembling all three

All three stores are resolved in the chat endpoint (server/chat/routes.py) and handed to the prompt builder in server/prompts/__init__.py, which owns section ordering. Session memory is placed before workspace memory (it carries a higher-priority slot), and WHISPER.md is its own project-instructions section. When prompt caching is on, these dynamic sections sit after the cacheable static prefix so they do not invalidate the cache. See Chat & tool orchestration for where this fits in a turn, and Settings for the caching flag.

Memory tools

Beyond the automatic pipeline, the model can manage persistent memory directly during a turn via four tools defined in server/memory/tools.py. They are registered through the executor, not exposed to the model as HTTP routes. Each read, write, and delete takes an optional scope parameter (global or project) that routes to the right tier; without it, a write updates a matching file in place or routes a new file by type, and reads and deletes search the project tier first, then global.

ToolWhat it does
memory_readRead a memory file by relative filename (optional scope).
memory_writeCreate or overwrite a topic file (frontmatter is added automatically) and rebuild the index (optional scope).
memory_listList every memory file across both tiers with type, description, and modification date.
memory_deleteDelete a memory file (optional scope).

Every write, from these tools or the extraction agent, is first screened for secrets by server/memory/secret_scanner.py (check_and_block()): if a credential is detected the write is blocked and the content is never persisted. See Agent tools for the full catalogue of tools the assistant can call.

Memory viewer and the /api/memory router

People manage memory through a small REST router (server/memory/router.py, mounted at /api/memory), which backs the in-app Memory Viewer modal (src/components/settings/MemoryViewerModal.tsx). Open it from the "+ More" menu in the chat toolbar via Memory files. It carries the same guards as the tools: path-traversal checks, secret scanning on every write, a read-only MEMORY.md, and an index rebuild after every mutation.

EndpointAction
GET /api/memoryList both tiers: file manifests plus each tier's index (project only when a workspace is open).
GET /api/memory/fileRead one file's raw content, by scope and filename.
PUT /api/memory/fileSave raw file content; secret-scanned, then the tier index is rebuilt.
DELETE /api/memory/fileDelete a file and rebuild that tier's index.
POST /api/memory/promoteMove a file from the project tier up to global, so every project can use it.

Extraction and consolidation finish after the chat stream closes, so their results surface as live "Memory updated" toasts over the session's long-lived event channel rather than inline in the turn.

Consolidation (dream)

Over time a tier can drift: duplicates pile up, facts go stale, the index grows past its cap. A consolidation pass in server/memory/dream.py periodically distills each tier independently. The post-turn hook record_and_maybe_dream() bumps a per-tier counter after every turn and runs the pass when a tier is due: at least MIN_HOURS_SINCE_LAST (24 hours) and MIN_SESSIONS_SINCE_LAST (5 recorded turn-ends) since that tier's last run, with a PID-based lock (stale after one hour) so two consolidations never run at once.

When it runs, a single agent works through four phases: Orient (list files, read the index), Gather (find stale, contradicted, or overlapping entries), Consolidate (merge related files, update facts, delete superseded entries), and Prune (rewrite MEMORY.md to reflect the current state, keeping it under 200 lines). Think of it as the persistent-memory counterpart to how a single session's memory is periodically resummarized.

Key files

server/memory/session_memory.py
Per-session summaries: thresholds, the background updater, cloud vs local paths, and the <session-memory> wrapper.
server/memory/recall.py
Query-time selection (Haiku side-query or recent-files fallback) and the <memory-context> block builder.
server/memory/extract.py
Post-turn extraction: flag, throttle, mutual-exclusion, cursor, and the extractor agent run.
server/memory/memdir.py
Two-tier directory layout, slug and scope resolution, MEMORY.md loading and truncation, and the auto_memory guard.
server/memory/dream.py
Cross-session consolidation: trigger thresholds, locking, and the four-phase agent.
server/memory/prompts.py
System prompts for recall, extraction, consolidation, and session summarization.
server/memory/tools.py
The four memory tool schemas the model can call directly.
server/skills.py
WHISPER.md loading and its CRUD API.
