Security model

Boundaries versus defence-in-depth, at a glance. What actually contains a determined attacker, what merely catches the easy cases first, and which residual risks are accepted by design.

Whisper Studio runs as a single local process. Its threat model is not "protect a multi-tenant server" but "let an agentic assistant read and change files, run commands, and fetch web pages, without letting a prompt-injected page or a runaway tool call quietly reach your secrets or your LAN." This page is the reference summary. For the mechanisms behind each row, see Sandbox & security and Approval & permissions.

Headline points

Eleven properties hold by default, out of the box, with no configuration:

  • Localhost-only by default. The server binds 127.0.0.1. LAN exposure is an explicit opt-in via HOST=0.0.0.0 (see server/main.py). This closes the untrusted coffee-shop Wi-Fi surface an 0.0.0.0 default would create.
  • Every Markdown render is sanitized. All marked output passes through DOMPurify 3.4.6 before it reaches the React DOM, at every dangerouslySetInnerHTML site. A prompt-injected page fetched by web_fetch cannot fire arbitrary JS in the SPA origin.
  • Model-authored HTML runs sandboxed, never same-origin. HTML from a code-block preview or open, or a program artifact, renders only inside an iframe with sandbox="allow-scripts" and no allow-same-origin (an opaque origin), including the "Open in new tab" path. A prompt-injected artifact therefore cannot reach the local backend API even if the user opens it (see src/utils/openHtmlSandboxed.ts).
  • No client-asserted command approval. The old body.user_approved boolean on /api/workspace/shell is gone. Consent for risky actions routes through server/approval/router.py, which holds a server-side nonce the client cannot fabricate.
  • aws_cli does not use a shell. Commands are shlex.split into an argv list, so metacharacters like ;, &&, and $() become literal arguments to aws rather than shell operators.
  • Shell and Python execution is sandboxed. Every subprocess on the code-execution paths is wrapped in sandbox-exec(5) on macOS (and bwrap on Linux), including the hidden terminal_run PTY.
  • Plugins are opt-in. Dropping a .py into plugins/ does nothing until you explicitly enable it. This closes the drive-by plugin-via-download vector.
  • Project hooks are inert until trusted. Hook code shipped in a repo's .whisper/settings.json does not run until you explicitly trust that exact hook set - a SHA-256 canonical hash recorded in hooks_trust.json, read once so the validated bytes and the executed bytes are identical, mirroring trusted skills. A freshly cloned repo cannot run hook code on its own (see server/hooks/config_loader.py).
  • The ultracode workflow harness is isolated, but not a certified boundary. Model-authored workflow JavaScript runs in a Node-24 vm.createContext (server/workflows/harness/harness.mjs) whose isolation rests on layers, not a kernel sandbox: the Node globals (require, process, fs, child_process, timers) are simply absent from the context; determinism guards neuter Math.random, Date.now, and the timers; the node flag --disallow-code-generation-from-strings makes eval / new Function throw (so a script cannot reach the host through constructor.constructor); the env is scrubbed; and every new script is approved upfront via the phase-preview card. Real side effects flow through agent() into run_agent tools, which stay permission-gated and sandboxed. See The ultracode workflow runtime.
  • The Tavily key reads from the environment first. TAVILY_API_KEY overrides config.json:tavily_api_key, so rotation is one export away and the key need never be committed.
  • No telemetry. The only outbound call the backend makes on its own is to Amazon Bedrock for chat. Tool calls and skills reach the network only when a user message invokes them - including the /ci command, which shells out to the GitHub CLI (gh) only on explicit invocation, so the "network only when a user acts" property still holds.

The system deliberately treats the OS sandbox as the real boundary. The validator and the AWS read-only guard above it are defence-in-depth: they catch the easy cases without being relied upon to contain a determined attacker. When you reason about what is truly enforced, trust the sandbox last and most.

Boundary versus defence-in-depth

The distinction matters. A boundary is enforced by something the attacker cannot talk their way past (the kernel, a secret the server never reveals). Defence-in-depth is a layer that catches common cases but is bypassable on its own, so it sits in front of a boundary, never in place of one. Read this table top to bottom as least-to-most trusted for the execution paths.

What the sandbox actually enforces

The sandbox is allow-by-default with denials, not deny-by-default. The macOS profile is (allow default) plus deny rules for a curated _DENIED_PATHS list; the Linux bwrap setup read-only-binds the whole filesystem and masks the same paths (directories via tmpfs, files via /dev/null). What it concretely enforces:

server/sandbox.py - _DENIED_PATHS
Read and write denial of a curated secret-store list: SSH / GPG keys, cloud credentials (~/.aws is denied for generic commands but re-allowed per-call for the AWS tools), git credentials, browser cookies and keychains, shell history, password-manager stores, and assorted CLI token files.

Because the default is allow, the deny-list is broad but not exhaustive, and it is what the sandbox leans on. That shapes the accepted risks below.

Accepted residual risks

These are stated plainly rather than hidden. Each is a deliberate trade-off for a single-user local tool, and each has a fix that was judged not worth its cost today.

Network stays open, and non-listed secret reads are possible. Because the sandbox is allow-default, it does not restrict network access and does not block reads of secret files that are not on _DENIED_PATHS. Network is left open on purpose: sandboxed git, npm, pip, aws, and build commands need it. Flipping to deny-default with network restriction is possible but would break those tools, so it is out of scope for now.

HOST=0.0.0.0 exposes an unauthenticated API. Enabling it puts the API on the LAN with no auth. The Origin / Host guard (server/infrastructure/security.py) blocks browser-driven cross-site requests and cross-site WebSocket handshakes, but not a deliberate non-browser client. Only enable 0.0.0.0 on trusted networks. See Environment variables.

A TOCTOU residual on workspace path containment (S8). Every workspace file op validates the target with _ws_validate_path (server/workspace/paths.py), which os.path.realpaths the path and confirms it stays inside the workspace root. Because this is check-then-use, an intermediate directory swapped to a symlink mid-write could in theory escape. The atomic os.replace does not follow a symlink at the final path component, so the realistic case is covered; a full fix (O_NOFOLLOW / fd-based writes) risks rejecting legitimate symlinked paths inside a workspace, so it is left as-is.

What is intentionally not a security boundary

Two things look protective but are not, and it helps to know which is which.

visible_chat_history

The cron_event filter (server/infrastructure/sessions.py) is a token-cost and prompt-cleanliness control, not a security boundary. Cron output rows persist in chat_history so they render on session resume, but they are stripped before the Bedrock prompt is built, so a job firing every ten seconds cannot grow the prompt indefinitely. See Cron / scheduled tasks.

The AWS read-only guard

_AWS_READONLY_GUARD stops accidental non-read AWS calls from user Python, but because it lives in the same process the code can remove it. Treat it as a guard rail, not a fence. The thing that actually contains that code is sandbox-exec.

The one path that bypasses validation

The approval flow is the only legitimate path that skips _validate_command, and it is safe precisely because the server holds the nonce. When a tool needs consent it issues a nonce, pauses the turn, and streams an approval request over SSE. The client confirms by re-posting the action, payload, and that exact nonce; the server matches it, runs the validated command under the sandbox, and returns the result. A client cannot mint its own nonce, so it cannot self-approve. See Approval & permissions for the full nonce lifecycle and the per-mode behavior of the Mode dial.
