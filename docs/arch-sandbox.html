Sandbox & security

The OS sandbox is the boundary you should trust last and most. Everything in front of it is layered defence that catches the easy cases before code ever reaches the sandboxed subprocess.

Whisper Studio runs shell commands and executes code on your machine, so the interesting question is not "does it have security" but "which layer is a hard boundary and which just catches the obvious cases". This page walks the path a command takes, names the deny-list it runs against, and is honest about the risks that are accepted on purpose. The diagram below traces that path left to right - click a box to light up what it reaches.

The sandbox, and everything it wraps

Every shell and code-execution path funnels through one function, run_sandboxed in server/sandbox.py. That includes the workspace shell, run_python, the hidden terminal_run PTY session, and aws_cli. The wrapper picks the strongest OS-level sandbox available:

PlatformMechanismWhat it does
macOSsandbox-exec(5)Generates a temporary .sb profile and runs sandbox-exec -f profile /bin/sh -c "...".
Linuxbwrap (bubblewrap)Mounts the whole filesystem read-only (--ro-bind / /), rebinds the workspace and /tmp read-write, then masks each secret path.
Neither availablePlain subprocessA graceful no-sandbox fallback. There is no OS boundary here, but the command validators still run.

The long-lived interactive PTY that backs terminal_run is wrapped separately by build_pty_sandbox_wrap, which returns the same profile so an interactive shell runs under the identical deny-list as a one-shot command. (On non-macOS or when sandbox-exec is missing, it returns the command unchanged and validation becomes the only enforcement.)

The fallback path has no OS boundary. On a machine without sandbox-exec or bwrap, commands run in a plain subprocess. The validator still rejects the obvious dangerous patterns, but the secret-store deny-list is not enforced by the kernel. On stock macOS, sandbox-exec is always present, so this is a Linux-without-bubblewrap edge case.

What the sandbox actually enforces

The single most important fact about this sandbox is that it is allow-default plus deny rules, not deny-by-default. The macOS profile is literally (allow default) followed by a block of deny rules; the Linux setup binds everything read-only and then masks the same paths (directories with an empty tmpfs, files with /dev/null). What it enforces, therefore, is a single thing done well: read/write denial of a curated secret-store list.

That list is _DENIED_PATHS, built from the canonical definitions in server/security/sensitive_paths.py so the OS sandbox and the command validator can never drift on the shared core. It covers:

  • SSH and GPG keys - ~/.ssh, ~/.gnupg.
  • Cloud credentials - ~/.aws, ~/.config/gcloud, ~/.azure, ~/.docker, ~/.kube. Note that ~/.aws is denied for generic commands but re-allowed per-call for the AWS tools (see below).
  • Git and package-registry credentials - ~/.git-credentials, ~/.netrc, ~/.npmrc, ~/.pypirc.
  • Browser cookies and keychains - Chrome / Chromium / Firefox / Brave / Safari profiles, ~/Library/Cookies, and ~/Library/Keychains.
  • Shell history - ~/.bash_history, ~/.zsh_history, fish history (these frequently contain pasted tokens).
  • Password managers and other CLI token stores - ~/.password-store, 1Password / op, ~/.config/gh/hosts.yml, cargo, gem, terraform, doctl, netlify, and heroku credential files.
  • Absolute system secrets - /etc/shadow, /etc/sudoers, /etc/ssl/private.

One deliberate asymmetry: /etc/passwd is checked by the command validator but is not sandbox-denied, because NSS reads it on every user and group lookup - denying it in the kernel would break ordinary commands. A command that mentions it explicitly is still suspicious, so the validator keeps watching for it.

Why ~/.aws is denied, then re-allowed

run_sandboxed takes an allow_paths argument that subtracts specific entries from the deny-list for that one call. The AWS tools (aws_cli, aws_boto3, and boto3 code inside run_python) pass their cloud-credential paths so authentication works, while every other secret (~/.ssh, git creds, /etc/shadow, and the rest) stays blocked for that command. The re-allow is scoped to the AWS tools; a generic shell command never sees ~/.aws.

Accepted residual risk

Because the default is allow, two things follow directly, and both are accepted on purpose:

  • Network is not restricted. Sandboxed git, npm, pip, aws, and build commands all need the network to function, so the sandbox leaves it open.
  • Reads of non-listed secrets are not blocked. The deny-list is broad but not exhaustive - a secret file that is not on _DENIED_PATHS can still be read.

Flipping to a deny-default profile with network restriction is possible in principle, but it would break the tools above, so it is out of scope. This is the trade-off that makes the sandbox the boundary you trust last: it does not try to be a jail, it guarantees the high-value secret stores stay unreadable even if every layer in front of it is defeated.

Layers in front (defence-in-depth)

Before a command reaches the sandbox, several cheaper checks catch the easy cases. It is worth being precise about which of these are real boundaries and which only raise the bar.

The command validator

server/security/command_validator.py runs a regex deny-list against each subcommand. validate_command() first strips heredoc bodies (so a Markdown table full of | pipes is not mistaken for a pipeline), splits compound commands on ;, &&, and ||, then runs every validator over each piece. The checks include:

  • Dangerous patterns - rm -rf /, mkfs, dd of=/dev/..., redirects into /etc/, and /proc/*/environ access.
  • Command and process substitution - $(...), backticks, and <(...) that could hide a nested command (with a narrow exception for the $(cat <<...) pattern used by git commit -m and gh pr create --body).
  • Environment-variable injection - references or assignments to $LD_PRELOAD, $BASH_ENV, $IFS, $PROMPT_COMMAND, and friends.
  • Obfuscation - control characters, deceptive Unicode whitespace, hex/octal escapes, and ANSI-C quoting used to smuggle characters past the other checks.
  • Sensitive paths and file types - the shared deny-list plus extensions like .pem, .key, id_rsa, and filenames like .env.production.
  • In-place edits - sed -i is blocked with a nudge to use ws_write_file instead.
  • A subcommand cap - more than MAX_SUBCOMMANDS (50) chained segments is rejected to prevent CPU starvation from deeply nested pipelines.

The validator is regex-based and bypassable on its own, so it is explicitly defence-in-depth: it sits in front of the sandbox, never in place of it.

The approval nonce

Risky commands do not run on the model's say-so. They pause the turn and wait for a server-held nonce that the client must echo back to confirm. Because the server holds the nonce, a client cannot fabricate one, which makes approval a genuine boundary rather than a suggestion. See Approval & permissions for the full flow.

The AWS read-only guard

When boto3 runs inside run_python, server/executors/code.py prepends _AWS_READONLY_GUARD, which monkey-patches botocore.endpoint.Endpoint.make_request to reject any operation whose name is not read-shaped (the allow-list is list, get, describe, head, query, and similar prefixes). This catches a model that thought it was in read-only mode and would otherwise burn write rate-limits or trip an alarm. It is defence-in-depth only: user code in the same process can simply del _ep.Endpoint.make_request to undo it. The layer that actually contains the code is sandbox-exec, not this patch.

The render path and the network edges

  • DOMPurify (3.4.6) sanitizes every Markdown render before it reaches the React DOM, stripping <script>, inline event handlers, and javascript: URLs. This is a real boundary for the render path: without it, a prompt-injected web page fetched through web_fetch could fire arbitrary JS in the SPA origin.
  • The 127.0.0.1 default bind closes LAN reachability at the kernel level. Setting HOST=0.0.0.0 is an explicit opt-in that exposes the (unauthenticated) API to the network - only do it on a trusted LAN.
  • The Origin / Host guard (server/infrastructure/security.py) blocks browser-driven cross-site requests and cross-site WebSocket handshakes. It does not stop a deliberate non-browser client, which is why the 0.0.0.0 warning still stands.

Workspace path containment

File operations have their own boundary, separate from the shell sandbox. Every workspace file op validates its target with _ws_validate_path (server/workspace/paths.py): it calls os.path.realpath and confirms the result stays inside the connected workspace root, also blocking /dev, /proc, /sys, and UNC paths. Writes then go through an atomic _atomic_write_text (temp file plus os.replace).

Accepted TOCTOU residual (S8). This is a check-then-use pattern, so in theory a symlink swapped between the realpath check and the write could redirect it. The atomic os.replace does not follow a symlink at the final path component (it replaces the link itself), so the realistic case is covered; only an intermediate directory swapped to a symlink mid-write would escape, which is contrived on a single-user local tool. An O_NOFOLLOW / fd-based write would close it fully but risks rejecting legitimate symlinked paths inside a workspace, so it is left as-is on purpose.
