Live preview & browser automation

The assistant can spin up your dev server, open a headless browser against it, click through the running app, and stream the whole thing into a dock so you watch it happen. All of it is named, sandboxed, and consent-gated.

Live preview turns "let me build this UI" into "let me build this UI and check it renders." A handful of tools spawn a real dev server (server/preview/process.py), drive a real Playwright Chromium (server/preview/browser.py), and surface the result two ways: a screenshot the model can see inline, and a live screencast you can watch in the right dock. The manager and its sessions

server/preview/manager.py holds a PreviewManager that tracks multiple concurrent, named PreviewSession objects (its shape mirrors the MCP manager). Each session bundles an optional dev-server process, its own lazily created browser, and a bit of metadata (port, base URL, creation time). Names are unique: starting a session whose name is already running is an error, so you stop it or pick a new name first.

PreviewSession
One named unit: a DevServerProcess | None, a BrowserSession, plus port, url, and created_at. A session can be browser-only (no subprocess) if you point it at an already-running server.
start_session(name, command, cwd, port, url, env)
Under an async lock: refuse a duplicate name, spawn the dev server if a command was given, register the session, and return it. Spawning is separate from loading a page.
stop_session(name)
Pop the session, close its browser, then kill the process group. Returns whether anything was actually stopped.
list_sessions()
A status snapshot per session: id, url, port, process_alive, browser_started, and created_at.

Spawning the dev server

Unlike a one-shot terminal command, a dev server must stay alive across many tool calls. So process.py spawns it as a plain asyncio subprocess (no PTY) and drains its output continuously in the background. DevServerProcess.spawn() does this in order:

  1. Validate the command

    The raw joined argv is run through validate_command() from the security layer. Validation happens on the raw argv (not a shell-quoted form) because the process is exec'd directly with no shell, so quoting would only hide sensitive paths from the scanner.

  2. Sandbox-wrap it

    The command is wrapped by build_pty_sandbox_wrap() (the same wrapping long-lived terminal sessions use), producing the real argv plus a temporary sandbox profile path.

  3. Exec under a new session

    It is launched with start_new_session=True, not a preexec_fn. A preexec_fn callable would fork Python, which under uvloop can deadlock and wedge the whole event loop; setsid in C between fork and exec avoids that while still making the child a process-group leader so it can be killed cleanly.

  4. Drain the output

    Two reader tasks copy stdout and stderr into bounded ring buffers, so preview_logs can tail them and a chatty server can never grow memory without bound.

Bounds, not guesses. The stdout/stderr ring buffer is capped at 1 MB per stream (older bytes are dropped in half when it overflows). Spawn itself is bounded at 30 seconds as a safety net against a pathological sandbox-wrap or exec hang, so starting a server can never block a chat turn forever. A validation failure raises DevServerSpawnError, which is surfaced to the model as text, never a silent fallback.

Where does the command come from? Either a per-workspace .whisper/launch.json entry resolved by name (same shape as Claude Code's own launch.json: a list of configurations, each with a name, runtimeExecutable, runtimeArgs, and optional port), or an ad-hoc command passed directly on the tool call. If neither is present, preview_start returns a clear error explaining both options.

The Playwright browser

Each session owns one BrowserSession (server/preview/browser.py). The Chromium browser, its context, and its page are created lazily on first navigation, so a session that only needs its dev server never pays for a browser. ensure_started():

  • Launches headless Chromium, bounded at 60 seconds for the cold start (the first launch is slow). A timeout raises a clear error pointing at an incomplete Playwright install.
  • Uses an ephemeral context (new_context(), never launch_persistent_context()): no cookies or profile persist across sessions, and nothing is shared with your real browser.
  • Sets default timeouts on the context: 15 seconds for actions and 30 seconds for navigation, so a not-yet-ready or unreachable dev server fails fast instead of hanging the turn.
  • Installs a scheme guard at the context level (before any page exists, so it also covers popups): every request whose scheme is not http or https is aborted. This blocks file://, data:, and chrome:// escapes as a last-line technical backstop under the approval gate.
  • Captures diagnostics into bounded ring buffers (500 entries each): console messages, uncaught page errors (recorded as error-level entries so a thrown TypeError is not invisible), and finished/failed network requests with their status.

That is 14 tools in all: 7 approval-gated and 7 read-only. They are dispatched from an async branch in server/tool_router.py rather than the usual sync executor path, because Playwright's async API must stay bound to the single running event loop. See Tool dispatch & executors for how that split works. The read-only inspection tools are worth reaching for before a screenshot: preview_inspect reads computed styles exactly, so it is the honest way to confirm a color or a spacing value.

Screenshots the model can actually see

A screenshot is not returned as a base64 blob buried in JSON text. server/preview/screenshot.py captures the page as a PNG, downscales it so the long edge is at most 1280px, re-encodes it as a quality-70 JPEG, and returns a [WS_PREVIEW_IMAGE] sentinel carrying the payload and a caption. The tool executor detects that sentinel and rewrites it into a real Anthropic image content block, so the model sees actual pixels in the tool result.

The tool description nudges the model to prefer preview_console_logs, preview_network, and preview_snapshot for precise checks, and to use a screenshot for visual judgment (does this look right?) rather than for reading exact values.

Verifying the UI in the loop

These tools are not just for ad-hoc looking. They are the UI half of verification, and that is what makes "check the page, then keep fixing until it renders" feel automatic during full-stack work. There is no separate component that decides to screenshot — the behavior emerges from three pieces cooperating:

  1. The tools drive the page

    Everything above: spawn the dev server, navigate, read the console and the DOM, click and type, and capture a screenshot the model can see.

  2. The verify-change skill decides to use them

    skills/verify-change.md has a ui_facing branch. When a change touches rendered UI, the skill tells the model to start (or reuse) the dev server, load the changed view, check read_console_messages for errors, read_page for the expected structure, drive the changed interaction and re-read to confirm it took, and capture a screenshot as evidence — then emit exactly VERIFY PASS or VERIFY FAIL: <reason>.

  3. The goal gate makes it non-optional

    The completion gate's evaluator weights a verify_change result above the model's own prose claims (server/goals/evaluator.py). So under a goal, a turn is not judged done until a passing verify token appears — a console error or an interaction that didn't persist yields VERIFY FAIL, the gate returns not-achieved, and the model re-drives the preview and screenshots again until the UI genuinely holds up (capped at 8 blocks).

The net effect: give a goal like "make the settings page save correctly" and Whisper will keep spinning the live preview, clicking through the flow, and reading the console on its own until the page actually works — you don't have to keep saying "check it in the browser". The same verify_change step ends a generated CI-autofix workflow, so an automated fix proves its UI before it's handed back.

Two honest limits. The check only runs when the preview tools are available — gated by the preview_tools flag and a live capability probe (preview_capability_ok() = Playwright importable + Chromium installed); if either is missing, the tools hide themselves and verify-change falls back to the CI gate (tsc, tests, build) only. And in a parallel workflow, each agent that verifies UI spins its own dev server on a distinct port — fine for a few, but fanning out many concurrent browser checks is where you'd feel resource pressure.

Practical tip: phrase a full-stack goal so the UI check is the proof — "the form submits and the row appears" gives the evaluator a concrete thing to verify, where "improve the form" does not. The Goals & autopilot tutorial covers how to write goals that pin down "done".

Watching it live: the right dock

The screencast is the human-facing half. server/preview/screencast.py exposes a WebSocket at /ws/preview/{name}/screencast. When the SPA's live-preview pane connects, it ensures the browser page exists (blank if the model has not navigated yet), opens a CDP session, and calls Page.startScreencast to receive JPEG frames, which it forwards to the panel.

View-only by design

Frames flow one way. No clicks or keystrokes are forwarded back from the pane, so watching never interferes with the model driving the page. You see exactly what its browser does: a navigation, a click, dark mode, an overlay swallowing a click.

Fresh over complete

A tiny frame queue (cap 2) drops the oldest frame under backpressure to keep latency low, while every frame is acked immediately so Chromium never pauses the stream. Each viewer opens its own CDP session, so multiple watchers and the agent coexist.

The path lives under /ws/... deliberately: Vite's dev proxy upgrades only /ws as a WebSocket, while /api stays plain HTTP. A session that does not exist gets an accept-then-close with code 4404, so the browser sees a clean close rather than a handshake failure. See Frontend architecture for how the dock and its SSE/WebSocket runtimes are wired.

The status API and kill switch

server/preview/routes.py is a small status API for the Settings "Live preview" panel: a user-visible kill switch for whatever the model has spun up. It is intentionally not a full remote-control surface (the page actions above are agent tools, not REST endpoints).

Method + pathPurpose
GET /api/preview/sessionsList sessions and their status for the panel.
POST /api/preview/sessionsStart (or reuse) a session by name, resolving its command from .whisper/launch.json. Powers the panel's Restart button, so a stopped dev server can come back without the assistant.
DELETE /api/preview/sessions/{name}Stop a session; 404 if there is no such session.
WS /ws/preview/{name}/screencastThe view-only live screencast described above.

Lifecycle and cleanup

Stopping a session (via the tool, the Settings panel, or a full shutdown) cancels the reader tasks, kills the dev server's process group, closes the browser context and browser, stops Playwright, and unlinks the temporary sandbox profile. PreviewManager.stop_all() tears down every session at once. Because the context is ephemeral and the profile is temporary, nothing survives a session, which keeps the preview browser cleanly isolated from your real one.
