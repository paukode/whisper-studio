Sub-agents & the event bus

The assistant can hand work to focused helper agents that run their own tool loops in the background. This page covers the runtime that executes them, the agent types, how a team is spawned, the mailbox agents talk through, and the per-session bus that streams their progress back into your chat.

Sub-agents exist so a single turn can do several things at once, or delegate a narrow job to a cheaper, more constrained worker. A sub-agent is not a separate service. It is a plain run_agent call in server/agents/runtime.py that reuses the same tools, the same dispatch path, and the same session as the chat that spawned it. What makes it feel live is the event bus: a per-session queue the runtime publishes to, which the chat SSE stream drains and forwards as team_progress frames.

The agent runtime

run_agent() is the entry point. Unlike the chat pipeline, which streams tokens, the runtime makes non-streaming model calls in a loop until the agent returns final text or hits its turn limit. Each call goes through a provider adapter, get_adapter(model_key_for_id(model_id), model_id).invoke(...) from server/agents/providers/, so agents are provider-agnostic and run on both Anthropic-on-Bedrock and OpenAI GPT-on-Bedrock (the Responses API via bedrock-mantle); there is no direct invoke_model path anymore. Concurrency is governed by AGENT_CALL_CONCURRENCY. On calling run_agent, the runtime:

  1. Guards recursion

    Depth is capped. An agent that spawns an agent that spawns an agent stops at MAX_DEPTH = 5 and returns a failed result rather than recursing forever.

  2. Registers an id and a mailbox

    A short random agent_id is minted, a mailbox is created on the message bus, and the agent is recorded in the server/agents/registry.py registry so list_agents and the UI can see it. Detached agents (below) and /subagent-launched runs are additionally recorded in the unified task registry (server/tasks/registry.py, the agent_tasks table from migration 007, kinds shell|agent|workflow|ci), which emits task_event lifecycle events through server/tasks/events.py so they surface in the global background-tasks panel.

  3. Resolves the model

    Agents inherit the session-selected model through a model_id_override threaded from /api/chat. There is no hardcoded per-type model. With no override it falls back to your configured default chat model (then sonnet, then the first configured model).

  4. Builds a filtered tool pool

    It starts from the same assemble_tool_pool() the chat turn uses, adds the agent-only tools (receive_messages, complete_coordination, and spawn_agent while below the depth ceiling), deduplicates by name, then applies the agent type's tool filter (see below).

  5. Runs the loop

    Up to config.max_turns rounds. Each round polls the mailbox, invokes the model, executes any tool_use blocks, and emits progress events. It ends on any stop reason other than tool_use, or when the turn limit is reached.

Bedrock calls for agents run on a shared 4-worker ThreadPoolExecutor, off the request's event loop. That is why publishing progress must be thread-safe (see the event bus).

Detached (background) agents. The loop above runs inline and blocks the turn. Passing detach: true to spawn_agent (in server/agent_tools.py) does not: it returns a task_id immediately through start_detached_agent in server/tasks/agents.py and runs the agent in the background. A detached agent runs read-only by default — no human is present to clear the [WS_APPROVAL] write gate — unless isolation: 'worktree' gives it its own git worktree to write into. Its result is folded into a later turn by server/agents/completion_inject.py (inject_completions); the model polls with task_status / task_output and cancels with task_cancel. At most DETACHED_PER_SESSION_CAP = 8 run concurrently per session.

Inside one turn

Each iteration of the loop is small and deliberate. At the top, it drains any messages that have arrived in the agent's mailbox since the last turn and appends them as a user message, so an agent always reads its inbox before it thinks. It then invokes the model through the provider adapter (non-streaming), parses the returned content blocks into text, thinking, and tool_use, and acts on the stop reason:

  • If the stop reason is not tool_use (or there are no tool uses), the collected text is the agent's final output and the loop returns a completed result.
  • Otherwise each tool use is executed and the results are appended as a user message, then the loop runs again.

Tool execution mirrors chat. Most tools route through route_tool() in server/tool_router.py (the same dispatch the chat loop uses), so a sub-agent reads files, greps, runs commands, and calls the web tools exactly as the assistant does. A handful of agent-only tools are handled inline instead: spawn_agent, send_message, list_agents, receive_messages, and complete_coordination. Tool output is capped at 50K characters per result before it is fed back to the model.

Approvals auto-resolve inside an agent. There is no user present to click a card, so a [WS_APPROVAL] response is executed inline. A tool that needs a workspace but finds none returns an explanatory error rather than pausing. Keep this in mind when you choose which agent type does the work, and lean on read-only types when you do not want unattended writes. See Approval & permissions and Sandbox & security.

Agent types

Every type is a frozen AgentConfig in server/agents/config.py. A config sets the turn limit, the token budget per response, and how the shared tool pool is filtered (a whitelist, a blacklist, and a read-only flag that strips the write tools). The types the assistant can spawn:

TypeToolsMax turnsPurpose
generalFull pool30The default worker. Reads, writes, runs, and can itself spawn children.
exploreRead-only10Fast search-and-read to answer questions. Write tools are removed.
planRead-only15Analyzes the codebase and returns a structured, step-by-step implementation plan.
verifyFull pool10Runs tests and validates behavior, ending with a VERDICT: PASS / FAIL / PARTIAL line.
coordinatorOrchestration only50Spawns and messages other agents; has no file access at all.

The coordinator's whitelist is exactly spawn_agent, send_message, list_agents, and complete_coordination. It cannot touch files by design. A separate memory_extractor type also exists for the memory subsystem; it is not one you spawn directly. See Memory system.

Each user-facing type shares a common working method appended to its system prompt: restate the task, list steps, gather evidence per claim, then run a short self-check before the final answer. That self-check is emitted as a text event, so it shows up in the live log.

How read-only is enforced

filter_tools_for_agent() applies three rules in order: an allowed_tools whitelist short-circuits everything else; then a denied_tools blacklist; then, if read_only is set, every tool in the WRITE_TOOLS set is removed. That set covers the obvious writers (ws_write_file, ws_edit_file, ws_run_command, the git write commands, config_set, memory_write, cron_create, and more), so an explore or plan agent physically cannot mutate your project.

Teams

team_create (in server/agent_tools.py) spawns several agents in parallel in the same session, each with its own agent id, and aggregates the results. The flow:

  1. Announce the team

    A team_started event is published with a team_id, the team name, and each member's name, task, and type. The SPA uses this to draw the report card scaffold before any agent produces output.

  2. Run members concurrently

    Each member is a run_agent call, gathered with asyncio.gather. Every member inherits the session model and is tagged with the shared team_id, so its events land in the same card.

  3. Aggregate and finish

    When all members finish, a team_completed event is published and the tool returns a JSON summary (each member's name, type, status, turns, and a trailing slice of its output) back to the model.

A running team can be stopped mid-flight: POST /api/teams/{team_id}/stop sets a stop flag and cancels the gather task, and each member publishes a stopped event as its loop is cancelled. Agents can also spawn their own children with spawn_agent; the child inherits the parent's team_id and event channel, so "agents spawning agents" renders as badged child rows under the same card rather than vanishing.

Inter-agent messaging

Agents coordinate through a mailbox, not shared memory. The MessageBus in server/agents/messaging.py is a thread-safe dictionary of per-agent message lists. It offers a small, deliberate API:

send(from_id, to_id, content)
Enqueue one message into a specific agent's mailbox.
broadcast(from_id, content)
Enqueue to every other mailbox in the sender's own session; returns the recipient count. Mailboxes are session-scoped — the bus records each agent_id's owning session via create_mailbox(agent_id, session_id=...) — so a broadcast never crosses into another session.
receive(agent_id)
Drain and return all pending messages (clears the mailbox by default).
peek(agent_id)
Count pending messages without consuming them.

The send_message tool wraps send and broadcast: pass a to_agent_id for a direct message, or set broadcast=true to reach every other agent in the same session. Delivery is pull-based. A recipient sees a message only when its loop next runs and polls its mailbox at the top of the turn, at which point the messages are injected as a user turn. When an agent finishes, its mailbox is deleted in the runtime's finally block.

The mailbox (MessageBus) and the event bus (AgentEventBus) are separate systems. The mailbox is agent to agent, pulled each turn. The event bus is agent to user, one-way progress that streams live.

The event bus

The AgentEventBus in server/agents/event_bus.py is a per-session publish/subscribe queue. A subscriber calls subscribe(session_id) and gets a bounded asyncio.Queue (max around 512 events); the runtime calls publish(session_id, event) at every milestone. Multiple subscribers per session are supported, and each event fans out to all of their queues.

Two properties matter:

  • Thread-safe publishing. Agent Bedrock calls run on a thread pool and cron jobs publish from a daemon thread, so publish must work off the subscriber's event loop. Each subscriber records the loop it lives on; a cross-thread publish hops into that loop via loop.call_soon_threadsafe, while a same-loop publish does a direct put_nowait.
  • Bounded and lossy on purpose. If a subscriber falls behind and its queue is full, the event is dropped with a log warning rather than blocking the producer. Progress is best-effort; the aggregated result the tool returns is the source of truth.

Event phases

Runtime events are plain payloads (no type field) carrying an agent_id, the optional team_id and parent_agent_id, and a phase. The phases you will see over a run:

PhaseWhen
team_startedA team is announced with its member roster.
startedAn agent begins; carries the task preview, model, and max turns.
turn_startA new loop iteration starts.
textThe agent produced text this turn (including its self-check).
tool_call / tool_resultA tool was invoked, then returned (with a short and a fuller preview).
completed / turn_limitThe agent finished, or stopped at its turn limit.
stopped / failedThe agent was cancelled by the user, or raised an error.
team_completedEvery member of a team has finished.

Getting to the client

The event bus does not talk to the browser directly. While the chat turn runs the tool batch, the SSE consumer in server/chat/routes.py subscribes to the session channel and drains it in a tight loop, re-emitting each event as a team_progress SSE frame. Draining happens during the batch, so a long-running team_create streams activity in real time instead of arriving in one burst when the tool finally returns.

Six event types are filtered out here: cron_event, memory_event, task_event, cron_progress, ci_progress, and ci_result are skipped, because a separate long-lived stream, /api/sessions/{id}/events, delivers those (task_event is the new unified background-task lifecycle role). Skipping them prevents double-rendering. On the client, the SPA groups team_progress frames by team_id and agent_id into the live report card, badging spawned children under their parent. See Chat & tool orchestration for the wider SSE frame set and Frontend architecture for how the card is rendered.
