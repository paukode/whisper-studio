Sub-agents & teams

Split a big task across several focused agents that run at the same time. You keep chatting; a live report card shows each agent spawn, its tool calls, and its final answer as they happen.

Most work is fine as a single conversation. But some tasks are naturally parallel: "search these five directories," "draft it, then have another agent verify it," "audit each service." For those, Whisper Studio can run sub-agents - independent worker agents, each with its own tool loop, sharing your session and its selected model (Claude, or OpenAI GPT-on-Bedrock when that is the session model). This tutorial covers the two ways to start them, the agent types you can pick, and how to read the progress card that streams back.

Two ways to start agents

There are two entry points, depending on whether you want one helper or several.

/subagent - one focused helper

Type /subagent <task> in the chat input to launch a single one-shot agent. It runs its own tool loop in the background and reports back into the same conversation. Good for a self-contained side task you do not want cluttering the main thread.

Let Claude spawn them - multi-agent work

Just describe the parallel work in plain language. Claude decides when to call spawn_agent (one independent agent) or team_create (a whole team at once) rather than doing everything sequentially. You do not call these tools yourself; Claude does, and asks approval like any other tool.

Detached (background) agents. Both entry points above run inline and report within the same turn. Claude can instead call spawn_agent with detach: true: it returns a task id right away, runs read-only by default (or writes safely with isolation: 'worktree'), and shows up in the background-tasks panel while it works. Its result is folded into a later turn, and you or Claude can check on it with task_status / task_output or stop it with task_cancel.

Agents share your current session and workspace. If no workspace folder is open, file tools (ws_read_file, ws_grep, ws_glob, and friends) are not available to the agents, and they will say so. Open a folder first when the task needs to touch files. See the workspace IDE.

Agent types

Every agent has a type that shapes its tools, its turn budget, and its instructions. Claude picks a type when it spawns an agent (or you can name one in a team brief). The read-only types can search and read but never write or run commands, which makes them fast and safe to parallelize.

TypeToolsTurn limitBest for
generalFull tool set30Do-anything worker: read, edit, run, commit (subject to the session's permission mode).
exploreRead-only10Fast search-and-read to answer a question. Concise findings, no side effects.
planRead-only15Study the codebase and return a step-by-step implementation plan with files to change.
verifyFull tool set10Check work by reading and running tests. Ends with VERDICT: PASS, FAIL, or PARTIAL.
coordinatorOrchestration only50Manages other agents. No file access at all - it can only spawn, message, list, and finish.

The read-only guarantee is enforced, not just requested: the runtime strips every write tool (file writes, ws_run_command, git commits, config_set, memory writes, and so on) from an explore or plan agent's tool pool. A coordinator gets an even tighter whitelist of just spawn_agent, send_message, list_agents, and complete_coordination, so it cannot touch your files even by accident.

A classic pattern is explore, then verify: one agent gathers the facts, another checks a change and returns a clear PASS or FAIL. Because each runs its own loop, the verify agent is a genuine second opinion, not the same context re-reading itself.

Running a team

When Claude uses team_create, it names the team and lists the members, each with a name, a task, and an agent type. All members then run in parallel, sharing your session-selected model. Here is what a run looks like end to end.

  1. Describe the parallel work

    For example: "Look at the auth, billing, and notifications modules in parallel and tell me which ones lack tests." Claude proposes a team with one agent per module.

  2. Approve the tool call

    team_create is a normal tool call, so it passes through your current permission mode. Approve it and the team starts.

  3. Watch the card fill in

    A team report card appears in the chat immediately (the "scaffold"), then each agent's row updates live as it spawns, calls tools, and finishes. The card header shows a running tally like 2/3 done - 1 running - 14s.

  4. Read the results

    When every member finishes, the card flips to done and Claude folds the combined summary back into its own answer. Expand any row to see that agent's full event log and final output.

Reading the team report card

The report card (rendered by src/components/chat/TeamReportCard.tsx) is the window into a run. It is one collapsible card per team, with a collapsible row per agent.

  • Header - team name, agent count, and a live status such as 3/3 done or 1/3 done - 2 running, plus elapsed time. A green check means all completed; a red mark means at least one failed.
  • Agent rows - each shows the agent name, its role badge (team or orchestrator), its type badge (explore, verify, and so on), and a live status like running - turn 3 or done - 5 turns - 2 tools. A running or failed row opens automatically so you see it without clicking.
  • Expanded row - the agent's task, a real-time event log (each tool call and its result, click a line to expand the full input or output), and the agent's final output once it is done.
  • Child agents - if an agent spawns its own sub-agent, that child appears as its own row badged ↵ child, so a nested tree stays visible in one card.

The card also surfaces each agent's own self-check. Every user-facing agent type is told to restate the task, work step by step, and run a brief self-assessment before its final answer, which shows up as a text event in the log. So you can see the agent reason about whether it actually finished, not just its conclusion.

How progress reaches you

Under the hood, each agent publishes typed events (started, turn_start, tool_call, tool_result, completed, and failure or turn-limit phases) to a per-session event bus. The chat endpoint drains that bus and relays the events to the browser as team_progress SSE frames, which the SPA folds onto the message to build the card you see. Because it is a live stream, the card is accurate the whole way through, not just at the end. For the runtime and bus internals, see Sub-agents & the event bus.

Agents that talk to each other

Agents in a session are not isolated. They can coordinate through two tools:

send_message
Message another agent by its ID, or set broadcast: true to reach every active agent at once. Messages are delivered between an agent's turns.
list_agents
List the active agents in the session with their type, status, and current task, so an agent (or you, by asking) can see who is working on what.

This is the pattern a coordinator agent uses: it spawns workers, messages them, and calls complete_coordination when the work is done, without ever touching a file itself. Agents can spawn agents up to a bounded nesting depth, so a coordinator can build a small tree without runaway recursion.

Stopping a run

While any agent is still working, a Stop button sits in the team card header. Click it to cancel the whole team mid-flight: the server cancels the parallel run (POST /api/teams/{team_id}/stop), each running agent flips to a stopped state through the same event stream, and the card reflects it. Pressing Esc and using the main chat Stop control also tears down in-flight agents along with the streaming turn. See Permissions & approvals for the broader kill-switch behavior.

Every extra agent is a full model run with its own token bill. Fan-out is powerful, but a five-agent team is roughly five conversations at once. Keep teams small, prefer read-only explore agents where you can, and watch the usage counter. See Cost tracking & budgets.

When to reach for agents

Good fits

Independent parallel work (per-file, per-module, per-source), a draft-then-verify loop, or a broad search you want split across directories.

Skip it when

The work is a single linear thread, or each step depends on the last. A normal chat turn with the tool loop is simpler, cheaper, and easier to follow.
