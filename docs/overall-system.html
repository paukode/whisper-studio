Overall system

Whisper Studio runs as a single process on your machine: a FastAPI server that serves both the REST/SSE/WebSocket API and the React SPA from the same origin. There is no separate frontend host and no remote backend; except for Amazon Bedrock and the optional Tavily and MCP integrations, nothing in the request path is remote.

This page is the whole architecture in one place, at two levels of zoom: the system map (every major component and the boundaries between them) and the chat-turn sequence (what happens when you send a message). Both diagrams are interactive - click any box to trace where its data flows, and use Expand for full-screen zoom and pan.

1. System overview

Key invariants

One origin, one process

The SPA is served from /static/dist/ by the same FastAPI app that exposes the API. There is no CORS layer because there is nothing to cross.

Localhost by default

The server binds 127.0.0.1, so an untrusted network can't reach it. LAN exposure takes an explicit HOST=0.0.0.0 opt-in.

Transcription is local

The Whisper model loads on first record and runs in-process; audio bytes never leave the machine. Only a transcript can become a Bedrock prompt - and only when you type something.

Bedrock is the only cloud LLM

No telemetry, no proxy, no gateway. The signed request goes straight from the local process to bedrock-runtime.<region>.amazonaws.com and back.

MCP servers are child processes

Spawned by server/mcp.py and piped over stdio. They never see HTTP.

2. Chat-turn sequence

When you submit a message, the server builds the prompt, streams the model, and drives an agentic tool loop - pausing for approval on any risky action - before persisting the result.

Notes on the sequence

  • visible_chat_history filter. Before building the Bedrock messages array, non-prompt rows (cron_event and future UI-only roles) are stripped. Cron output cards persist on disk and render in the chat, but they never inflate the prompt or the token bill.
  • Approval flow. The only legitimate path that bypasses _validate_command: consent is a server-held nonce the client cannot fabricate. See Security boundaries.
  • Session persistence. A full UPSERT, but under a per-session asyncio lock with merge logic so cron-event rows appended by the background daemon between renders aren't clobbered by the frontend's save.

3. Security boundaries

Some layers are hard boundaries (kernel- or server-enforced); others are defence-in-depth that catch easy cases before they reach a boundary.

What sandbox-exec actually enforces

The sandbox is allowlist-by-exception, not deny-by-default: the macOS profile is (allow default) plus deny rules for a curated secret-store list (_DENIED_PATHS in server/sandbox.py), and the Linux bwrap setup --ro-binds the filesystem read-only and masks the same paths. It denies read/write of SSH/GPG keys, cloud credentials (~/.aws is denied for generic commands but re-allowed per-call for the AWS tools), git credentials, browser cookies/keychains, shell history, and password-manager stores.

Accepted residual. Because the default is allow, the sandbox does not restrict network access (sandboxed git/npm/pip/aws/build commands need it) and does not block reads of secret files that aren't on the deny-list. This is the boundary you should trust last and most - everything above it is layered defence.

Workspace path containment

Every workspace file op validates the target with _ws_validate_path (server/workspace/paths.py): it os.path.realpaths the path and confirms it stays inside the connected workspace root (also blocking /dev, /proc, /sys, and UNC paths). Writes go through a temp file plus os.replace. The check-then-use pattern leaves a contrived intermediate-symlink TOCTOU residual, accepted for a single-user local tool.

What's deliberately not on the map

  • The companion subsystem - an optional sidecar that can be toggled off entirely.
  • Background tasks - used to run long commands without holding an HTTP connection.
  • The skill autocomplete and @-mention paths in the chat input - pure frontend, no backend round-trip.

These are real parts of the system but aren't on the critical path for "how does a chat turn complete and stay safe?", which is what this page is for. Every box above is also a link into a focused page - the whole Architecture section is a zoom into one part of this map.
