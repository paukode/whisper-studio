The goal loop & completion gate

Give the assistant a goal and it keeps working until the goal is met. An end-of-turn completion gate checks whether the work is actually done — with hooks and a cheap evaluator — and, if not, feeds back what's missing and lets the turn continue, capped so it can never loop forever.

The loop lives in server/goals/: store.py (the goal on the session), tail.py (the transcript the evaluator reads), evaluator.py (the verdict), gate.py (the gate that ties it together), and cron_verify.py (the same idea for cron). Click a box in the diagram to trace a turn through the gate.

Why a gate

Left alone, a model tends to stop as soon as it has produced something — often before the task actually holds up. The completion gate turns "the model emitted end_turn" from an unconditional stop into a decision: is the stated goal met, and did any Stop hook object? If the answer is no, the turn is not over — the gate re-injects what's missing and the model keeps going. It is opt-in per session (you set a goal), fails open on any error, and is capped so a stubborn evaluator can't trap you.

run_completion_gate: two phases

Every streaming loop calls run_completion_gate(ctx) at end of turn and acts on the returned GateDecision. It has two phases with different flags:

  1. Phase 1 — Stop hooks (always)

    check_stop_hooks() from the hooks engine runs regardless of the goal flag. A block re-injects the hook's reason and the turn continues. The goal_loop flag gates only Phase 2, so turning goals off never disables your Stop hooks.

  2. Phase 2 — the goal evaluator (flagged)

    When goal_loop is on and the session has an active goal, a cheap Haiku call judges the turn: {achieved | not_achieved | blocked, feedback, confidence}. achieved ends the turn; not_achieved re-injects the feedback and loops; a confident blocked (confidence ≥ 0.7) ends the turn rather than looping on something the model genuinely can't do.

Both phases fail open: the evaluator is wrapped so a transport error or malformed reply lets the turn end normally rather than trapping it. And the whole thing is bounded by a consecutive-block cap (goal_max_consecutive_blocks, default 8) — hit it and the gate gives up, records not_achieved, and lets the turn end.

The evaluator and the transcript

The evaluator (server/goals/evaluator.py) is a single non-streaming Haiku call — cheap enough to run at the end of every turn. It reads the goal plus a rendered tail of the turn from server/goals/tail.py, a provider-neutral renderer that flattens recent activity (Anthropic content blocks, OpenAI Responses items, and top-level function_call / function_call_output items) into text, head-and-tail sliced to a 12 KB cap so a huge turn can't blow the prompt. The verdict JSON is extracted with a balanced-brace scan and one retry.

The evaluator weighs a deterministic signal above prose: the verify-change skill (skills/verify-change.md) runs the repo's real verification gate and ends with exactly VERIFY PASS or VERIFY FAIL: <reason>. That is the same line a generated CI autofix workflow finishes on — proof beats a confident-sounding summary.

Full-stack goals: verifying the UI

For a change that touches rendered UI, that verification is not just tsc-and-tests — the verify-change skill's ui_facing branch drives the live preview: it spins up your dev server, loads the changed view, checks the browser console for errors, drives the changed interaction, and captures a screenshot before emitting its verdict. Because the gate weights that verdict above prose, a goal on full-stack work will not be judged done until the page actually renders and behaves — so Whisper keeps re-driving the preview and screenshotting on its own until the UI holds up, without you asking it to "check the browser". See Verifying the UI in the loop for the full mechanism (and its limits: it needs the preview tools available, and parallel UI checks each spin their own dev server).

Where the goal lives

The goal is stored on the session itself. server/goals/store.py adds two additive columns to the sessions row — goal (the text) and goal_state (JSON, including the consecutive-block counter) — via the migration that also carried the workflow tables (see Persistence). The block counter resets on each new user turn, so the cap counts blocks within one continuous stretch of work, not across a whole conversation.

Verified cron

The same idea guards scheduled work. server/goals/cron_verify.py lets a cron run verify it actually achieved its prompt before reporting ok: it folds any notify_user bodies in as a synthetic assistant message, runs the evaluator, and continues the run (up to two continuations) if the goal is unmet. A run that never meets its goal is marked failed and its report is prefixed [UNVERIFIED]. Gated by the cron_verify flag.

Setting a goal

/goal <text> sets the session goal; /goal clear ends it. The active goal shows in a GoalBanner above the composer (hydrated from the session on load), and the gate's decisions surface as goal_eval and goal_cap_reached frames. The REST surface is small: GET / POST / DELETE /api/sessions/{session_id}/goal.
