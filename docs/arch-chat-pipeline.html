Chat & tool orchestration

This is the component every other subsystem plugs into. A single streaming endpoint builds the prompt, calls the model, and drives an agentic tool loop - emitting Server-Sent Events the whole way.

The entire turn is one coherent state machine in server/chat/routes.py. Click a box below to trace a path - for example, click Streaming loop to see everything a single round can reach.

The request

A turn is a POST /api/chat whose body carries the question, prior history, the selected model and effort_level, the session_id, and any attachment_ids. The response is an SSE stream, not a single JSON body - tokens and tool activity arrive as they happen. An approval continuation re-enters the same endpoint with an approved_tool_result instead of a fresh question.

Building the turn

Before the model is ever called, the endpoint assembles everything the model will read:

  • Latch the session config. On the first turn, the model, effort level, and related settings are snapshotted so mid-session Settings changes never disrupt an in-flight conversation.
  • Assemble the system prompt. Base rules, the workspace WHISPER.md, selected memories, and per-session memory are composed with cache-aligned static/dynamic splits (see prompt caching). A third, moving cache checkpoint is placed on the last message each round (annotate_messages_cache, server/chat/caching.py), so the conversation history caches round-over-round instead of being re-billed as uncached input.
  • Resolve @file mentions inline into the user message.
  • Expand attachments - documents are capped (around 150K characters) and images become vision blocks. See attachment extraction.
  • Index grounding. If the user selected workspace indexes, relevant passages are retrieved and injected, and a "grounding" chip is surfaced over SSE. See Index / GraphRAG.
  • Condense an oversized transcript. A transcript above a size threshold is condensed to per-chunk extracts before it enters the prompt - both the transcript block and the transcript handed to the summary skills, so the prompt never overflows. It self-gates: below the threshold the transcript passes through unchanged. See Transcript summarization.

The streaming loop

The Anthropic path enters a loop of up to 50 rounds. Each round:

  1. Check the budget

    If the session or daily cost cap is exceeded, emit an error and finish.

  2. Invoke the model, streaming

    invoke_model_with_response_stream is wrapped in server/infrastructure/bedrock_retry.py - exponential backoff on throttling, and reactive compaction if the prompt is too long.

  3. Parse content blocks

    Chunks are parsed into text, thinking, and tool_use blocks and streamed onward.

  4. Act on the stop reason

    tool_use runs the tools, appends the results, and loops again. Any other stop reason (such as end_turn) no longer ends the turn on its own: the non-tool-use branch first runs the completion gate (run_completion_gate, server/goals/gate.py). Phase 1 always runs the WS-I Stop hooks; phase 2 runs a Haiku goal evaluator when the goal_loop flag is on and the session has an active goal. A Stop-hook block or an unmet goal re-injects [completion gate] feedback as a user message and loops again, bounded by goal_max_consecutive_blocks (default 8). See The goal loop & completion gate.

Executing tools

When the model asks for tools, execute_tool_batch() (server/tool_executor.py) runs them safely:

  • Partition for parallelism. Consecutive read-safe tools run concurrently; each write tool runs on its own. So [read, read, write, read] becomes [[read, read], [write], [read]].
  • Permission checks. Every tool is evaluated against the current permission mode and per-tool rules - allow, ask, or deny.
  • Approval gate. A tool that needs consent issues a nonce, pauses the turn (state is stashed in _paused_sessions), and waits for the client to confirm.
  • Hooks. Hooks are a blocking subsystem (server/hooks/): run_hooks(event, payload) fires PreToolUse, which can deny or rewrite a call, and PostToolUse, which can inject additionalContext. A parallel check_stop_hooks() gate (server/hooks/engine.py) runs at end-of-turn across chat, subagents, and cron — see Blocking hooks.

Dispatch itself is a pure mapping in server/tool_router.py; the implementations live in server/executors/. See Tool dispatch & executors.

Streaming back to the client

Throughout the turn, the endpoint yields typed SSE frames the SPA folds into the UI:

user_questionThe turn paused on an approval or an ask_user_question.
[DONE]The stream is complete.

Compaction & persistence

If a prompt grows too long mid-stream, reactive compaction drops the oldest message pairs (or summarizes them via Haiku) and retries - so a long agentic session degrades gracefully instead of erroring. When the turn ends, the assistant message is appended and the session is UPSERTed into SQLite under a per-session lock, so background cron_event rows are never clobbered. Those cron rows render in the chat but are filtered by visible_chat_history before the next prompt, so they never inflate the token bill. See Persistence and Cron.

Key files

server/chat/routes.py
The SSE streaming endpoint and turn state machine.
server/chat/tool_pool.py
Assembles the per-turn tool catalogue (mode filter + dedup); with the progressive_tools flag it splits the catalogue into a core prefix plus a deferred, session-activated index.
server/chat/caching.py
System-prompt resolution with cache-aligned splits, plus the moving annotate_messages_cache checkpoint on conversation history.
server/hooks/
The blocking hook engine: run_hooks (Pre/PostToolUse) and the check_stop_hooks end-of-turn gate.
server/goals/gate.py
run_completion_gate: Stop hooks plus the optional goal evaluator that can refuse to end the turn.
server/chat/compaction.py
History compaction when the prompt is too long.
server/tool_router.py
Pure tool dispatch.
server/tool_executor.py
Batching, permissions, approvals, hooks, result shaping.
