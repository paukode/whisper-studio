Working with documents

Drop a PDF, Word doc, deck, spreadsheet, or image into chat and ask about it in plain English. Everything is parsed on your laptop, and Claude remembers the document for every follow-up in the same conversation.

You have a 40-page vendor contract and three minutes before a call. You need the payment terms. This tutorial walks that exact flow: attach the file, ask, and keep asking, without re-uploading anything.

The goal

Analyze a document fast. Instead of skimming 40 pages yourself, you hand the file to Claude and ask targeted questions. The whole loop is: attach, ask, read, ask again. No conversion step, no separate viewer, no copy-pasting text into the chat box.

Attach the file

There are two ways to get a file onto the composer, and both end up in the same place:

  1. Drag it into the chat

    Drop a PDF, Word document, PowerPoint deck, Excel or CSV file, or an image straight onto the chat composer. This is the fastest path when the file is already on your desktop.

  2. Or use the paperclip

    Click the paperclip (Attach) button in the composer and pick a file. If the file already lives in a connected workspace, you can also pull it in with the /file: picker.

  3. Watch for the chip

    A chip with the file name appears in the composer. While it uploads, the chip shows an "uploading" badge; once /api/upload resolves, the badge clears and the chip is ready to send. To drop a file, click the x on its chip, which also cancels an in-flight upload.

Uploads start the moment you attach, not when you submit. By the time you finish typing your question, the file is usually already parsed and waiting, so pressing Enter rarely stalls.

Ask, then follow up

With the chip in place, type your question and press Enter. A worked example:

What are the payment terms, in plain English?

Claude reads the document text and streams back an answer, for example "Net 30 from the invoice date, with a 1.5% monthly late fee." Now the useful part. Ask a follow-up without touching the file again:

And the termination clause?

You do not re-upload. The document is already in the conversation, so Claude answers about termination from the same source. Keep going as long as you like, one question per turn.

What happens on your machine

The important detail: the file is parsed locally. When you attach a document, Whisper Studio converts it to Markdown text on your laptop. The raw bytes do not go to the cloud when you attach; they only travel (as extracted text) when you actually send the chat message. Even then, Claude sees the document text, not the original file.

Under the hood, server/extract/__init__.py dispatches by file type: a general converter (MarkItDown) handles most formats, and specialized paths handle PDFs, spreadsheets, and images. Claude reads that text once and keeps it in the conversation history, which is why every follow-up in the same session stays grounded in the document.

Because parsing is local, a document you never send stays entirely on your laptop. The extraction pipeline runs whether or not you are online; the network is only used for the chat call itself.

Supported file types

Whisper Studio handles the everyday office formats plus images:

TypeWhat you get
PDF (.pdf)Text extraction, with an OCR fallback for scanned or image-only PDFs so scans still become readable text.
Word (.docx)Document body converted to Markdown, headings preserved.
PowerPoint (.pptx)Slide text pulled out deck-wide.
Excel / CSV (.xlsx, .xls, .csv)Small sheets in full; large ones sampled (see below).
Images (PNG, JPG, and more)Sent as a vision block, and OCR'd to text so text-only models can still read them.

Large spreadsheets are sampled

A spreadsheet with thousands of rows would blow past the context window if dumped whole, so a big sheet is turned into a compact schema plus sample instead: each sheet's dimensions, the header row, and the first 20 rows. Claude sees the shape of the data and can ask for specific rows or columns when it needs more, rather than reading everything at once.

Size limits

Documents are accepted up to about 50 MB. A file over that limit is rejected with a clear message (surfaced from the backend, not a blanket "upload failed"), so you know to trim or split it. Very large documents that fit under the size limit are still capped at roughly 150,000 characters of injected text per attachment, which keeps a single big file from swamping the conversation.

If an upload fails, read the toast. It surfaces the real reason, such as an unsupported file type or a file over the 50 MB limit. If it says the server is unreachable, the app's backend is not running.

Tips that make this smoother

Attach several at once

Drop or pick multiple files in one go. Each gets its own chip, and Claude can compare or cross-reference them in a single question.

Be specific

"Summarize the payment and termination sections" beats "summarize this." Targeted questions get targeted answers from a long document.

Pull from your workspace

If a file already lives in a connected repo, the /file: picker attaches it without leaving the chat. See The workspace IDE.

One document, many turns

You only pay the upload once. Keep asking follow-ups in the same session and the document stays in context the whole time.
