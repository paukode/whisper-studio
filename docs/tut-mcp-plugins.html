MCP & plugins

Add tools from external MCP servers, and extend the backend with opt-in Python plugins. Both paths hand the assistant new capabilities without touching the core code.

Whisper Studio ships with a large built-in tool pool, but you can grow that set two ways. An MCP server connects an outside system and exposes its tools over the Model Context Protocol. A plugin is a local Python file that registers new executors, API routes, or pre-tool hooks inside the backend. Both are opt-in: configuring a source does nothing until you explicitly enable it.

Which one do you want?

Pick the mechanism by where the capability lives and who owns it.

MCP server

A tool already exists as an MCP server (a database bridge, an issue tracker, a filesystem gateway). You want to connect it without writing Python. It runs as its own process, isolated from the backend, and you flip it on and off per conversation.

Plugin

You want custom behaviour that lives inside the backend: a bespoke tool executor, a new REST route, or a PreToolUse hook that inspects or blocks other tools. You are comfortable writing a small Python file and restarting the server.

Connecting an MCP server

MCP (the Model Context Protocol) is a standard way for a host application to talk to external tool providers. Whisper Studio speaks it over stdio: each server you configure is launched as a child process, and the backend exchanges JSON-RPC messages with it over standard input and output. Nothing goes over the network unless the server itself reaches out.

  1. Open Settings and add a server

    Click the gear icon, go to the MCP tab, and add a server entry. Each entry is a command the backend runs to start the server (an executable plus its arguments), the same shape you would put in an MCP client config elsewhere.

  2. Let the backend launch it

    On enable, the backend spawns the command as a stdio child process and performs the MCP handshake to discover the tools and resources the server offers. If the command fails to start, the server shows as unavailable rather than silently doing nothing.

  3. Toggle it per turn from the chat toolbar

    A configured MCP server does not force itself into every conversation. Enable or disable each server for the current turn from the chat toolbar, so you only pay the token cost of its tool schemas when you actually want it.

  4. Use its tools in chat

    Once enabled, the server's tools appear alongside the built-ins and the model can call them during the tool loop, gated by the same permission mode as everything else.

MCP tools count against your per-turn token budget just like built-in tools: every enabled tool's schema is sent to the model before your message. Enable only the servers a conversation needs, and disable them again when you are done. See Agent tools for how tool schemas are billed.

Reading MCP resources

Beyond tools, MCP servers can expose resources (documents, records, or other readable content). Two built-in tools let the assistant work with them:

ToolWhat it does
list_mcp_resourcesLists the resources exposed by the connected MCP servers.
read_mcp_resourceReads a specific resource so its content enters the conversation.

You do not call these yourself. Ask a question that needs the data ("what tables does the connected database expose?") and the model uses list_mcp_resources and then read_mcp_resource to pull in what it needs.

Writing a plugin

Plugins live in plugins/ at the repository root. A plugin is a single Python file that declares a version and a description, then exposes a register() function the backend calls at startup.

__version__ = '1.0.0'
__description__ = 'My custom plugin'

def register(app, executor_registry):
    # Register a new executor tool
    def my_tool(tool_input, transcript, attachments):
        return 'Hello from plugin!'
    executor_registry['my_tool'] = my_tool

register() receives two handles:

app
The FastAPI application, so a plugin can mount custom REST routes.
executor_registry
The tool-executor mapping. Add a callable under a tool name and the assistant can call that tool.

An executor callable takes (tool_input, transcript, attachments) and returns the result the model sees. Because plugins run in-process, they can also import backend modules, for example to register a hook (covered below).

Enabling a plugin

This is the important part, and it is deliberately not automatic.

  1. Drop the file in

    Place your plugins/my_plugin.py in the folder. On its own, this does nothing. The file is not imported and its register() is never called.

  2. Enable it in Settings

    Open Settings, go to the Plugins tab, and turn your plugin on. Enablement is per plugin, so you choose exactly which files load.

  3. Restart the server

    Backend modules are imported once at startup, so a newly enabled plugin loads on the next run. Stop the app (Ctrl+C) and start it again with bash setup.sh. The same applies to edits: changing a loaded plugin needs a restart to take effect.

Why opt-in? A plugin runs arbitrary Python inside the backend with the same privileges as the app. If dropping a .py into plugins/ auto-executed it, a drive-by file (pulled in by a dependency, a clone, or a careless copy) would run code the moment the server started. Requiring an explicit enable plus a restart means nothing runs that you did not choose to run. See Sandbox & security and the security model.

Migrating from the old opt-out behaviour is lossless: anything already loaded stays loaded, and only new files stop auto-executing.

Pre-tool hooks: inspecting and blocking calls

Plugins are not limited to adding tools. They can also intercept every tool call before it runs. A plugin registers a PreToolUse hook by importing the hook registry and calling register_pre_tool_hook() from inside register():

def register(app, executor_registry):
    from server.infrastructure.plugin_hooks import register_pre_tool_hook
    register_pre_tool_hook(check_security)

A hook is an async function fn(tool_name, tool_input). Return None to allow the call. Return a dict such as {"reason": "...", "findings": [...]} to block it: the tool never runs, and the reason is surfaced back to the model. Hooks run in order and the first block wins.

The bundled plugins/security_checks.py is a working example. It hooks ws_write_file, ws_edit_file, and ws_create_file, scans the content being written for risky patterns (an eval( call, an os.system( shell call, a hardcoded-looking secret), and blocks the write with a summary of what it found. It is disabled by default like any other plugin, so enabling it is a one-line demonstration of the whole flow.

Read plugins/security_checks.py before writing your own. It shows the full shape: module __version__ and __description__, a register() entry point, and an async hook that returns a block dict. Copy it as a starting point.

Troubleshooting

  • MCP tools are not showing up. Confirm the server is enabled for the current turn in the chat toolbar, not just configured. A configured-but-disabled server contributes nothing.
  • The MCP server shows unavailable. The launch command failed. Check that the executable and arguments are correct and runnable from the same shell that started Whisper Studio.
  • My plugin does not load. Enablement plus a restart are both required. Enabling in Settings without restarting, or restarting without enabling, both leave it dormant. Watch the server logs at startup for the "Plugin hook registered" line if it installs a hook.
  • My plugin edit did not take. Loaded plugins are imported once at startup. Restart the server after editing the file.
